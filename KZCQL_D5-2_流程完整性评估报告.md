## 二、检查项 2：SOLO适配器 WRITING_AGENT_SEQUENCE 与编排指南一致性

### 2.1 序列对比

**编排指南定义的写作流程**（主Agent编排指南 2.1节）:
```
阶段一: [D2调研 → D3角度挖掘 → 人类选择角度] → W1 → ImageCheck
阶段二: R1 → R2（如R1=PASS）→ R4
阶段四: W2 → R3 → R2 → R4
```

**SOLO适配器定义的 WRITING_AGENT_SEQUENCE**:
```python
WRITING_AGENT_SEQUENCE = [
    "D2",         # 调研Agent（长文必须，短文可跳过）
    "D3",         # 角度挖掘Agent（长文必须，短文可跳过）
    "W1",         # 初稿撰写Agent（强制）
    "ImageCheck", # 配图检查
    "R1",         # 事实核查（强制）
    "R2",         # 全量评审（强制）
    "R3",         # 差异评审（如需要）
    "R4",         # 评级判定（强制）
    "D1",         # 配图设计（如需要）
]
```

### 2.2 一致性分析

| 对比项 | 编排指南 | SOLO适配器 | 一致性 |
|--------|----------|-----------|--------|
| D2/D3位置 | W1之前 | W1之前 | 一致 |
| W1位置 | D2/D3之后 | D2/D3之后 | 一致 |
| ImageCheck位置 | W1之后、R1之前 | W1之后、R1之前 | 一致 |
| R1→R2→R4顺序 | 串行 | 序列中相邻 | 一致 |
| R3位置 | 迭代轮次>=2时 | 序列中R2之后 | **存在歧义** |
| D1位置 | 未在主流程中明确 | 序列末尾 | **存在歧义** |
| W2 | 阶段四 | **序列中缺失** | **不一致** |

### 2.3 发现的问题

**问题 S-01 [严重程度: 高]**
- **描述**: `WRITING_AGENT_SEQUENCE` 中**缺少 W2（迭代修改Agent）**。W2是阶段四迭代循环的核心Agent，但在写作序列中完全没有出现。
- **影响**: SOLO适配器无法生成包含W2的迭代执行计划。`generate_writing_execution_plan()` 方法只生成首次写作流程（D2→D3→W1→ImageCheck→R1→R2→R3→R4→D1），不包含迭代修改循环（W2→R1复检→R3→R2→R4）。
- **修复建议**:
  1. 在 `OrchestratorConfig` 中新增 `ITERATION_AGENT_SEQUENCE = ["W2", "R1", "R3", "R2", "R4"]`
  2. 新增 `generate_iteration_execution_plan()` 方法，生成迭代循环的执行计划
  3. 在 `WRITING_AGENT_SEQUENCE` 中将W2标注为条件触发（迭代轮次>=1时）

**问题 S-02 [严重程度: 中]**
- **描述**: `WRITING_AGENT_SEQUENCE` 将R3放在R2之后、R4之前，这在首次评审流程中是不正确的。R3（差异评审）仅在**迭代轮次>=2时**才启用，首次评审流程是R1→R2→R4，不经过R3。
- **影响**: 如果主Agent严格按照序列顺序执行，首次评审会错误地调用R3（此时没有修改前版本可供对比）。
- **修复建议**: 将序列拆分为两个独立序列：
  - `FIRST_REVIEW_SEQUENCE = ["R1", "R2", "R4"]`（首次评审）
  - `ITERATION_REVIEW_SEQUENCE = ["R1", "R3", "R2", "R4"]`（迭代评审，R1复检→R3差异→R2全量→R4评级）

**问题 S-03 [严重程度: 低]**
- **描述**: `WRITING_AGENT_SEQUENCE` 中D1（配图设计）放在序列末尾，但编排指南中D1的职责是"生成、质检、嵌入配图"，应在W1阶段执行（W1规范1.3节明确W1负责配图生成）。D1作为独立Agent被放在写作流程末尾，语义不清晰。
- **影响**: 可能导致配图生成时机混乱。
- **修复建议**: 明确D1在序列中的触发条件。如果D1是独立于W1的配图优化Agent，应在R4评级后、交付前触发；如果D1的职责已由W1承担，则应从序列中移除。

**问题 S-04 [严重程度: 中]**
- **描述**: `generate_writing_execution_plan()` 方法中，R3的 `depends_on` 设为 `["R2"]`，但R3的输入需要"修改前文章"和"修改后文章"。在首次评审流程中，没有"修改前文章"，R3不应被调用。
- **影响**: 执行计划会在首次流程中生成R3步骤，但R3缺少必要输入。
- **修复建议**: 在R3步骤中增加条件判断：`"condition": "iteration_round >= 2"`。

## 三、检查项 3：D2调研→D3角度挖掘→W1初稿→R1→R2→R4 完整链路通畅性

### 3.1 链路逐段检查

| 链路段 | 状态 | 详情 |
|--------|------|------|
| D2→D3 | 通畅 | D3输入包模板要求D2信息地图作为[MUST]输入；D2输出格式（JSON/Markdown）与D3输入格式兼容；编排详细流程手册定义了D2失败时的降级机制 |
| D3→人类选择→W1 | 通畅 | 编排详细流程手册1.3节定义了完整的Step C（C1展示→C2等待选择→C3确认→C4传递给W1）；W1输入包模板的[CONDITIONAL]字段接收D3角度选项 |
| W1→ImageCheck | 通畅 | SOLO适配器中ImageCheck的depends_on为["W1"]；编排指南P35补丁定义了ImageCheck的三项检查 |
| ImageCheck→R1 | 通畅 | SOLO适配器中R1的depends_on为["ImageCheck"]；R1输入包模板定义了完整的[MUST]输入 |
| R1→R2 | **有条件通畅** | 编排详细流程手册2.2节明确定义：R1=PASS→执行R2；R1=FAIL→跳过R2直接执行R4。但SOLO适配器中R2的depends_on仅为["R1"]，未区分PASS/FAIL条件 |
| R2→R4 | 通畅 | SOLO适配器中R4的depends_on为["R1", "R2", "R3"]；R4输入包模板要求R1报告和R2/R3评分报告 |

### 3.2 发现的问题

**问题 L-01 [严重程度: 中]**
- **描述**: SOLO适配器中R2→R4的链路缺少R1=FAIL时的分支处理。当R1=FAIL时，编排指南要求跳过R2直接执行R4，但`generate_writing_execution_plan()`生成的执行计划中R2是固定步骤（`type: "mandatory"`），没有条件分支。
- **影响**: 执行计划无法表达"R1=FAIL时跳过R2"的条件逻辑。主Agent需要自行判断是否跳过R2，执行计划本身不提供此信息。
- **修复建议**: 将R2步骤的type改为`"conditional"`，增加条件字段：`"condition": "R1_result == 'PASS'"`。

**问题 L-02 [严重程度: 低]**
- **描述**: D2→D3链路中，D2失败时的降级机制在编排详细流程手册中有详细定义（询问用户是否跳过D2、重试3次后强制跳过），但SOLO适配器中D3步骤的`depends_on`为`["D2"]`，如果D2未执行或失败，D3的依赖无法满足。
- **影响**: 当D2被跳过时，D3的depends_on机制会阻断D3执行，但编排指南允许D2跳过后直接执行D3。
- **修复建议**: 将D3的depends_on改为可选依赖，增加降级说明：`"depends_on_optional": ["D2"], "fallback": "D2未启用时基于主题直接生成角度"`。

## 四、检查项 4：W2修改→R3差异评审→R2重评 的回退链路通畅性

### 4.1 回退链路检查

| 链路段 | 状态 | 详情 |
|--------|------|------|
| W2→R3 | **不通畅** | 迭代循环手册定义了W2→R1复检（如前置R1=FAIL）→R3的流程，但SOLO适配器中完全没有W2和迭代循环的执行计划生成逻辑 |
| R3→R2重评 | **文档通畅，代码不通畅** | 迭代循环手册4.3节定义了阶段四出口检查：R3标记"需R2重评"时强制进入R2全量评审。但SOLO适配器无此逻辑 |
| R3→R2回退判定 | 文档通畅 | R3输入包模板5.2节定义了"修改幅度>30%时自动回退R2全量评审"的机制 |

### 4.2 发现的问题

**问题 R-01 [严重程度: 高]**
- **描述**: SOLO适配器（`orchestrator_solo_adapter.py`）**完全没有实现迭代循环的执行计划生成**。`SoloOrchestrator`类只有`generate_execution_plan()`（架构评估）和`generate_writing_execution_plan()`（首次写作流程），没有`generate_iteration_execution_plan()`方法。
- **影响**: 阶段四的迭代循环（W2→R1复检→R3→R2→R4）在SOLO适配器层面完全缺失。主Agent无法通过SOLO适配器获得迭代循环的执行计划，只能依赖文档手动编排。
- **修复建议**:
  1. 新增 `generate_iteration_execution_plan(iteration_round: int, prev_r1_failed: bool)` 方法
  2. 实现以下逻辑：
     - W2（强制）→ R1复检（如前置R1=FAIL，条件强制）→ R3差异评审（迭代轮次>=2时强制）→ R2全量评审（如R3标记需重评或修改幅度>30%）→ R4评级（强制）
  3. 在 `OrchestratorConfig` 中新增 `ITERATION_AGENT_SEQUENCE`

**问题 R-02 [严重程度: 中]**
- **描述**: 迭代循环手册4.1.2b节定义了"R3阻断处理"机制（R3标记"需R1复检"时回到R1复检），但此回退链路在SOLO适配器中完全没有实现。此外，R3输入包模板中虽然定义了"修改幅度>30%时回退R2"的机制，但缺少对"R3标记需R1复检"场景的明确处理。
- **影响**: R3→R1复检的回退链路在代码层面缺失，主Agent需完全依赖文档理解来实现此逻辑。
- **修复建议**:
  1. 在R3输入包模板的[MUST]字段中增加"前置R1事实核查报告"（已存在，确认覆盖）
  2. 在SOLO适配器的迭代执行计划中增加R3→R1复检的回退路径
  3. 在迭代循环手册中增加R3阻断处理的流程图

**问题 R-03 [严重程度: 中]**
- **描述**: 迭代循环手册4.3节定义了"阶段四出口检查"，要求W2修改后检查"修改后评审触发判定"章节。但W2输入包模板的[调用后验证清单]中未包含"修改说明中是否包含评审触发判定"的检查项。
- **影响**: W2输出可能缺少"修改后评审触发判定"章节，导致阶段四出口检查无法执行。
- **修复建议**: 在W2输入包模板的[调用后验证清单]中增加：`- [ ] **修改说明中包含"修改后评审触发判定"章节** — 明确列出哪些修改可能触发R2重评`

- **描述**: D2调研Agent的规范文件（D2_调研Agent.md Step 9）明确要求查阅素材索引（`/workspace/KZCQL/01_共享知识库/创作素材/素材索引.md`），但D2的输入包模板和执行代码（D2_research_agent.py）中**均未集成此步骤**。
- **影响**: D2在执行时不会查阅已有的创作素材库，导致：
  1. 可能重复调研已有素材覆盖的主题
  2. 遗漏素材库中与主题相关的随笔素材
  3. 规范文件与实际执行不一致（D5-2执行验证映射会标记此为执行缺口）
- **修复建议**:
  1. 在D2_输入包模板.md的[SHOULD]字段中增加：`| 创作素材索引 | /workspace/KZCQL/01_共享知识库/创作素材/素材索引.md | Markdown | 不查阅已有素材，从零开始调研 |`
  2. 在D2_research_agent.py的`execute_research()`方法中增加素材索引查阅步骤（在WebSearch之前先检查已有素材）

**问题 M-02 [严重程度: 中]**
- **描述**: W1输入包模板中"创作素材索引"列为[MUST]，但W1规范文件（初稿撰写Agent.md）的2.1节必需输入中未显式列出素材索引路径。W1规范仅提到"原始材料/主题"作为必需输入。
- **影响**: 规范文件与输入包模板存在不一致。如果子Agent仅读取规范文件而不读取输入包模板，可能遗漏素材查阅步骤。
- **修复建议**: 在W1规范文件2.1节必需输入中增加"创作素材索引"条目，或在规范文件中引用输入包模板。

**问题 M-03 [严重程度: 低]**
- **描述**: 素材库的查阅流程在D2和W1之间的分工不明确。D2规范要求查阅素材索引，W1输入包模板也要求提供素材索引，但两者查阅素材的目的和时机不同（D2用于调研参考，W1用于写作素材），目前没有明确说明。
- **影响**: 可能导致D2和W1重复查阅同一素材，或两者对素材的使用方式不一致。
- **修复建议**: 在D2和W1的规范中明确素材查阅的分工：D2查阅素材索引用于调研方向参考（标注已有素材覆盖的领域），W1查阅素材索引用于写作素材获取（提取具体随笔内容）。

## 八、总体评估

### 8.1 流程完整性评分

| 维度 | 评分（10分制） | 说明 |
|------|---------------|------|
| **文档层流程定义** | 8.5/10 | 编排指南和各手册覆盖了所有场景，仅在少数边界情况（W1重写后路径、R1复检标注）存在模糊 |
| **输入包模板体系** | 9.0/10 | 模板完整、结构规范、质量高，仅有少量格式和分类问题 |
| **SOLO适配器代码实现** | 5.0/10 | 仅实现了首次写作流程和架构评估流程，**迭代循环完全缺失**，条件分支逻辑不完整 |
| **规范-代码一致性** | 6.0/10 | D2素材查阅、W2迭代、R3条件触发等规范要求在代码中未实现 |
| **素材库集成** | 6.5/10 | W1侧集成较好，D2侧规范与代码/模板存在断裂 |

### 8.2 核心结论

1. **文档层的流程设计是完整且严谨的**。编排指南、迭代循环手册、输入包模板使用指南等文档构成了一个完整的流程体系，覆盖了长文/短文/修改/评审/迭代/架构评估等所有场景。

2. **最大的缺口在SOLO适配器的代码实现**。`orchestrator_solo_adapter.py` 仅实现了"首次写作流程"和"架构评估流程"两个执行计划生成方法，**完全缺少迭代循环的执行计划生成**。这意味着阶段四（W2→R3→R2→R4）的自动化编排在代码层面是断裂的。

3. **D2素材库查阅存在"规范写了但代码没做"的典型执行缺口**。D2规范文件明确要求查阅素材索引，但输入包模板和执行代码均未集成此步骤，这正是D5-2执行验证映射应当捕获的问题类型。

4. **输入包模板体系是整个系统中质量最高的部分**。16个模板结构统一、字段完整、错误处理到位，且通过三重保障机制（规范文件+注入器+模板）确保子Agent读取自身规范。
