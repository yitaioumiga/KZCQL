|------|------|

| **全名** | 综合评级官 |

| **输出格式** | 综合评级报告，包含：综合评级（S/A/B/C/D/F）、各维度加权得分、路由建议、关键问题摘要 |

**评级标准**（来源：`01_共享知识库/后置评审规则/评分体系.md`）：

| 评级 | 分数范围 | 含义 | 路由建议 |
|------|----------|------|----------|
| S | 95-100 | 卓越，可直接发布 | 人工终审 |
| A | 85-94 | 优秀，通过评审 | 人工终审 |
| B | 75-84 | 良好，需小幅修改 | W2 迭代修改 |
| C | 65-74 | 合格，需较大修改 | W2 迭代修改 |
| D | 60-64 | 勉强，需大幅修改 | W1 重写 |
| F | <60 | 不合格，建议重写 | W1 重写 |

### 1.14 Orchestrator — 编排器（P30补丁新增）

| 字段 | 内容 |
|------|------|
| **代号** | Orchestrator |
| **全名** | 主智能体编排器 |
| **角色定位** | 编排指挥官，强制主智能体按照规范正确调用子智能体，禁止跳过任何步骤 |
| **输入格式** | 任务ID + 主题 + 执行计划配置 |
| **输出格式** | Agent调用记录（agent_call_records.json）+ 执行日志（orchestrator.log）+ 任务目录 |
| **触发条件** | 主Agent需要执行多Agent协作任务时；架构评估等复杂任务编排时 |
| **禁止事项** | 1. 禁止跳过MANDATORY_AGENTS（W1/R1/R2/R4）<br>2. 禁止自动决策OPTIONAL_AGENTS（必须显式决策）<br>3. 禁止不记录调用（每个调用必须记录）<br>4. 禁止异常时自动继续（必须进入阻断流程）<br>5. 禁止逆序或重复调用Agent |

**核心功能**：
1. **强制顺序**：按照AGENT_SEQUENCE顺序调用（D2→D3→W1→R1→R2→R4）
2. **禁止跳过**：MANDATORY_AGENTS绝对禁止跳过
3. **显式决策**：OPTIONAL_AGENTS必须显式决策（启用/跳过）
4. **记录完整**：每个调用必须有记录，禁止"隐形"调用
5. **异常阻断**：失败时必须进入阻断流程，禁止自动继续

**配置项**：
- `AGENT_SEQUENCE`: 子智能体调用顺序
- `MANDATORY_AGENTS`: ["W1", "R1", "R2", "R4"]
- `OPTIONAL_AGENTS`: ["D2", "D3", "ImageCheck", "R3", "D1"]
- `MAX_RETRIES`: 3（最大重试次数）
- `TIMEOUT_SECONDS`: 300（超时时间）

**规范文件**：`02_子Agent规范/架构专家组/Orchestrator_编排器.md`（待创建）

**可执行代码**：
- `03_执行代码/主智能体编排/orchestrator.py`（基础版本）
- `03_执行代码/主智能体编排/orchestrator_solo_adapter.py`（SOLO适配版本）

**使用方式**：

## 二、路由决策树

### 2.1 任务类型判定

主Agent 在接收到用户任务后，首先判定任务类型：

```
用户输入
  │
  ├── 包含「写一篇文章」「帮我写」「公众号文章」等关键词
  │   → 任务类型 = 「完整流程」
  │
  ├── 包含「评审」「打分」「看看这篇文章」等关键词
  │   → 任务类型 = 「仅评审」
  │
  ├── 包含「微调」「改一下」「调整」「优化」等关键词
  │   → 任务类型 = 「微调」
  │
  ├── 包含「更新规则」「修改规范」「规则演进」等关键词
  │   → 任务类型 = 「规则演进」
  │
  └── 无法判定
      → 向用户确认任务类型
```

```
任务类型 = 「完整流程」

├── Step 0: 调研与角度挖掘决策（P28补丁新增，可选）

│   │
│   ├── 用户选择「跳过」→ 直接进入 Step 1

│   │
│   └── 用户选择「启用」→ 执行以下步骤
│       │
│       ├── Step 0.1: D2 调研Agent
│       │   ├── 输入：用户主题 + 可选领域偏好
│       │   ├── 输出：信息地图（热点+跨领域+竞品+历史）
│       │   └── 保存：Markdown + JSON 报告
│       │
│       ├── Step 0.2: D3 角度挖掘Agent
│       │   ├── 输入：D2信息地图 + 用户主题
│       │   ├── 输出：3-5个角度选项 + 推荐理由

│           ├── 记录选定角度

│   ├── 1.1 文章主题是什么？

│   ├── 1.3 核心观点/结论是什么？

│   ├── 读取用户身份档案
│   ├── 读取公众号创作规范
│   ├── 调用 khazix-writer Skill
│   └── 输出：初稿 v0.1
│
├── Step 3: 首轮全量评审
│   ├── R1 事实核查
```

## 三、单一职责约束

### 3.1 核心原则

> **严禁合并 Agent 调用。** 每个 Agent 必须独立执行、独立输出、独立验证。

### 3.2 具体约束

| 约束项 | 说明 |
|--------|------|
| 禁止合并评审 | R1 和 R2 不能合并为一次评审调用，即使它们评审同一篇文章 |
| 禁止评审+修改 | 评审 Agent（R1/R2/R3/R4）不能同时执行修改操作 |
| 禁止修改+评审 | 修改 Agent（W1/W2）不能同时执行评审操作 |
| 禁止提取+修改 | A1 和 E1 不能合并为一次调用 |
| 禁止写作+配图 | W1/W2 不能同时执行配图生成操作（配图由 D1 专职负责） |
| 禁止跳级 | 不能跳过中间 Agent（如 R1→R4，跳过 R2） |
| 禁止并行评审 | R1 和 R2 必须串行执行（R1=FAIL 时 R2 被跳过，这是路由规则，不是并行） |

### 3.3 违规检测

主Agent 在每次调度 Agent 时，必须检查：

1. 当前 Agent 的输入是否已准备就绪？
2. 是否存在未完成的依赖 Agent？
3. 是否违反了单一职责约束？

如果检测到违规，主Agent 必须中止调度并向用户报告。

## 四、R1/R2 串行策略

### 4.1 设计理由

R1（内容评审）和 R2（风格评审）采用串行策略而非并行策略，原因如下：

1. **内容优先**：如果文章在事实准确性、逻辑完整性上存在严重问题，风格评审没有意义
2. **避免浪费**：R1=FAIL 的文章大概率需要大幅修改，R2 的评审结果将失效
3. **效率优化**：串行策略在 R1=FAIL 时可以节省一次 R2 调用

### 4.2 执行规则

```
R1 执行
  │
  ├── R1 = PASS
  │   └── 执行 R2
  │       ├── R2 = PASS → 进入 R4
  │       └── R2 = FAIL → 进入 R4（R4 会综合两个 FAIL 降级）
  │
  └── R1 = FAIL
      └── 跳过 R2 → 直接进入 R4
          └── R4 基于 R1 的 FAIL 结果给出降级评级
```

### 4.3 特殊情况

| 场景 | 处理方式 |
|------|----------|
| R1=PASS，R2=FAIL | R4 综合两个结果，通常评级为 B 或 C |
| R1=FAIL，R2=FAIL（假设执行了） | R4 以 R1 为主要依据，评级通常为 C 或 D |
| R1=FAIL，但内容问题可快速修复 | W2 修复内容问题后，R1 复审 PASS，再执行 R2 |
