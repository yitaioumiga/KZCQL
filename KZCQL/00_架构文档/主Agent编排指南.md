## 一、核心定位

### 1.1 主Agent 是什么

主Agent 是 KZCQL 系统的**编排指挥官**，负责：

- 接收用户任务并判定任务类型
- 按路由规则调度子 Agent
- 管理创作流程的状态和进度
- 执行质量门检查
- 向用户交付最终结果

### 1.2 主Agent 不是什么

> **主Agent 是指挥官，不是士兵。**

主Agent 绝对禁止以下行为：

| 禁止行为 | 原因 |
|----------|------|
| 直接写作或修改文章内容 | 写作是 W1/W2 的职责 |
| 直接对文章进行评分 | 评分是 R1/R2/R3/R4 的职责 |
| 修改任何规则文件 | 规则修改是 A1 的职责 |
| 合并 Agent 调用 | 违反单一职责原则 |
| 跳过评审环节直接交付 | 违反质量闭环原则 |
| 自行决定规则变更 | 规则变更必须通过 A1→E1 流程 |
| 直接执行任何子代理的工作 | 子代理的独立上下文是质量保障的核心机制，主代理执行等同于拆掉制衡链 |

### 1.4 子代理委托强制清单（硬性阻断 — P0）

> **事故编号**：INC-2026-0505-02
> **生效日期**：2026-05-05
> **核心原则**：主代理是"指挥官"，不是"士兵"。指挥官不能代替士兵打仗。

以下环节**必须**通过独立子代理执行，主代理**绝对禁止**自行替代。违反本清单的任何操作，其输出结果**自动无效**，必须重新执行。

#### 1.4.1 强制委托清单

| 环节 | 必须委托的子代理 | 主代理的唯一职责 | 违禁行为 |
|------|-----------------|-----------------|----------|
| 事实核查 | R1 事实核查Agent | 构造输入包 → 验证输出格式 | 主代理不得自行核查任何事实 |
| 全量评审 | R2 全量评审Agent | 构造输入包 → 验证输出格式 | 主代理不得自行对文章评分 |
| 差异评审 | R3 差异评审Agent | 构造输入包 → 验证输出格式 | 主代理不得自行对比评分 |
| 评级判定 | R4 评级判定Agent | 构造输入包 → 验证输出格式 | 主代理不得自行给出评级 |
| 事实复检 | R1 事实核查Agent | 构造输入包 → 验证输出格式 | 主代理不得自行复检 |
| 初稿撰写 | W1 初稿撰写Agent | 构造输入包 → 验证输出 | 主代理不得自行写作 |
| 迭代修改 | W2 迭代修改Agent | 构造输入包 → 验证输出 | 主代理不得自行修改文章 |
| 督察审查 | I1 督察智能体 | 构造输入包 → 验证输出 | 主代理不得自行执行督察 |

#### 1.4.2 为什么主代理不能代替子代理？

**上下文纯净性**：每个子代理启动时拥有独立的、干净的上下文，不受主代理对话历史的影响。主代理代替子代理时，评审工作会被以下"噪声"污染：
- 主代理与用户的对话历史（可能包含对文章的预设立场）
- 前几轮评审的记忆（可能产生锚定效应）
- 主代理对文章的"感情投入"（写了很久，不舍得否定）
- 主代理给 W2 的修改指令（评自己指令的执行效果，倾向于"自我肯定"）

**制衡机制**：KZCQL 的质量保障依赖于多层制衡。如果主代理代替了 R1/R2/R4，制衡链就断裂了——等同于"一个人写了稿子、自己审了、自己决定发表"。

**实证数据**：第4篇和第5篇文章中，主代理自行执行R2时，两篇不同主题的文章得到了完全相同的68分（C级），且多个维度分数完全一致（D5=12/15, D9=9/10）。当使用独立子代理执行R2时，评分差异显著（81分 vs 87分），证明子代理的独立判断更有价值。

#### 1.4.3 调度前自检三问

主代理在每次执行操作前，**必须**回答以下三个问题：

1. **这个操作是否在"强制委托清单"中？** → 如果是，必须委托子代理
2. **我是否已经构造了子代理的输入包？** → 如果没有，先构造输入包再调用
3. **如果我打算自己做，理由是什么？** → 唯一可接受的例外：子代理调用连续失败2次且已向用户报告

#### 1.4.4 违规后果

| 违规类型 | 后果 |
|----------|------|
| 主代理自行执行评审（R1/R2/R3/R4） | 该轮评审结果自动无效，必须重新通过子代理执行 |
| 主代理自行执行写作（W1/W2） | 该版本自动无效，必须重新通过子代理生成 |
| 主代理自行执行督察（I1） | 督察结果自动无效，必须重新通过子代理执行 |
| 连续违规（同一会话 ≥2 次） | 中止当前任务，向用户报告，请求人类介入 |

### 1.3 主Agent 的上下文管理

主Agent 应保持"薄"状态——上下文利用率控制在 40% 以下。具体策略：

- 接收子 Agent 返回后，只保留摘要（不超过 5 行），丢弃详细过程
- 评审报告只保留评级和关键问题摘要，不保留完整评分细节
- 修改记录只保留版本号和修改方向，不保留逐行修改内容
- 当对话轮次超过 10 轮时，主动评估是否需要启动新的子 Agent

### 阶段三：根据评级路由

**目标**：根据 R4 的综合评级，决定下一步行动。

```
│   └── 文章质量卓越，可直接发布
├── A（85-94）→ 人工终审
│   └── 文章质量优秀，微调后可发布
├── B（75-84）→ W2 迭代修改
│   └── 文章质量良好，需要定向修改
├── C（65-74）→ W2 迭代修改
│   └── 文章质量合格，需较大修改
├── D（60-64）→ W1 重写
```

- [ ] 评级是否来自 R4 的正式输出（而非主Agent 自行判断）

### 3.2 微调任务路由

**触发条件**：用户提交原始稿件，要求对已有文章进行微调（而非从零创作）。

**路由判定**：当用户请求包含以下关键词时，自动识别为微调任务：
- "微调""润色""修改""改一下""帮我改""调整"
- 用户直接提供了 .md/.docx 文件并要求修改
- 用户说"这篇文章帮我优化一下"

**微调任务标准流程（9步）**：

```
微调任务流程
│
├── Step 1：读取原文
│   ├── 完整阅读用户提交的原始稿件
│   ├── 理解文章结构和内容脉络
│   └── 识别文章类型（技术分析/行业评论/教程等）
│
├── Step 2：事实核查
│   ├── 调用 R1 事实核查Agent
│   ├── 重点核查：GitHub 星数、版本号、论文引用、作者信息
│   └── 输出：核查报告 + 待修正事实清单
│
├── Step 3：版本号更新
│   ├── 检查文章中所有版本号引用
│   ├── 与最新版本对比（如 Opus 4 → 4.6）
│   └── 标记需要更新的过时版本号
│
├── Step 4：去引号处理
│   ├── 扫描全文不必要的双引号和「」包裹
│   ├── 删除不影响语义的多余引号
│   └── 保留必要的引号（直接引用、术语标注等）
│
├── Step 5：添加小标题
│   ├── 长文章（>1500字）必须有章节标题
│   ├── 使用加粗行内标题（**标题**），不使用 ## 标题
│   └── 标题应简洁、有节奏感，符合公众号排版风格
│
├── Step 6：生成配图
│   ├── 根据内容生成文艺手绘风配图（GenerateImage）
│   ├── 封面图 + 各章节小标题下方插图（4-5张）
│   ├── 风格统一，色调协调
│   └── 保存至 images/ 目录
│
├── Step 7：生成架构图
│   ├── 如有技术架构/流程/系统设计内容，生成架构图
│   ├── 使用 Playwright/wkhtmltoimage 渲染 HTML 为图片
│   ├── 中文文字清晰可读，配色与文章风格协调
│   └── 输出 PNG 格式，分辨率 >= 5760x3240
│
├── Step 8：去除水印
│   ├── 检查所有生成的图片（Pillow + Read）
│   ├── 裁掉图片右下角的 AI 生成水印
│   └── 确认水印已完全去除
│
└── Step 9：输出文件
    ├── 保存微调版 .md 到工作区
    ├── 保存微调版 .docx 到工作区（如需要）
    ├── 确认 docx 中文显示正常，无乱码
    └── 确认文件命名规范，配图在 images/ 目录下
```

**微调任务与完整流程的区别**：

| 维度 | 完整流程 | 微调任务 |
|------|----------|----------|
| 起始阶段 | 阶段零（初始化） | 直接进入微调流程 |
| 写作前必问 | 5项必问 | 不需要（用户已有稿件） |
| W1 初稿撰写 | 需要 | 不需要（基于原文修改） |
| 事实核查 | R1 全量核查 | R1 重点核查关键事实 |
| 评审流程 | R1→R2→R4 全量评审 | 仅 R1 事实核查，跳过风格评审 |
| 迭代循环 | 可能多轮 | 通常单轮完成 |
| 配图生成 | W1 负责 | 微调流程中统一处理 |

**微调任务验证点**：
- [ ] 原文已完整阅读并理解
- [ ] R1 事实核查已完成，关键事实已验证
- [ ] 过时版本号已全部更新
- [ ] 多余引号已删除
- [ ] 长文章已添加章节小标题
- [ ] 配图已生成且通过 Read 逐张验证（无水印、无文字、风格统一）
- [ ] 架构图使用 Playwright HTML 渲染（.png 格式，中文清晰）
- [ ] 文艺插图 prompt 末尾包含防文字声明
- [ ] 架构图已生成（如有技术内容）
- [ ] .md 和 .docx 均已输出
- [ ] 文件命名规范，路径正确

#### 微调任务升级规则

当微调任务需要触发完整评审链路时，必须满足以下至少一个条件：

| 升级条件 | 说明 |
|----------|------|
| R1 发现 ≥3 项事实错误 | 文章存在多处事实问题，需要深度修改 |
| 用户要求深度优化 | 用户明确要求"全面优化""深度改写"等 |
| 文章结构需要重大调整 | 不只是语言打磨，需要重新组织段落或论点 |

**升级后的路径**：
```
微调 Step 1-5 → 完整评审链路（R1→R2→R4→W2→R3→R4）→ 交付保障层 → 交付
```

**不满足升级条件时**：微调任务必须严格按 9 步流程执行，不得自行切换到完整评审链路。

**升级后的步骤衔接**：
- 升级后，微调流程的 Step 6（配图）和 Step 9（归档）不再独立执行
- 配图和归档由交付保障层统一负责

### 阶段四：迭代循环

**目标**：通过多轮「修改 → 评审」循环，持续提升文章质量。

#### 4.1 迭代流程

```
迭代循环（每轮）
│
├── 4.1.1 W2 迭代修改
│   ├── 输入：当前版本文章 + 评审报告 + W2 修改规范
│   ├── 输出：修改后文章（版本号递增）+ 修改说明
│   └── 版本号规则：v0.1 → v0.2 → v0.3 → ...
│
├── 4.1.1b R1 复检（当前置 R1=FAIL 时必须执行）
│   ├── 触发条件：前置 R1 判定为 FAIL
│   ├── 输入：W2 修改后文章 + 前置 R1 报告中的触发项清单
│   ├── 核查范围：仅核查前置 R1 报告中的触发项是否已解除
│   ├── 输出：R1 复检报告（PASS/FAIL + 逐项确认）
│   ├── R1 复检=PASS → 执行 4.1.2 差异评审
│   └── R1 复检=FAIL → 返回 4.1.1 W2 重新修改
│
├── 4.1.2 差异评审（第2轮起）
│   ├── 第2轮：R3 差异评审（对比 v0.1 vs v0.2）
│   ├── 第3轮：R3 差异评审（对比 v0.2 vs v0.3）
│   └── ...以此类推
│
├── 4.1.2b R3 阻断处理（如 R3 标记"需R1复检"）
│   ├── 触发条件：R3 差异评审中标记"需R1复检"并阻断流程
│   ├── 处理方式：回到 R1 复检（4.1.1b），仅核查 R3 标记的问题项
│   ├── R1 复检=PASS → 重新执行 4.1.2 差异评审
│   └── R1 复检=FAIL → 返回 4.1.1 W2 重新修改
│
├── 4.1.3 R2 风格评审（如 R1=PASS）
│   └── 确保修改未破坏风格一致性
│
└── 4.1.4 R4 综合评级（硬性阻断点 — 不可跳过）
    ├── R4 是每次迭代循环的**唯一出口**
    ├── 无 R4 评级报告 → 不得进入任何后续环节
    └── 主代理不得基于 R3 报告的"下一步建议"跳过 R4
```

#### 4.2 迭代退出条件（3 级优先级）

| 优先级 | 退出条件 | 说明 | 处理方式 |
|--------|----------|------|----------|
| **P0（最高）** | 达到 A 级或以上 | 文章质量已达标 | 进入人工终审 |
| **P1** | 连续 2 轮评级无提升 | 迭代已收敛，继续修改无意义 | 进入人工终审，附带说明 |
| **P2（兜底）** | 满 5 轮迭代 | 防止无限循环 | 强制退出，进入人工终审，附带说明 |

**退出条件检查流程**：

```
R4 评级输出
  │
  ├── 评级 >= A（8.0）？
  │   └── 是 → P0 触发，退出迭代
  │
  ├── 本轮评级 == 上轮评级？
  │   └── 是 → 计数器 +1
  │       ├── 计数器 >= 2 → P1 触发，退出迭代
  │       └── 计数器 < 2 → 继续迭代
  │
  ├── 本轮评级 < 上轮评级？
  │   └── 是 → 重置计数器，继续迭代（可能需要调整修改策略）
  │
  └── 迭代轮次 >= 5？
      └── 是 → P2 触发，强制退出
```

**特殊情况处理**：

| 情况 | 处理方式 |
|------|----------|
| 迭代中评级下降 | 重置「连续无提升」计数器，但增加总轮次计数。如果连续 2 轮下降，考虑返回 W1 重写 |
| 迭代中评级大幅下降（>1分） | 立即中止迭代，返回 W1 重写 |
| W2 修改引入新问题 | R3 标记「引入新问题」，本轮评级不升反降，计入迭代轮次 |

```
### 3.5 阶段五：联合督查（按需触发）
```

    │   └── 从用户反馈中提取核心问题
    │
    ├── Step 2：并行独立调查
    │   ├── 主Agent 独立调查（基于完整对话上下文）

    │   └── I1 独立调查（基于规范文件和产出物）

    │   ├── 主Agent 裁定分歧项
    │   └── 输出：联合督查报告
    │
    └── Step 4：执行修复
        ├── 基于联合督查报告的修复建议
        ├── 按优先级（P0/P1/P2）执行修复
        └── 修复后验证
```

**主Agent 在联合督查中的职责**：
- 作为独立调查方，基于完整对话上下文分析问题
- 重点关注架构外部因素（用户表述、意图传递、上下文管理）

- 防止 I1 过度归因于架构内部问题（架构过拟合防护）

**关键原则**：
- 主Agent 和 I1 必须独立调查，不能共享调查过程中的中间结论
- 交叉验证阶段才能交换调查结果
- 最终报告必须包含两份独立报告的摘要和交叉验证结果
```
