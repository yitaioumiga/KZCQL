## 一、工作流全景

KZCQL 的完整工作流分为 5 个阶段，形成闭环：

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  阶段1：准备          阶段2：创作         阶段3：评审        │
│  ┌──────────┐       ┌──────────┐       ┌──────────┐       │
│  │ 写作前必问 │──────→│ W1撰写初稿 │──────→│ R1→R2→R4 │       │
│  │ 读取规范   │       │ (Skill)  │       │ 首轮评审  │       │
│  └──────────┘       └──────────┘       └─────┬────┘       │
│                                                 │           │
│                                                 ▼           │
│  阶段5：演进          阶段4：迭代                      │
│  ┌──────────┐       ┌──────────┐                    │
│  │ A1→E1   │←──────│ W2修改   │←──── 评级路由       │
│  │ 规则进化  │       │ R1/R3评审 │                    │
│  └────┬─────┘       └──────────┘                    │
│       │                                              │
│       └──────→ 更新规范 → 回到阶段1 ────────────────┘
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 各阶段核心目标

| 阶段 | 名称 | 核心目标 | 关键产出 |
|------|------|----------|----------|
| 1 | 准备 | 充分了解需求，加载创作规范 | 写作需求文档 |
| 2 | 创作 | 通过 khazix-writer Skill 产出初稿 | 初稿 v0.1 |
| 3 | 评审 | 全面评估文章质量，给出评级 | 评审报告 + 综合评级 |
| 4 | 迭代 | 根据评审反馈定向修改，持续提升质量 | 迭代版本 + 修改说明 |
| 5 | 演进 | 提取经验，进化规则，形成闭环 | 经验条目 + 规则变更 |

### 阶段间的依赖关系

```
阶段1 → 阶段2：写作需求文档是 W1 的输入
阶段2 → 阶段3：初稿是评审的输入
阶段3 → 阶段4：评审报告是 W2 修改的依据
阶段4 → 阶段3：修改后的版本需要再次评审（循环）
阶段4 → 阶段5：迭代结束后进入经验提取
阶段5 → 阶段1：更新后的规范用于下一轮创作
```

## 二、15 条铁律

以下铁律是 KZCQL 系统运行的基石，所有 Agent 必须严格遵守。在 MTCQL 14 条铁律基础上，KZCQL 新增了第 15 条。

### 铁律 1：单一职责，绝不越界

> 每个 Agent 只做一件事。写作的不要评审，评审的不要修改，修改的不要评分。

**违反后果**：质量闭环失效，无法追溯问题来源。

### 铁律 2：规则驱动，拒绝感觉

> 所有创作决策必须基于可量化、可追溯的规则文件。禁止「我觉得这样更好」式的决策。

**违反后果**：输出质量不可预测，无法持续改进。

### 铁律 3：评审先行，修改后行

> 先评审（发现问题），再修改（解决问题）。禁止没有评审反馈的盲目修改。

**违反后果**：修改可能偏离方向，引入新问题。

### 铁律 4：一轮只改一件事

> 每轮迭代只针对评审反馈中的问题进行修改，不主动引入新的改动。

**违反后果**：无法验证修改效果，问题越改越多。

### 铁律 5：先治本，后治标

> 优先解决结构性问题（逻辑、论证、结构），再处理表面问题（措辞、格式、标点）。

**违反后果**：表面光鲜但内核空洞，治标不治本。

### 铁律 6：最小改动原则

> 每次修改只改动必要的内容，不做「顺手优化」。改动的每一段都必须能追溯到评审反馈。

**违反后果**：引入不可控的副作用，破坏已有优点。

### 铁律 7：R1 不通过，R2 不执行

> 内容是基础。如果事实准确性或逻辑完整性有问题，风格评审没有意义。

**违反后果**：浪费评审资源，风格修饰掩盖内容缺陷。

### 铁律 8：评级来自 R4，路由来自规则

> 综合评级只能由 R4 基于评审报告给出，路由决策只能基于评级标准和路由规则做出。主Agent 不得自行判断。

**违反后果**：评级不一致，路由混乱，质量失控。

### 铁律 9：版本号是生命线

> 每个版本必须有唯一的版本号，修改必须递增版本号。禁止覆盖已有版本。

**违反后果**：无法追溯修改历史，无法对比版本差异。

### 铁律 10：经验必须基于事实

> E1 提取的每条经验必须引用具体的评审报告和文章版本，不能凭空臆造。

**违反后果**：规则进化失去事实基础，规则质量下降。

### 铁律 11：规则变更必须验证

> A1 的每条规则变更必须经过范文验证和边界测试，禁止未经验证的规则上线。

**违反后果**：新规则可能矫枉过正或与现有规则冲突。

### 铁律 12：写作前必问不可跳过

> W1 在调用 khazix-writer Skill 之前，必须确认写作前必问的 5 项信息。缺少硬性前置条件时不得撰写。

**违反后果**：初稿方向偏差，后续迭代成本极高。

### 铁律 13：风格锚点不可偏移

> 所有创作活动必须服务于「有见识的普通人在认真聊一件打动他的事」这个风格定位。任何偏离这个定位的内容都应被修正。

**违反后果**：文章失去辨识度，读者无法建立预期。

### 铁律 14：退出条件是红线

> 迭代退出条件（P0/P1/P2）是硬性红线，不可突破。满 5 轮必须退出，连续 2 轮无提升必须退出。

**违反后果**：无限循环，资源浪费，边际收益递减。

### 铁律 15：配图是文章的第二语言（KZCQL 新增）

> 公众号长文不是纯文字产品。配图（封面图、文中配图、架构图、数据可视化图）是文章表达体系的重要组成部分，与文字同等重要。好的配图能让复杂概念一目了然，让抽象论证具象可感。

**违反后果**：文章信息密度下降，读者阅读疲劳，传播力减弱。

### 铁律 16：khazix-writer Skill 是写作引擎但不是写作标准（KZCQL 新增）

> khazix-writer Skill 是 W1 撰写初稿的核心引擎，但 Skill 的输出不是最终标准。Skill 产出的初稿必须经过完整的评审闭环（R1→R2→R4），任何问题都应通过迭代修改解决，而非归咎于 Skill。

**违反后果**：过度依赖 Skill 输出，放弃质量把关，或因 Skill 输出不完美而否定整个系统。

### 铁律 17：方法论必须有降级动作和边界声明（汲取评论部）

> 方法论/工具分享类内容必须包含「做不动时怎么办」的降级方案（至少2条）和「适用前提」的边界声明（至少2条）。不允许存在「放之四海而皆准」的方法论。

**违反后果**：读者遇到困难时无路可退，方法沦为空洞口号。

### 铁律 18：事实核查需双来源验证，注意事实漂移（汲取资讯部）

> 关键字段（时间/政策/入口/对象/门槛）必须至少双来源交叉验证。不仅要检查完全虚构的事实，还要检查微小偏差（如将「预计Q2」写为「6月」）。

**违反后果**：微小的事实偏差积累起来会严重损害文章可信度。

### 铁律 19：去AI感不等于去修辞，矫枉过正同样有问题（汲取评论部）

> 「去AI感」的目标是消除机械感，不是消除所有修辞。合格的卡兹克风格文章应包含适度的排比（2-4处）、递进（1-2处）、倒装/倒叙（1-2处）。修辞真空（全文零排比、零递进、零倒装）同样需要修正。

**违反后果**：文章失去文学性和节奏感，读起来像白开水。

### 铁律 20：主题锚点必须落地，核心概念>=3次且有叙事支撑（汲取内容部v2）

> 文章标题核心概念在正文中至少出现3次，每次有叙事支撑，分布均匀（开头/中段/结尾各至少1次）。防止主题「悬空」——标题说的和正文聊的不是同一件事。

**违反后果**：读者读完文章觉得「标题党」，核心观点没有真正展开。

### 铁律 21：评审建议必须采用「问题-做法-效果」三段式（汲取内容部v2）

> R2/R3 的每条改进建议必须包含三个要素：问题（具体描述哪里不好）、做法（建议怎么改）、效果（改完后预期改善什么）。禁止只说「这里不好」不给方案。

**违反后果**：W2 无法理解修改意图，迭代效率低下。

### 铁律 22：修改前标注预期效果，修改后验证（汲取内容部v2）

> W2 在执行修改前必须标注预期提升的维度和分数，修改后必须验证实际提升是否达到预期。如果未达预期，必须在修改说明中分析原因。

**违反后果**：修改缺乏目标感，无法衡量迭代效果，陷入盲目修改。

## 三、迭代策略

### 3.1 先治本后治标

修改优先级从高到低：

| 优先级 | 问题类型 | 示例 | 修改策略 |
|--------|----------|------|----------|
| P0 | 事实错误 | 数据引用错误、时间线错误 | 立即修正，不可妥协 |
| P1 | 逻辑缺陷 | 论证链条断裂、结论无支撑 | 补充论证或调整论证结构 |
| P2 | 结构问题 | 段落顺序不合理、缺少过渡 | 调整段落顺序或补充过渡 |
| P3 | 内容深度 | 分析停留在表面、缺少案例 | 补充案例或深化分析 |
| P4 | 风格问题 | 人称不一致、语言过于学术 | 调整语言风格 |
| P5 | 格式问题 | 标点不规范、段落过长 | 调整格式 |

**执行规则**：
- 同一轮迭代中，P0-P2 的问题必须全部解决
- P3 的问题应尽量解决，但可以分多轮处理
- P4-P5 的问题可以在 P0-P3 解决后再处理
- 禁止在 P0-P2 未解决的情况下处理 P4-P5

### 3.2 一轮只改一件事

**原则**：每轮迭代聚焦一个修改方向，避免「全面开花」。

**执行方式**：

| 轮次 | 修改方向 | 预期效果 |
|------|----------|----------|
| 第1轮 | 解决 P0 事实错误 | 消除硬伤 |
| 第2轮 | 解决 P1 逻辑缺陷 | 论证完整 |
| 第3轮 | 解决 P2 结构问题 | 结构清晰 |
| 第4轮 | 解决 P3 内容深度 | 分析深入 |
| 第5轮 | 解决 P4-P5 风格和格式 | 打磨细节 |

**例外**：如果某轮评审中只有 P4-P5 的问题（P0-P3 全部通过），可以合并处理。

### 3.3 最小改动原则

**原则**：每次修改只改动必要的内容，不做「顺手优化」。

**执行规则**：
1. 修改前，明确列出本轮要解决的问题清单
2. 修改时，只针对清单中的问题进行修改
3. 修改后，逐项核对清单，确认每个问题都已处理
4. 如果修改过程中发现了新的问题，记录下来但不立即修改（留到下一轮）

**修改说明模板**：

```markdown
## 修改说明

### 本轮修改范围
仅针对以下评审问题进行修改：
1. R1-严重1：[问题描述]
2. R2-一般2：[问题描述]

### 未修改的问题（留待下轮）
1. R1-一般3：[问题描述] — 原因：与本轮修改方向不同
2. R2-轻微1：[问题描述] — 原因：优先级较低

### 修改详情
| 评审来源 | 问题描述 | 处理方式 | 修改位置 |
|----------|----------|----------|----------|
| R1-严重1 | [描述] | [已修改] | 第X段 |
| R2-一般2 | [描述] | [已修改] | 标题 |
```

## 四、khazix-writer Skill 集成规范

### 4.1 何时调用

khazix-writer Skill 在以下场景中被调用：

| 场景 | 调用者 | 触发条件 | 说明 |
|------|--------|----------|------|
| 初稿撰写 | W1 | 写作前必问完成后 | 标准调用场景 |
| 重写 | W1 | 评级为 C/D/F 时 | 需要重新传递写作需求 |
| 大幅扩写 | W2 | 需要补充大量内容时 | 需要明确扩写范围和方向 |

**禁止调用场景**：

| 场景 | 原因 |
|------|------|
| 微调/修改 | 微调应由 W2 直接执行，不需要 Skill |
| 标题优化 | 标题优化应由 W2 基于评审反馈直接调整 |
| 格式调整 | 格式调整不需要 Skill |

### 4.2 如何传递创作规范

W1 在调用 khazix-writer Skill 时，必须将以下信息作为上下文传递：

```
khazix-writer Skill 调用上下文
│
├── 1. 写作需求
│   ├── 文章主题
│   ├── 目标读者
│   ├── 核心观点/结论
│   ├── 参考资料/素材
│   └── 特殊要求
│
├── 2. 用户身份档案（摘要）
│   ├── 作者身份定位
│   ├── 语言风格偏好
│   └── 禁忌话题/表达方式
│
├── 3. 创作规范（关键规则摘要）
│   ├── 文章风格定位：「有见识的普通人在认真聊一件打动他的事」
│   ├── 结构规范：开头引入 → 核心论证 → 结尾升华
│   ├── 字数规范：3000-5000字
│   ├── 人称规范：统一第一人称
│   └── 配图规范：至少1张封面图 + 2张文中配图
│
└── 4. 文章类型原型（如适用）
    ├── 观点文章
    ├── 深度解析
    ├── 经验分享
    ├── 行业观察
    └── 人物特写
```

**传递原则**：
- 不要将所有规则文件全文传递（上下文过长会影响 Skill 效果）
- 传递关键规则的摘要，而非完整文本
- 如果用户有特殊要求，放在最前面（优先级最高）
- 创作规范的传递应聚焦于「约束条件」，而非「操作指南」

### 4.3 Skill 输出的四层自检体系

khazix-writer Skill 自身带有四层自检体系。W1 在收到 Skill 输出后，应确认以下四层自检的结果：

| 层级 | 自检内容 | W1 的处理 |
|------|----------|-----------|
| 第1层 | 基础格式检查（标题、段落、字数） | 如不通过，要求 Skill 重新生成 |
| 第2层 | 内容完整性检查（是否有完整开头/正文/结尾） | 如不通过，要求 Skill 补充 |
| 第3层 | 风格一致性检查（人称、语言风格是否统一） | 如不通过，要求 Skill 调整 |
| 第4层 | 创作规范合规检查（是否符合传递的规范要求） | 如不通过，记录问题，进入评审闭环 |

**关键原则**：
- 第1-3层的问题，W1 应要求 Skill 重新生成或调整，**不进入评审闭环**
- 第4层的问题，W1 不应自行修改，而是将 Skill 输出作为初稿直接进入评审闭环
- 这是因为第4层的问题（如论证深度、信息密度）需要 R1/R2 的专业评审，W1 不应越权判断

### 4.4 Skill 与评审闭环的配合

```
khazix-writer Skill                    评审闭环
┌──────────────────┐              ┌──────────────────┐
│                  │              │                  │
│  四层自检         │              │  R1→R2→R4        │
│  第1层：格式      │──通过──→     │  内容评审         │
│  第2层：完整性    │              │  风格评审         │
│  第3层：风格      │              │  综合评级         │
│  第4层：规范合规  │──不通过──→   │                  │
│                  │              │  评审反馈         │
└──────────────────┘              │      │           │
                                  │      ▼           │
                                  │  W2 迭代修改      │
                                  │  （不调用 Skill）  │
                                  │      │           │
                                  │      ▼           │
                                  │  再次评审         │
                                  │  （循环直到达标）  │
                                  └──────────────────┘
```

**配合要点**：
1. Skill 负责「从 0 到 1」的初稿生成
2. 评审闭环负责「从 1 到 100」的质量提升
3. Skill 的输出质量决定了迭代起点，但不决定最终质量
4. 迭代修改由 W2 直接执行，不需要回调 Skill（除非需要大幅重写）

## 五、前置规则对迭代效率的影响

### 5.1 核心洞察

> **前置规则的质量直接决定了迭代效率。** 前置规则越完善，初稿质量越高，需要的迭代轮次越少。

以下数据基于 KZCQL 系统的实战统计（模拟数据，用于说明趋势）：

### 5.2 实战数据表格

| 前置规则完善度 | 初稿平均评级 | 平均迭代轮次 | 平均耗时（相对值） | 最终评级达标率 |
|----------------|-------------|-------------|-------------------|---------------|
| 低（仅基础结构规则） | C (6.2) | 4.2 轮 | 100% | 65% |
| 中低（+语言规范） | B- (6.8) | 3.5 轮 | 85% | 75% |
| 中（+配图规范） | B (7.3) | 2.8 轮 | 70% | 82% |
| 中高（+引用规范+事实核查） | B+ (7.8) | 2.1 轮 | 55% | 90% |
| 高（完整规则体系） | B (8.1) | 1.3 轮 | 35% | 95% |

### 5.3 关键发现

**发现 1：前置规则每增加一个维度，平均迭代轮次减少 0.5-0.7 轮**

```
迭代轮次 vs 前置规则维度数

5轮 │ ●
    │
4轮 │   ●
    │
3轮 │     ●
    │
2轮 │       ●
    │
1轮 │         ●
    └──────────────────
      2   4   6   8  10
         前置规则维度数
```

**发现 2：事实核查类前置规则对迭代效率的影响最大**

| 前置规则类型 | 对初稿评级的提升 | 对迭代轮次的减少 |
|-------------|-----------------|-----------------|
| 事实核查规范 | +0.5 分 | -0.7 轮 |
| 结构规范 | +0.3 分 | -0.5 轮 |
| 语言规范 | +0.2 分 | -0.3 轮 |
| 配图规范 | +0.3 分 | -0.4 轮 |
| 引用规范 | +0.4 分 | -0.6 轮 |
| 人称规范 | +0.1 分 | -0.2 轮 |

**发现 3：前置规则缺失导致的常见迭代陷阱**

| 前置规则缺失 | 导致的初稿问题 | 迭代成本 |
|-------------|---------------|----------|
| 无事实核查规范 | 数据引用错误、来源不可验证 | 需要至少 2 轮迭代修复 |
| 无结构规范 | 文章缺少清晰的结构框架 | 需要至少 1 轮重写 |
| 无人称规范 | 人称频繁切换，风格混乱 | 需要至少 1 轮全文修改 |
| 无配图规范 | 文章纯文字，信息密度低 | 需要至少 1 轮补充配图 |
| 无引用规范 | 引用格式混乱，缺少来源标注 | 需要至少 1 轮逐条修正 |

### 5.4 规则投资回报分析

| 投资方向 | 投入（规则编写时间） | 回报（每篇文章节省的迭代时间） | ROI |
|----------|---------------------|---------------------------|-----|
| 事实核查规范 | 高 | 高 | 最高 |
| 结构规范 | 中 | 中 | 高 |
| 引用规范 | 中 | 中高 | 高 |
| 配图规范 | 中 | 中 | 中高 |
| 语言规范 | 低 | 低中 | 中 |
| 人称规范 | 低 | 低 | 中 |

**结论**：优先投资事实核查规范和结构规范，这两类规则的 ROI 最高。

## 六、5 种文章类型原型

KZCQL 支持 5 种公众号长文原型，每种原型有特定的结构要求和评审侧重。

### 6.1 原型一览

| 原型 | 核心特征 | 典型结构 | 评审侧重 |
|------|----------|----------|----------|
| **观点文章** | 有明确的立场和论证 | 现象引入 → 立场阐述 → 论证展开 → 回应反驳 → 结论 | 论证深度、逻辑完整性 |
| **深度解析** | 对某个话题的深入分析 | 背景介绍 → 多维度分析 → 数据支撑 → 趋势判断 | 信息密度、数据准确性 |
| **经验分享** | 基于个人经历的分享 | 故事引入 → 经验总结 → 方法论提炼 → 行动建议 | 可操作性、真实性 |
| **行业观察** | 对行业趋势的观察和判断 | 行业现状 → 关键变化 → 深层原因 → 未来预判 | 洞察力、前瞻性 |
| **人物特写** | 对人物或团队的深度描写 | 人物引入 → 关键事件 → 性格刻画 → 价值提炼 | 叙事能力、细节真实性 |

### 6.2 原型选择规则

W1 在撰写前，应根据用户需求选择最匹配的原型：

| 用户需求关键词 | 推荐原型 |
|---------------|----------|
| 「我认为」「我的看法」「观点」 | 观点文章 |
| 「分析」「解读」「为什么」 | 深度解析 |
| 「我做了」「我的经验」「踩坑」 | 经验分享 |
| 「行业」「趋势」「未来」 | 行业观察 |
| 「人物」「故事」「他/她」 | 人物特写 |

如果用户需求无法明确匹配任何原型，默认使用「观点文章」原型。

## 七、经验教训模板

### 7.1 创作后复盘模板

每次创作流程结束后，E1 应使用以下模板进行复盘：

```markdown
# 创作复盘 - [文章标题]

## 基本信息
- 文章标题：[标题]
- 文章原型：[观点文章/深度解析/经验分享/行业观察/人物特写]
- 最终评级：[S/A/B/C/D/F]
- 迭代轮次：N 轮
- 总耗时：约 N 分钟

## 做得好的地方
1. [具体描述]
2. [具体描述]

## 需要改进的地方
1. [具体描述] — 根因：[分析]
2. [具体描述] — 根因：[分析]

## 经验条目提取
- EXP-[YYYYMMDD]-001：[经验条目摘要]
- EXP-[YYYYMMDD]-002：[经验条目摘要]

## 规则变更建议
- 建议 1：[具体建议]
- 建议 2：[具体建议]

## 下次创作注意事项
1. [具体注意事项]
2. [具体注意事项]
```

### 7.2 经验教训分类

| 分类 | 说明 | 示例 |
|------|------|------|
| **选题经验** | 什么样的选题适合公众号长文 | 「结合热点但不过度蹭热点，有长期价值的选题效果更好」 |
| **结构经验** | 什么样的结构读者更容易跟上 | 「先抛结论再展开论证，比先铺垫再揭晓结论的完读率高 30%」 |
| **语言经验** | 什么样的表达方式更受欢迎 | 「口语化的比喻比学术化的定义更容易被理解和传播」 |
| **配图经验** | 什么样的配图能提升阅读体验 | 「流程图比纯文字描述的理解效率高 3 倍」 |
| **迭代经验** | 什么样的修改策略最有效 | 「先修逻辑再修语言，比反过来效率高 50%」 |
| **评审经验** | 什么样的评审标准最有区分度 | 「信息密度比字数更能预测文章质量」 |
