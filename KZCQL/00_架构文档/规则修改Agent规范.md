## 角色职责边界

### A1（架构审查专家）职责

A1是规则修改流程的**架构评估者**，负责：
- **架构评估**：评估规则变更对系统架构的影响
- **方案设计**：设计规则变更的整体方案
- **传播验证**：验证规则变更的传播范围（前后置一致性、权重一致性、路由一致性）
- **边界测试**：设计并执行边界测试
- **审批把关**：对需要审批的修改类型进行把关

**A1不直接执行**：规则文件的具体编辑操作（由E1执行）

### E1（规则修改专员）职责

E1是规则修改流程的**执行者**，负责：
- **经验提取**：从评审报告中提取经验条目
- **规则修改执行**：根据A1设计的方案，实际执行规则文件的编辑操作
- **变更记录**：创建规则变更记录
- **文件更新**：更新所有受影响的规则文件

**E1不直接负责**：架构评估、方案设计、边界测试设计（由A1负责）

### 协作流程

```
评审报告 → E1提取经验 → A1评估并设计方案 → E1执行修改 → A1验证 → 完成
                ↑___________________________↓
                        （迭代反馈）
```

| 阶段 | A1职责 | E1职责 |
|------|--------|--------|
| 经验提取 | - | 提取经验条目 |
| 缺口分析 | 评估架构影响 | 提供经验详情 |
| 方案设计 | 设计变更方案 | - |
| 范文验证 | 设计验证方法 | 执行验证 |
| 边界测试 | 设计测试用例 | 执行测试 |
| 规则修改 | 审批 | **执行文件编辑** |
| 传播验证 | **验证一致性** | 配合验证 |
| 变更记录 | 审核 | **创建记录** |

## 一、规则设计四大原则

每条规则（无论是新增还是修改）必须同时满足以下四个原则。A1 在设计规则变更方案时，必须逐条验证。

### 原则一：具体可量化

> **规则必须能用「是/否」或「数值」来判定，不能依赖主观感受。**

**合格示例**：
- 「文章标题长度应控制在 15-25 个字之间」— 可用字数统计验证
- 「每个段落的字数不应超过 200 字」— 可用字数统计验证
- 「文章中引用的数据必须标注来源」— 可检查是否存在来源标注
- 「小标题层级不超过 3 级」— 可检查 Markdown 标题层级

**不合格示例**：
- 「标题要有吸引力」— 「吸引力」无法量化
- 「语言要生动」— 「生动」无法量化
- 「论证要有深度」— 「深度」无法量化
- 「文章要有灵魂」— 「灵魂」无法量化

**量化转换技巧**：

| 模糊表述 | 量化转换 |
|----------|----------|
| 标题有吸引力 | 标题包含具体数字/悬念/反差/疑问句式 |
| 语言生动 | 每千字至少包含 1 个比喻或类比 |
| 论证有深度 | 每个核心论点至少有 2 层支撑论证 |
| 段落太长 | 单段不超过 200 字 |
| 过渡自然 | 相邻段落之间必须有逻辑连接词或语义承接 |

### 原则二：有判定示例

> **每条规则必须附带至少 2 个示例：1 个合规示例 + 1 个违规示例。**

**示例格式**：

```markdown
### 规则：文章标题长度应控制在 15-25 个字之间

**合规示例**：
> 「为什么90%的创业公司活不过三年」
> 字数：17字 ✓

**违规示例**：
> 「关于创业公司存活率的一些个人观察和深度思考」
> 字数：22字 ✓（字数合规，但标题过于平淡，违反了「标题应包含具体数字或悬念」的规则）

**违规示例（字数不合规）**：
> 「我」
> 字数：1字 ✗（过短）
```

**示例来源优先级**：
1. 实际创作中的真实案例（最佳）
2. 公众号领域的知名文章案例
3. 构造的典型案例（最低优先级，但必须真实可信）

### 原则三：有豁免条件

> **每条规则必须明确在什么情况下可以豁免，防止规则被机械执行。**

**豁免条件设计原则**：
- 豁免条件必须比规则本身更具体
- 豁免条件不能过于宽泛（否则规则形同虚设）
- 豁免条件必须有明确的触发场景

**示例**：

```markdown
### 规则：每个段落的字数不应超过 200 字

**豁免条件**：
1. 引用他人原话的段落（引用内容不计入字数限制）
2. 列表型段落（每项不超过 50 字即可）
3. 文章开头的故事性引入段落（允许放宽至 300 字，但仅限第 1-2 段）
```

### 原则四：防矫枉过正验证

> **每条规则变更必须验证不会导致矫枉过正——即不会因为严格执行该规则而破坏文章质量。**

**验证方法**：

| 验证维度 | 验证问题 | 判定标准 |
|----------|----------|----------|
| 过度约束 | 如果 100% 严格执行这条规则，是否会导致文章变得机械/僵化？ | 如果是，需要增加豁免条件 |
| 规则冲突 | 这条规则是否与现有规则存在冲突？ | 如果是，需要明确优先级或修改其中一条 |
| 边界模糊 | 在规则的边界情况下，判定是否清晰？ | 如果不清晰，需要补充边界案例 |
| 例外遗漏 | 是否存在合理的例外情况未被豁免条件覆盖？ | 如果是，需要补充豁免条件 |

**矫枉过正案例**：

> 规则：「文章中禁止使用任何被动语态」
> 矫枉过正：导致技术类文章中无法准确描述实验过程（「实验结果被记录」比「我们记录了实验结果」更客观）
> 修正：改为「非技术类文章中避免使用被动语态；技术类文章中被动语态比例不超过 10%」

## 二、规则分类体系

KZCQL 的规则分为两大类：前置撰写规则和后置评审规则。

### 2.1 前置撰写规则

前置撰写规则在写作前加载，指导 W1（通过 khazix-writer Skill）的撰写行为。

| 子类 | 说明 | 示例 |
|------|------|------|
| **标题规范** | 标题的长度、格式、风格要求 | 「标题应包含具体数字或悬念」 |
| **结构规范** | 文章的整体结构要求 | 「文章必须包含开头引入、核心论证、结尾升华三部分」 |
| **语言规范** | 语言风格、用词、句式要求 | 「避免使用学术化表达，优先使用口语化表达」 |
| **人称规范** | 人称使用的一致性要求 | 「全文统一使用第一人称，禁止突然切换为第三人称」 |
| **配图规范** | 配图的类型、数量、位置要求 | 「文章必须包含至少 1 张封面图和 2 张文中配图」 |
| **架构图规范** | 架构图的类型、格式要求 | 「涉及流程说明时必须配流程图」 |
| **引用规范** | 引用的格式、来源标注要求 | 「引用数据必须标注来源和日期」 |
| **字数规范** | 文章总字数和各部分字数要求 | 「文章总字数控制在 3000-5000 字」 |

**前置撰写规则的存储位置**：

### 2.2 后置评审规则

后置评审规则在评审时加载，指导 R1/R2/R3/R4 的评审行为。

| 子类 | 适用Agent | 说明 | 示例 |
|------|-----------|------|------|
| **内容评审规则** | R1 | 事实准确性、逻辑完整性、论证深度、信息密度 | 「数据引用必须有可验证的来源」 |
| **风格评审规则** | R2 | 语言风格、标题质量、可读性、配图适配度 | 「标题不能使用夸张的感叹号」 |

## 四、规则变更传播要求

### 4.1 传播范围

当一条规则被修改时，A1 必须评估该修改的传播范围，并确保所有受影响的文件同步更新。

**传播检查清单**：

| 变更类型 | 可能受影响的文件 | 检查动作 |
|----------|-----------------|----------|
| 前置撰写规则修改 | W1 撰写规范、R1/R2 评审规范 | 确认评审规则与撰写规则一致 |
| 评审维度修改 | R4 评级规范（权重分配） | 确认权重总和仍为 100% |
| 评级标准修改 | 路由决策树 | 确认路由阈值与评级标准一致 |
| 新增规则类型 | 检查清单 | 确认检查清单覆盖新增规则 |
| 删除规则 | 经验教训（引用该规则的条目） | 标记相关经验条目为「规则已废弃」 |

### 4.2 传播验证

A1 在完成规则变更后，必须执行以下传播验证：

```
规则变更完成
    │
    ├── 1. 检查前置撰写规则与后置评审规则的一致性
    │   └── 如果撰写规则要求 X，评审规则是否检查 X？
    │
    ├── 2. 检查评审维度与权重的一致性
    │   └── 如果新增了评审维度，权重是否重新分配？
    │
    ├── 3. 检查评级标准与路由规则的一致性
    │   └── 如果评级标准变化，路由阈值是否相应调整？
    │
    └── 4. 检查检查清单的完整性
        └── 如果新增了规则，检查清单是否覆盖？
```

## 五、规则修改流程

A1 的规则修改遵循以下 6 步流程：

### 步骤 1：解析反馈

**输入**：E1 输出的经验条目列表

**动作**：
1. 逐条阅读 E1 的经验条目
2. 提取每条经验中的「建议规则变更方向」
3. 将经验条目按问题类型分类（标题类/结构类/语言类/配图类/引用类/...）
4. 识别是否存在多条经验指向同一条规则（合并处理）

**输出**：经验条目分析摘要

### 步骤 2：分析缺口

**输入**：经验条目分析摘要 + 当前规则文件

**动作**：
1. 对比经验条目与当前规则，识别「规则缺口」：
   - 缺口类型 A：存在经验指出的问题，但当前没有对应规则 → 需要新增规则
   - 缺口类型 B：存在对应规则，但规则不够具体/不够量化 → 需要修改规则
   - 缺口类型 C：存在对应规则，但豁免条件不足 → 需要补充豁免条件
   - 缺口类型 D：多条规则之间存在冲突 → 需要调整规则优先级
2. 对每个缺口进行严重程度评估（高/中/低）

**输出**：规则缺口分析报告

### 步骤 3：设计方案

**输入**：规则缺口分析报告

**动作**：
1. 针对每个缺口，设计规则变更方案
2. 每个方案必须包含：
   - 变更类型（新增/修改/删除）
   - 变更内容（具体规则文本）
   - 判定示例（至少 2 个）
   - 豁免条件（如适用）
   - 防矫枉过正验证结果
3. 评估方案的影响范围

**输出**：规则变更方案（草案）

### 步骤 4：范文验证

**输入**：规则变更方案（草案）+ 历史文章样本

**动作**：
1. 从 `05_反馈演进/` 中选取 3-5 篇历史文章作为样本
2. 用新规则对历史文章进行「回溯评审」
3. 检查新规则是否能正确识别历史文章中的问题
4. 检查新规则是否会对历史文章产生「误判」（即把好的内容判定为违规）

**输出**：范文验证报告

### 步骤 5：边界测试

**输入**：规则变更方案（草案）+ 范文验证报告

**动作**：
1. 构造规则的边界案例（至少 3 个）：
   - 刚好合规的案例
   - 刚好违规的案例
   - 模糊地带的案例
2. 用新规则对边界案例进行判定
3. 检查判定结果是否符合预期
4. 如果不符合预期，调整规则文本或豁免条件

**边界测试模板**：

```markdown
### 边界测试：[规则名称]

| 案例 | 描述 | 预期判定 | 实际判定 | 是否一致 |
|------|------|----------|----------|----------|
| 边界案例1 | [描述] | 合规/违规 | 合规/违规 | 是/否 |
| 边界案例2 | [描述] | 合规/违规 | 合规/违规 | 是/否 |
| 边界案例3 | [描述] | 合规/违规 | 合规/违规 | 是/否 |

### 测试结论
[通过/未通过]
[如未通过，说明调整方案]
```

**输出**：边界测试报告

### 步骤 6：输出方案

**输入**：规则变更方案（草案）+ 范文验证报告 + 边界测试报告

**动作**：
1. 汇总所有验证结果
2. 如验证全部通过，输出最终规则变更方案
3. 如验证未通过，返回步骤 3 重新设计
4. 执行规则变更（修改规则文件）
5. 创建变更记录

**输出**：
- 最终规则变更方案
- 更新后的规则文件
- 变更记录

## 六、E1 经验提取规范

### 6.1 提取流程

E1 的经验提取遵循以下流程：

```
E1 经验提取流程
│
├── 1. 读取本轮所有评审报告（R1/R2/R3/R4）
├── 2. 读取 W2 修改记录（如有）
├── 3. 读取用户反馈（如有）
│
├── 4. 逐条分析评审问题
│   ├── 问题是否反复出现？（跨多轮/跨多篇文章）
│   ├── 问题的根因是什么？（规则缺失/规则不够具体/执行偏差）
│   └── 问题是否可以通过规则变更来预防？
│
├── 5. 提取经验条目
│   └── 每条经验必须包含：
│       ├── 场景描述（在什么情况下出现的问题）
│       ├── 问题类型（标题/结构/语言/配图/引用/...）
│       ├── 根因分析（为什么会出现这个问题）
│       └── 建议规则变更方向（应该新增/修改/补充什么规则）
│
└── 6. 去重与合并
    └── 删除与现有经验条目重复的条目
```

### 6.2 经验条目格式

```markdown
## 经验条目：EXP-[YYYYMMDD]-[序号]

### 场景描述
[在什么情况下出现的问题，尽可能具体]

### 问题类型
[标题类/结构类/语言类/配图类/引用类/逻辑类/...]

### 根因分析
[为什么会出现这个问题，分析到规则层面]

### 建议规则变更方向
[应该新增/修改/补充什么规则，给出具体建议]

### 来源
- 评审报告：[R1/R2/R3 报告编号]
- 文章版本：[版本号]
- 出现频次：[首次出现/第N次出现]
```

### 6.3 提取原则

| 原则 | 说明 |
|------|------|
| 基于事实 | 每条经验必须引用具体的评审报告和文章版本，不能凭空臆造 |
| 根因导向 | 不只记录「出现了什么问题」，更要分析「为什么会出现」 |
| 可操作 | 建议的规则变更方向必须具体到可以转化为规则文本 |
| 去重合并 | 相同根因的问题只提取一条经验，但标注出现频次 |
| 正负兼顾 | 不只提取失败经验，也要提取成功经验（「什么做法效果好」） |

## 七、规则版本管理

### 7.1 版本号规则

规则文件的版本号采用语义化版本号：

```
v[主版本].[次版本].[修订号]

主版本：规则体系发生重大变更（如新增评审维度、修改评级标准）
次版本：新增规则或修改现有规则
修订号：修正规则文本（错别字、表述优化、补充示例）
```

### 7.2 版本记录

每条规则应包含版本记录：

```markdown
### 规则：[规则名称]

**版本**：v1.2.0
**最后更新**：2026-04-23
**变更历史**：
- v1.2.0 (2026-04-23)：补充豁免条件，增加边界案例
- v1.1.0 (2026-03-15)：修改量化标准，从「不超过150字」调整为「不超过200字」
- v1.0.0 (2026-02-01)：初始版本
```
