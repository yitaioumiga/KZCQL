## 一、目的

本规范旨在解决架构评估发现的 **D5-2 执行验证映射缺陷**（评分1/15分，严重不合格）：
- 检查执行日志保存路径不明确
- I1督察访问机制未定义
- 日志保留期限未规定
- 无法防止"事后补录"

通过建立标准化的检查执行日志管理体系，确保：
1. 检查执行过程**可追溯、可审计**
2. I1督察能够**独立验证**检查执行的真实性
3. 主Agent的检查执行处于**客观监督**之下

---

## 二、日志存储规范

### 2.1 存储路径

```
/workspace/KZCQL/04_工作区/检查日志/
└── {文章关键词}/
    ├── 检查日志_{YYYYMMDD}_{HHMMSS}_{PID}.md
    └── 检查结果_{YYYYMMDD}_{HHMMSS}_{PID}.json
```

**示例**:
```
/workspace/KZCQL/04_工作区/检查日志/理财访谈第5篇/
├── 检查日志_20260520_143052_12345.md
└── 检查结果_20260520_143052_12345.json
```

### 2.2 命名规范

| 组件 | 格式 | 示例 |
|------|------|------|
| 时间戳 | `YYYYMMDD_HHMMSS` | `20260520_143052` |
| 进程ID | `$$` (Bash自动获取) | `12345` |
| Session ID | `{时间戳}_{PID}` | `20260520_143052_12345` |

### 2.3 文件格式

#### Markdown日志文件 (.md)

包含以下内容：
- 检查时间戳
- Session ID
- 文章信息（关键词、文件路径）
- 每项检查的详细过程和结果
- 检查摘要统计

#### JSON摘要文件 (.json)

结构化数据，便于程序解析：
```json
{
  "session_id": "20260520_143052_12345",
  "timestamp": "2026-05-20T14:30:52+08:00",
  "article_key": "理财访谈第5篇",
  "md_file": "/workspace/.../理财访谈第5篇_v4.md",
  "total_checks": 4,
  "pass_checks": 4,
  "fail_checks": 0,
  "pass_rate": 100.0,
  "overall_result": "PASS",
  "log_file": "/workspace/.../检查日志_20260520_143052_12345.md"
}
```

## 三、日志保留期限

### 3.1 保留策略

| 日志类型 | 保留期限 | 清理策略 |
|----------|----------|----------|
| 检查日志 (.md) | **永久保留** | 不自动清理 |
| 检查结果 (.json) | **永久保留** | 不自动清理 |

### 3.2 理由

- **审计需求**：I1督察可能需要追溯历史检查记录
- **事故调查**：如发生交付事故，需要检查日志作为证据
- **趋势分析**：长期保留便于分析检查通过率和常见问题

## 四、I1督察访问机制

### 4.1 访问权限

I1督察智能体拥有对以下目录的**只读访问权限**：
```
/workspace/KZCQL/04_工作区/检查日志/
```

### 4.2 审计职责

I1督察在联合督查或定向审查时，必须：

1. **随机抽查**：每次督查至少抽查1篇最近交付文章的检查日志
2. **交叉验证**：比对检查日志与主Agent声称的检查结果是否一致
3. **时间戳验证**：检查Session ID的时间戳是否与交付时间逻辑一致
4. **内容审计**：检查日志内容是否完整、是否有明显伪造痕迹

### 4.3 审计检查清单

```markdown
- [ ] 检查日志文件存在（.md + .json）
- [ ] Session ID时间戳与交付时间逻辑一致
- [ ] 检查项数量与规范一致（至少IMG-01, IMG-01b, IMG-02, FMT-01）
- [ ] 每项检查有详细的工具输出记录
- [ ] JSON摘要与Markdown日志内容一致
- [ ] 无"事后补录"迹象（时间戳异常、内容矛盾等）
```

## 五、防篡改机制

### 5.1 技术措施

1. **独立执行**：`pre_delivery_check.sh` 由系统自动执行，主Agent不可修改脚本
2. **系统时间戳**：使用系统时间生成Session ID，防止手动伪造
3. **进程隔离**：每个检查会话有独立的PID，日志中记录
4. **只读存储**：检查日志目录对主Agent只读（写入后不可修改）

### 5.2 流程措施

1. **交付前强制检查**：主Agent交付前必须执行脚本，否则阻断交付
2. **日志作为交付附件**：检查日志路径必须在交付报告中注明
3. **I1随机审计**：I1定期审计检查日志，发现异常触发深度调查

## 六、使用流程

### 6.1 主Agent使用流程

```bash
# 1. 在交付前执行检查脚本
bash /workspace/KZCQL/03_执行代码/交付前检查/pre_delivery_check.sh \
  /workspace/理财访谈系列/03_文章产出/稿件/理财访谈第5篇_v4.md \
  "理财访谈第5篇"

# 2. 检查脚本返回码
#    - 返回 0: 所有检查通过，允许交付
#    - 返回 1: 存在检查失败，禁止交付

# 3. 在交付报告中引用检查日志路径
```

**Session ID**: `20260520_143052_12345`

## 七、违规处理

### 7.1 违规行为

| 行为 | 等级 | 处理 |
|------|------|------|
| 未执行pre_delivery_check.sh就交付 | P0 | 视为生产事故，触发联合督查 |
| 检查失败后仍强行交付 | P0 | 视为严重违规，暂停交付权限 |
| 伪造检查日志 | P0 | 视为严重违规，启动深度调查 |
| 检查日志路径未在交付报告中注明 | P1 | 要求补正，记录违规 |
