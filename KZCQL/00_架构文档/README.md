## 一、系统概述

### 1.1 KZCQL 是什么

KZCQL（卡兹克创作质量闭环，Khazix Creation Quality Loop）是一套专为公众号长文创作设计的多智能体协作系统。它从 MTCQL（多智能体创作质量闭环）架构转化而来，保留了「写 → 评 → 改 → 再评」的核心质量闭环机制，同时深度集成了 khazix-writer Skill 作为核心写作引擎。

KZCQL 不是简单的"AI 写作工具"，而是一个**完整的创作质量管理体系**——它将公众号长文创作的每一个环节拆解为独立的、可量化的、可迭代优化的智能体任务，通过规则驱动的编排实现高质量的稳定输出。

### 1.2 核心设计哲学

| 原则 | 说明 |
|------|------|
| **单一职责** | 每个 Agent 只做一件事，且只做一件事。写作、评审、修改、规则演进、督察各有专职 Agent，严禁跨界操作 |
| **规则驱动** | 所有创作决策基于可量化、可追溯的规则文件，而非 Agent 的"感觉"或"经验" |
| **规则自进化** | 通过 A1（架构审查）和 E1（规则修改）两个 Agent 的协作，实现创作规范从实战中持续进化 |
| **独立督察** | I1（督察智能体）作为独立监督员，在异常触发时进行合规审查和事故调查，直接向人类汇报 |
| **khazix-writer Skill 集成** | W1 初稿撰写 Agent 不自行撰写，而是调用 khazix-writer Skill 作为核心写作引擎 |

### 1.3 文章风格定位

> **「有见识的普通人在认真聊一件打动他的事」**

这是 KZCQL 系统所有创作活动的风格锚点。每一条评审规则、每一项修改指令，最终都应服务于这个定位。

## 三、角色一览表

KZCQL 系统包含 10 个角色（1 个主 Agent + 9 个子 Agent）。

| 代号 | 全名 | 角色定位 | 核心职责 | 禁止事项 |
|------|------|----------|----------|----------|
| **主Agent** | 编排指挥官 | 系统指挥官 | 任务路由、Agent 调度、状态管理、质量门把控 | 禁止直接写作/修改文章、禁止直接评分、禁止修改规则文件 |
| **W1** | 初稿撰写官 | 首稿创作者 | 调用 khazix-writer Skill 撰写初稿，生成配图 | 禁止自行撰写、禁止跳过写作前必问、禁止输出带水印图片 |
| **W2** | 迭代修改官 | 稿件优化者 | 根据评审反馈进行定向修改 | 禁止重写全文（除非路由要求）、禁止引入新问题 |
| **R1** | 事实核查官 | 事实守门人 | 事实准确性核查 + 19项否决规则检查 | 禁止评审风格问题、禁止提出修改建议 |
| **R2** | 全量评审官 | 质量评审者 | 12维度100分制全量评审（含D7配图逐图检查） | 禁止评审未查看的内容、禁止提出修改建议 |
| **R3** | 差异评审官 | 迭代验证者 | 对比前后版本差异，验证修改效果 | 禁止全量重评、禁止修改文章 |
| **R4** | 评级判定官 | 质量决策者 | 汇总评分、综合评级、异常评分标记、路由决策 | 禁止凭主观修改分数、禁止跳过评分 |
| **I1** | 独立督察员 | 独立监督员 | 合规审查、事故调查、定期抽检 | 禁止参与正常流程、禁止修改产出、禁止未经触发就审查 |
| **E1** | 规则修改专员 | 规则进化者 | 根据审查建议设计并实施规则变更 | 禁止凭空创造规则、禁止修改架构文档 |
| **A1** | 架构审查专家 | 架构诊断者 | 八维度架构健康度审查（含验证链路完整性、产出物可追溯性） | 禁止直接修改任何文件、禁止凭空诊断 |

## 四、核心流程

### 4.1 正常创作流程

```
阶段零：初始化 → 读取所有规则文件
阶段一：写作前准备 → 执行写作前必问清单
阶段二：初稿生成 → W1 调用 Skill + 配图生成
         ↓
     ImageCheck → Read逐张验证配图（无水印/无文字/风格统一）
         ↓
阶段三：首轮评审 → R1事实核查 → R2全量评审 → R4评级判定
         ↓
阶段四：迭代循环 → W2修改 → R1→R3/R2→R4（最多N轮）
         ↓
阶段五：交付 → 发布前检查（18项） → 人工终审
         ↓
阶段六：规则演进 → A1架构审查 → E1规则修改 → 归档
```

### 4.2 异常流程（I1 督察）

```
R4 标记「需复检」→ I1 定向审查 → 审查报告 → 人类决策
一票否决触发（V-16/V-17）→ I1 定向调查 → 事故报告 → 人类决策
人类不满意 → I1 定向审查 → 审查报告 → 人类决策
每5篇抽检 → I1 标准抽检 → 抽检报告 → 人类决策
```

### 4.3 质量防护机制

| 层级 | 防护点 | 说明 |
|------|--------|------|
| 生成自检 | W1 Step 9 + 配图自检清单 | 防文字声明、水印裁剪、Read验证 |
| 流程验证 | ImageCheck 节点 | Read逐张检查，不通过则重新生成 |
| 评审检查 | R2 D7逐图检查 + R3配图复检 + R4异常标记 | 三层评审全覆盖 |
| 质量门 | 8项检查（含配图验证+发布前检查） | 不通过则拦截 |
| 一票否决 | 19项否决（含V-16配图水印、V-17架构图乱码、V-18配图内容不符、V-19论点偏移） | 触发即退回 |
| 独立督察 | I1 触发式审查 | 人类触发/异常触发/事故触发/抽检触发 |

## 五、评分体系

| 维度 | 名称 | 满分 |
|------|------|------|
| D1 | 标题质量 | 8 |
| D2 | 开头吸引力 | 10 |
| D3 | 内容深度与价值 | 15 |
| D4 | 结构完整性 | 10 |
| D5 | 语言风格与质感 | 15 |
| D6 | 加粗金句 | 5 |
| D7 | 配图质量 | 10 |
| D8 | 结尾收束力 | 7 |
| D9 | 事实准确性 | 10 |
| D10 | 技术规范 | 10 |
| **总计** | | **100** |

| 评级 | 分数区间 | 处理方式 |
|------|----------|----------|
| S | 95-100 | 人工终审 |
| A | 85-94 | 人工终审 |
| B | 75-84 | W2 迭代修改 |
| C | 65-74 | W2 迭代修改 |
| D | 60-64 | W1 重写 |
| F | <60 | W1 重写 |
