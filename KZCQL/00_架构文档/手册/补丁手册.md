### 5.3.1 P5：交付前内容质量最终检查

主Agent在向用户交付v1/v2/v3等任何版本前，**必须**亲自通读全文，检查以下三项：

| 检查项 | 检查方法 | 不通过时的处理 |
|--------|----------|----------------|
| **逻辑跳跃/矛盾** | 通读全文，标记是否存在前后矛盾（特别是数字/比例/规则之间） | 返回W2修改，不得交付 |
| **AI味句式** | Grep搜索`不是.*而是`，统计出现次数 | 超过2次返回W2改写 |
| **断链/跳跃** | 检查段落间是否有逻辑断链（观点→操作缺解释、问题→方案缺分析） | 返回W2补充 |

**执行要求**：
- 主Agent必须亲自执行，不得委托子代理
- 检查结果记录在STATE.md中
- 未通过P5检查的版本不得交付给用户
- P5聚焦于文本内容质量检查，交付物完整性检查见P5.1

**检查执行记录规则（SG-2026-0514-02补丁P11 + SG-2026-0515-02补丁P15）**：
- P5、P5.1、交付保障清单、R2评审等所有检查必须记录至STATE.md
- 记录格式：| 检查项 | 执行时间 | 执行结果 | 不通过项 |
- **无STATE.md记录的检查视为未执行（硬性阻断）**
- 主Agent在交付前必须确认STATE.md中包含所有检查记录
- **如STATE.md缺失检查记录，必须补执行后再交付，不得跳过**

**P5检查强制执行规则（SG-2026-0515-02补丁P15）**：
- **事故编号**：SG-2026-0515-02
- **生效日期**：2026-05-15
- **触发事件**：第10篇P5检查未执行，导致D12标准误判未被拦截

交付前强制检查清单：
1. LS确认文章目录下存在STATE.md文件
2. Read确认STATE.md中包含P5检查记录
3. 如STATE.md不存在或P5记录缺失 → **阻断交付**，执行P5后重新检查
4. 只有STATE.md存在且P5记录完整 → 方可进入P5.1和后续交付流程

**阻断规则**：无STATE.md或P5记录缺失，主Agent不得向用户交付任何版本。

### 5.3.1b P5.1：交付物完整性快速检查（SG-2026-0514-02补丁P10）

**事故编号**：SG-2026-0514-02
**生效日期**：2026-05-14
**触发事件**：第3篇v2交付物完整性问题（配图引用缺失、docx未生成、配图目录混乱）

主Agent在向用户交付任何版本前，**必须**在P5检查通过后执行P5.1检查：

| 检查项 | 检查方法 | 不通过时的处理 |
|--------|----------|----------------|
| **配图引用完整性** | Grep搜索`!\[.*\]\(.*\)`，统计图片引用数量 | 引用数量为0且非纯文字文章 → 返回补充配图引用 |
| **配图文件存在性** | 对每个引用路径执行LS/Read验证文件存在 | 文件不存在 → 返回修复路径或重新生成配图 |
| **docx双格式输出** | LS确认同目录下存在同名.docx文件 | 不存在 → 生成docx后再交付 |
| **docx配图嵌入验证** | 检查docx文件大小（含配图应>500KB） | 大小异常 → 重新生成docx |
| **配图目录规范性** | LS确认配图存放在images/{文章编号}/目录 | 不符合 → 提醒整理（不阻断交付） |

**执行要求**：
- P5.1在P5之后执行，两项均通过方可交付
- 检查结果必须记录至STATE.md（遵循P11检查执行记录规则）
- P5.1为硬性阻断检查，任一项不通过不得交付（配图目录规范性除外）
- 主Agent必须亲自执行，不得委托子代理

## 强制阻断点（R4 评级 A 级后）

R4 评级达到 A 级后，在进入人工终审前，必须执行以下步骤（缺一不可）：
1. 执行发布前检查清单（18项逐项检查）
2. 执行归档完成流程（通用归档规则 §五）
3. 以上两项全部完成后，才能向用户交付

阻断逻辑：
- 发布前检查未通过 → 返回对应 Agent 修复 → 重新检查
- 归档未完成 → 执行归档 → 确认完成
- 两项均完成 → 进入人工终审

## 5.3 架构与规则变更版本控制（P27补丁）

**强制规则**：但凡KZCQL的架构文档和规则文件有变动，必须通过GitHub远程仓库进行版本控制。

**触发条件**（满足任一即触发）：
1. 修改了 `00_架构文档/` 下的任何文件
2. 修改了 `01_共享知识库/` 下的任何规则文件
3. 修改了 `02_子Agent规范/` 下的任何Agent规范
4. 新增或删除了Agent
5. 修改了 `05_人类反馈与规则演进/` 下的补丁记录

**执行流程**：
```
架构/规则变更完成
    ↓
git add <变更文件>
    ↓
git commit -m "<类型>: <简述>（P<补丁号>）"
    ↓
git push origin main
    ↓
确认推送成功
```

**commit message规范**：

| 类型 | 前缀 | 示例 |
|------|------|------|
| 新增规则 | `feat` | `feat: 新增V-20数据时效性否决项（P16）` |
| 修改规则 | `fix` | `fix: 修正D9评分标准引用（P13）` |
| 新增Agent | `feat` | `feat: 新增D2调研Agent规范（P28）` |
| 架构评估 | `docs` | `docs: A1架构评估98分A级（20260518）` |
| 事故审查 | `fix` | `fix: SG-2026-0518-01数据时效性事故修复` |

**验证标准**：
- [ ] `git push` 返回成功
- [ ] GitHub仓库中可看到最新commit
- [ ] commit message符合规范

**阻断规则**：架构/规则变更未推送到GitHub，不得进入下一阶段。
