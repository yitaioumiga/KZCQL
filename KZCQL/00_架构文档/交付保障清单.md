## 一、配图保障（5项）

- [ ] **IMG-01**：文章 Markdown 中存在图片引用（`![描述](路径)` 格式），数量 ≥ 4 张（封面 1 + 插图 3+）
  - 不通过时 → 按通用配图设计规则生成配图，插入文章
- [ ] **IMG-02**：所有引用的图片文件实际存在（使用 Read 工具或 `ls` 验证路径）
  - 不通过时 → 重新生成缺失图片
- [ ] **IMG-03**：图片无可见水印、Logo、平台标识（使用 Read 工具目视检查）
  - 不通过时 → 使用 Pillow 填充法去除水印（参考通用配图设计规则 §7）
- [ ] **IMG-04**：配图文件已归档至 `产出归档/{日期}_{关键词}/images/` 目录
  - 不通过时 → 执行 `cp` 命令归档
- [ ] **IMG-05**：图片分辨率与质量达标（数据图表类 ≥ 1920×1080，艺术类 ≥ 1280×720）；图片中无文字被图形元素遮挡、无乱码标签（使用 Read 工具目视检查或 `identify` 命令验证分辨率）
  - 不通过时 → 重新生成不合格图片，优先使用 wkhtmltoimage 渲染图表类配图，GenerateImage 生成艺术类配图后用 Pillow 去水印
- [ ] **IMG-06**：数据图表类配图内容准确性（仅对 HTML 渲染的图表检查）
  - 检查方法：使用 Read 工具目视抽查每篇文章的数据图表类配图，确认以下三项
  - **数据准确性**：图表中的数值、标签、分界线与文章声称的一致（如分数等级边界 0-59/60-74/75-89/90-100）
  - **视觉对齐**：坐标轴标签与对应数据点对齐，等级标签与对应色块/区域对齐
  - **颜色语义**：颜色与含义匹配（绿=优秀/通过，红=不合格/警告，蓝=良好，琥珀=合格），不得反转
  - 不通过时 → 修正 HTML 源文件中的数据/定位/颜色，重新用 wkhtmltoimage 渲染

## 二、归档保障（4项）

- [ ] **ARC-01**：产出归档目录已创建，结构为 `04_工作区/产出归档/{YYYYMMDD}_{文章关键词}/`
  - 不通过时 → 执行 `mkdir -p` 创建目录及子目录
- [ ] **ARC-02**：稿件已归档至 `产出归档/{...}/稿件/`（.md 和 .docx 两个格式）
  - 不通过时 → 执行 `cp` 归档
- [ ] **ARC-03**：评审报告已归档至 `产出归档/{...}/评审/`（R1/R2/R3/R4 报告）
  - 不通过时 → 检查 `04_工作区/产出归档/{日期}_{关键词}/评审/` 目录，执行 `cp` 归档
- [ ] **ARC-04**：`04_工作区/产出归档/` 目录中本次任务的归档结构完整（稿件/、评审/、images/ 子目录均存在且有对应文件）
  - 不通过时 → 执行 `mkdir -p` 补建缺失子目录，`cp` 补归档缺失文件

## 三、格式保障（2项）

- [ ] **FMT-01**：.docx 文件已生成且包含配图（文件大小应 > 500KB，纯文字文章除外）
  - 不通过时 → 使用 python-docx 生成 .docx，插入图片
- [ ] **FMT-02**：.md 文件为最终版本（含配图引用、加粗金句、行内标题）
  - 不通过时 → 确认使用的是最新版本文件

## 三b、评审保障（1项）

- [ ] **REV-01**：最终版本的 R4 评级报告已生成且保存至评审归档目录（`产出归档/{...}/评审/`）
  - 检查方法：确认存在文件名包含 `R4` 且版本号匹配最终版本的评审报告
  - 不通过时 → 回退执行 R4 评级判定，生成报告后再重新检查

## 四、清理保障（1项）

- [ ] **CLN-01**：`/data/user/work/` 目录中的本次任务临时文件已无保留价值
  - 不通过时 → 确认临时脚本和中间产物不影响最终交付后，可保留或清理

## 五、迭代保障（1项）

- [ ] **ITR-01**：迭代轮次未超过5轮上限
  - 检查方法：确认当前文章版本号中的迭代次数（如 v0.3 表示第3轮），未超过5轮
  - 不通过时 → 停止迭代，向用户报告当前最佳版本并说明迭代已超限，建议人工终审或重写
