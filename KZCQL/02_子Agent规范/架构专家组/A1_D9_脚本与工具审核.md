## 二、审核维度（满分12分）

### D9.1 脚本存在性与完整性（3分）

**检查项**：

| 检查点 | 说明 | 判定标准 |
|--------|------|----------|
| 脚本存在性 | 所有声明的脚本是否都存在 | 所有在规范中引用的脚本文件都存在 |
| 入口完整性 | 脚本是否有明确的入口点 | 每个脚本都有main函数或__main__入口 |
| 依赖完整性 | 依赖是否都明确声明 | 所有import都有对应的文件或pip包 |

**评分标准**：
- 3分：所有脚本存在，入口完整，依赖完整
- 2分：缺少1-2个非关键脚本
- 1分：缺少关键脚本或入口不完整
- 0分：核心脚本缺失

---

### D9.2 脚本功能正确性（3分）

**检查项**：

| 检查点 | 说明 | 判定标准 |
|--------|------|----------|
| 功能实现 | 脚本是否实现了声明的功能 | 功能与规范描述一致 |
| 错误处理 | 是否有完善的错误处理 | 有try-except，有错误日志 |
| 返回值 | 返回值是否符合预期 | 返回值类型、格式符合规范 |

**评分标准**：
- 3分：功能正确，错误处理完善，返回值规范
- 2分：功能基本正确，有少量问题
- 1分：功能有缺陷，错误处理不完善
- 0分：功能无法运行

---

### D9.3 与架构规范一致性（3分）

**检查项**：

| 检查点 | 说明 | 判定标准 |
|--------|------|----------|
| 命名一致性 | 脚本命名是否符合规范 | 命名与规范中的代号一致 |
| 配置一致性 | 配置项是否与规范一致 | AGENT_SEQUENCE、MANDATORY_AGENTS等 |
| 流程一致性 | 脚本流程是否与规范一致 | 脚本实现与编排指南描述一致 |
| **配置同步性** | **编排脚本与架构文档配置是否同步** | **见下方D9.3.1** |

**评分标准**：
- 3分：完全一致的命名、配置、流程
- 2分：有1-2处轻微不一致
- 1分：有多处不一致
- 0分：严重偏离规范

#### D9.3.2 编排执行合规性检查（P33补丁新增）

**⚠️ 重要**：这是架构评估必须检查的关键项，用于检测"执行是否符合规范"。

**检查目标**：确保编排流程的执行符合规范，而不是只检查规范文本的一致性。

**检查项**：

| 检查点 | 说明 | 判定标准 |
|--------|------|----------|
| **Step 4执行合规性** | 架构评估是否按并行+串行方式执行 | A1-主、D5-1/2/3/4、A1-整合分别调用 |
| **D5并行执行证据** | D5的四个子代理是否并行执行 | 必须有4个独立的子报告 |
| **执行与规范对比** | 实际执行与规范要求的对比 | 执行流程必须与编排指南一致 |

**⚠️ 常见违规**：

| 违规类型 | 说明 | 严重程度 |
|----------|------|----------|
| ❌ 串行代替并行 | 只调用一个子代理执行全部工作 | P0 |
| ❌ 跳过并行步骤 | 未执行D5-1/2/3/4并行调用 | P0 |
| ❌ 执行证据缺失 | 没有D5子报告 | P1 |
| ❌ 执行与规范不符 | 执行流程与编排指南不一致 | P1 |

**评分标准**：
- 3分：执行完全符合规范，有完整执行证据
- 2分：执行基本符合规范，有轻微偏差
- 1分：执行与规范有重大偏差
- 0分：执行严重违反规范（P0级违规）

**验证方法**：
```bash
# 检查是否有D5并行执行证据
ls /workspace/KZCQL/04_工作区/架构归档/中间结果/D5-*.md

# 检查报告是否包含D5-1/2/3/4子报告
grep -c "D5-1\|D5-2\|D5-3\|D5-4" /workspace/KZCQL/04_工作区/架构归档/架构审查_*.md

# 检查执行记录
cat /workspace/KZCQL/04_工作区/产出归档/*/orchestrator.log
```

**阻断规则**：
- 串行代替并行 → **P0级违规，架构评估无效**
- 跳过并行步骤 → **P0级违规，架构评估无效**
- 执行证据缺失 → **P1级问题，必须补充证据**

#### D9.3.2a 写作流程执行合规性检查（P33补丁扩展）

**⚠️ 重要**：确保 W1→R1→R2→R4 写作流程的执行符合规范。

**检查项**：

| 检查点 | 说明 | 判定标准 |
|--------|------|----------|
| **R1后R2执行** | R1=PASS时R2是否强制执行 | R2必须有独立评审报告 |
| **R4强制执行** | R4是否在R1结果后执行 | R4必须有独立评级报告 |
| **评审路径正确性** | R1=FAIL时是否正确跳过R2 | R2跳过有R1=FAIL作为依据 |
| **执行与规范对比** | 实际执行与写作流程规范的对比 | 执行流程必须与编排指南一致 |

**⚠️ 常见违规**：

| 违规类型 | 说明 | 严重程度 |
|----------|------|----------|
| ❌ R1=PASS后跳过R2 | 违反强制串行要求 | P0 |
| ❌ R1=FAIL后跳过R4 | R4是强制Agent不能跳过 | P0 |
| ❌ R4评级与R1/R2报告矛盾 | R4必须基于实际评审 | P1 |

**验证方法**：
```bash
# 检查R2报告是否存在（R1=PASS时）
ls /workspace/KZCQL/04_工作区/产出归档/*/评审/*_R2_*.md

# 检查R4报告是否存在
ls /workspace/KZCQL/04_工作区/产出归档/*/评审/*_R4_*.md

# 检查R1结果与R2执行的一致性
grep -l "R1.*PASS\|R1.*FAIL" /workspace/KZCQL/04_工作区/产出归档/*/评审/*_R1_*.md
```

**阻断规则**：
- R1=PASS后跳过R2 → **P0级违规，评审链无效**
- R1=FAIL后跳过R4 → **P0级违规，评级无效**

## 五、与回归测试的关系

### 5.1 审核触发条件

| 触发条件 | 说明 |
|----------|------|
| 新增脚本 | 每次新增脚本必须进行D9审核 |
| 修改脚本 | 每次修改脚本核心逻辑必须进行回归测试 |
| 架构升级 | 每次架构升级后进行完整D9审核 |

### 5.2 审核与测试的关系

```
D9审核（静态+动态）
    ↓
├── 静态检查（脚本存在性、语法、规范一致性）
│   └── 发现问题 → 修复 → 重新审核
│
└── 动态测试（回归测试套件）
    └── 测试通过 → 审核完成
    └── 测试失败 → 修复 → 重新测试
```

### 5.3 强制要求

**以下情况禁止通过审核**：
1. 回归测试通过率 < 100%
2. 核心脚本无法运行
3. 严重偏离架构规范
4. 破坏向后兼容性且无迁移方案
