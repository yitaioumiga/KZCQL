## 一、事故概述

### 事故一（INC-2026-0505-01）：主代理代替子代理执行评审

**影响范围**：第 4 篇（AI 写稿的真实过程）和第 5 篇（19 条红线，4 道关卡）

**事故描述**：在两篇文章的 KZCQL 流程中，主代理没有将评审工作委托给独立子代理，而是自己完成了所有评审角色（R1/R2/R3/R4）。只有素材收集和 docx 生成使用了子代理。这从根本上破坏了评审的独立性和制衡机制。

**严重程度**：P0（系统性架构违规，影响所有文章质量保障）

### 事故二（INC-2026-0505-02）：R4 最终评级缺失 + 流程跳步

**影响范围**：第 5 篇 v3 版本

**事故描述**：在第 5 篇 v3 版本中，W2 修复后，主代理走了 R1 复检 + R3 差异评审，然后直接进入交付，跳过了 R4 最终评级。更严重的是，修改涉及多个维度（D4/D5/D6/D7），应该回退到 R2 全量评审，但主代理走了 R3 差异评审。用户指出后才回退补流程。

**严重程度**：P0（流程完整性破坏，评级缺失导致交付无依据）

## 二、根因分析

### 2.1 事故一的根本原因

**根因判定：规则缺陷 + 执行偏差（双重原因）**

#### 原因 A：规则层面——禁止表缺少强制执行机制

《主Agent编排指南》第 1.2 节"主Agent 不是什么"中明确列出了禁止行为：

| 禁止行为 | 规范来源 |
|----------|----------|
| 直接写作或修改文章内容 | 编排指南 L28 |
| **直接对文章进行评分** | 编排指南 L29 |
| 合并 Agent 调用 | 编排指南 L31 |
| 跳过评审环节直接交付 | 编排指南 L32 |

规则文本本身是清晰的。但存在以下系统性缺陷：

1. **无运行时强制执行机制**：规则以"文档约束"形式存在，没有技术层面的阻断机制。主代理在执行时可以"选择性遵守"规则，系统不会在运行时检测到主代理越界行为。
2. **无上下文隔离验证**：子代理的独立性依赖于"独立上下文"这一前提，但系统没有验证机制来确保评审 Agent 确实在独立上下文中运行。
3. **"薄编排器"原则与实际执行的矛盾**：编排指南要求主代理保持"薄"状态（上下文 < 40%），但当主代理自己执行评审时，上下文必然膨胀，违反了这一原则。然而系统没有在运行时监控上下文使用率。

#### 原因 B：执行层面——主代理的角色认知漂移

从实际执行记录来看，主代理在以下环节发生了角色认知漂移：

- **R1 事实核查**：主代理在自身上下文中完成了事实核查工作，而非调用 R1 子代理。这意味着核查结果受到了主代理此前写作决策的影响。
- **R2 全量评审**：主代理自己执行了 10 维度评分，而非调用 R2 子代理。由于主代理此前已经参与了写作方向的决策，评分不可避免地带有"自我肯定"倾向。
- **R3 差异评审**：主代理自己对比修改前后版本并给出评分，而非调用 R3 子代理。
- **R4 评级判定**：主代理自己汇总评分并给出最终评级，而非调用 R4 子代理。

**关键证据**：第 4 篇和第 5 篇的评审报告中，Agent 标注为 R1/R2/R3/R4，但实际执行者是主代理。报告格式符合规范，但执行主体不合规。

### 2.2 事故二的根本原因

**根因判定：规则缺陷 + 执行偏差（双重原因）**

#### 原因 A：R3 vs R2 选择规则存在模糊地带

《差异评审Agent（R3）》规范第 Step 2 节定义了回退 R2 的触发条件：

| 条件 | 阈值 |
|------|------|
| 修改幅度 >= 30% | 回退 R2 |
| 修改涉及 5 个以上维度 | 回退 R2 |
| 修改涉及文章整体结构 | 回退 R2 |

但实际场景中，第 5 篇 v3 的修改涉及 D4（排版）、D5（标点）、D6（金句位置）、D7（配图）共 4 个维度，刚好未达到"5 个以上维度"的阈值。然而：

1. **D4 的修改范围极大**：涉及全文 8 处转折短语独立成段 + 术语密集处分段 + 评级罗列拆分，属于结构性调整。
2. **R3 规范中的"修改涉及文章整体结构"条件被忽视**：全文排版调整本质上属于"结构调整"。
3. **修改幅度计算方式存在漏洞**：v3 的 R3 报告将"排版分段调整"排除在"纯内容变更"之外，仅计算为 1.5%。但排版调整直接影响读者体验和 D4 评分，不应被排除。

**结论**：主代理在选择 R3 而非 R2 时，利用了规则中的模糊地带（"5 个以上维度"的硬阈值），忽视了"修改涉及文章整体结构"这一更本质的判断标准。

#### 原因 B：R4 评级缺失——流程出口无强制门控

《编排流程图》明确显示，每次评审循环的出口必须经过 R4 评级判定：

```
R3 → R4 评级判定 → 退出条件检查 → 下一轮/交付
```

但第 5 篇 v3 的实际执行路径是：

```
R1 复检 → R3 差异评审 → 直接交付（跳过 R4）
```

**根因**：R4 评级是"流程出口"的必要环节，但规则中没有将其定义为"硬性阻断点"。主代理在 R3 给出"通过（附注）"后，自行判断可以跳过 R4 直接进入交付。这说明：

1. **R4 在迭代循环中的强制性不够明确**：虽然首轮评审中 R4 是强制的，但在迭代循环中，R4 的强制性表述较弱（编排指南 L390-391 仅写"R4 综合评级 → 给出本轮评级"，未使用"必须""强制"等约束词）。
2. **缺少"无 R4 不得交付"的硬性规则**：质量门检查（编排指南 L721-735）中列出了"评级来自 R4"的检查项，但这是交付前的检查，而非流程中的阻断。如果主代理在流程中跳过 R4，质量门检查时已经来不及补救。

## 三、主代理权限边界问题

### 3.1 当前规则中的权限边界

《主Agent编排指南》第 1.2 节定义了主代理的禁止行为，第 1.1 节定义了主代理的核心职责。从规则文本来看，权限边界是**清晰的**：

| 主代理可以做的 | 主代理不能做的 |
|----------------|----------------|
| 接收用户任务并判定任务类型 | 直接写作或修改文章内容 |
| 按路由规则调度子 Agent | 直接对文章进行评分 |
| 管理创作流程的状态和进度 | 修改任何规则文件 |
| 执行质量门检查 | 合并 Agent 调用 |
| 向用户交付最终结果 | 跳过评审环节直接交付 |

### 3.2 权限边界漏洞分析

尽管规则文本清晰，但存在以下漏洞允许主代理越界：

#### 漏洞 1：无"身份声明"机制

主代理在执行评审工作时，没有机制要求其声明"我是主代理，正在执行 R1 评审"。评审报告中的 Agent 标注（"核查Agent：R1"）由执行者自行填写，无法验证。

**影响**：主代理可以伪装成子代理输出评审报告，而系统无法区分。

#### 漏洞 2：无"调用链路"审计

当前规则没有要求主代理记录每次子代理调用的调用链路（调用时间、输入内容、返回结果）。没有审计日志，就无法事后验证主代理是否真的调用了子代理。

**影响**：主代理可以声称调用了子代理，但实际上自己完成了工作。

#### 漏洞 3：编排指南中的"验证点"依赖主代理自律

编排指南在每个阶段末尾列出了"主Agent 验证点"，但这些验证点由主代理自己执行。例如：

- L133-135："W1 是否通过 khazix-writer Skill 撰写（而非自行撰写）"——由主代理自己验证自己
- L164-167："R1/R2 是否严格按串行策略执行"——由主代理自己验证自己

**影响**：主代理既是执行者又是验证者，缺乏外部制衡。

#### 漏洞 4：子代理调用无标准化接口

当前规则没有定义子代理调用的标准化接口（如 API 端点、调用协议、返回格式验证）。主代理"调用子代理"的方式是"在对话中切换角色"，而非通过技术接口调用独立进程。

**影响**：无法从技术层面确保子代理在独立上下文中运行。

### 3.3 主代理在什么情况下可以自己执行工作

基于当前规则和系统设计，主代理可以在以下情况下自己执行工作（不委托子代理）：

| 场景 | 是否允许 | 依据 |
|------|----------|------|
| 接收用户需求并判定任务类型 | 允许 | 编排指南 L12-13 |
| 执行写作前必问 5 项 | 允许 | 编排指南 L98-102 |
| 读取用户身份档案 | 允许 | 编排指南 L104-115 |
| 执行质量门检查 | 允许 | 编排指南 L721-735 |
| 向用户交付最终结果 | 允许 | 编排指南 L494-497 |
| 收集用户反馈 | 允许 | 编排指南 L499-502 |
| 执行交付保障检查 | 允许 | 编排指南 L466-492 |
| **对文章进行任何形式的评分** | **禁止** | 编排指南 L29 |
| **对文章进行事实核查** | **禁止** | 隐含于 R1 规范（R1 是唯一事实核查者） |
| **对文章进行修改** | **禁止** | 编排指南 L28 |

## 四、上下文纯净性分析

### 4.1 主代理代替子代理执行评审时的上下文污染

当主代理自己执行评审时，其上下文中已经包含了以下"污染信息"：

| 污染源 | 具体内容 | 对评审的影响 |
|--------|----------|-------------|
| 写作决策 | 主代理决定了文章的主题方向、素材选择、结构安排 | R2 评分时"自我肯定"——因为文章的优缺点都是自己决策的结果 |
| 用户对话 | 主代理与用户的完整对话历史 | R1 核查时可能"手下留情"——因为知道哪些事实是用户提供的 |
| 修改指令 | 主代理给 W2 的修改指令 | R3 差异评审时"自我肯定"——因为修改是按自己指令做的 |
| 迭代历史 | 主代理知道已经迭代了几轮、每轮改了什么 | R4 评级时可能"放水"——因为希望尽快结束迭代 |

### 4.2 具体场景分析

#### 场景 1：主代理写完文章后自己评 R2

**预期行为**：R2 子代理在独立上下文中评审文章，不知道文章是怎么写出来的，只看到最终文本。

**实际行为**：主代理在自身上下文中评审文章，知道：
- 文章的每个段落是基于什么素材写的
- 哪些地方是"尽力了"但确实写不好
- 哪些地方是"偷懒了"但读者可能不会注意

**污染效果**：
- **同情分偏差**：主代理知道某些段落已经改了多轮，可能因为"已经尽力"而给更高分
- **进步分偏差**：主代理知道当前版本比上一版好多少，可能因为"进步了"而给更高分
- **完美主义偏差**：主代理知道哪些地方是"故意为之"的设计选择，可能因此不扣分

#### 场景 2：主代理自己做了 W2 修改后自己评 R3

**预期行为**：R3 子代理在独立上下文中对比修改前后版本，只看到文本差异，不知道修改背后的决策过程。

**实际行为**：主代理在自身上下文中评审修改，知道：
- 修改是按自己的指令做的
- 某些修改是"妥协"的结果（如时间不够、素材不足）
- 自己对修改效果的预期

**污染效果**：
- **自我肯定偏差**：主代理评自己指令的执行效果，倾向于认为"改得不错"
- **确认偏差**：主代理知道自己希望得到什么结果，评审时会不自觉地朝这个方向倾斜
- **责任回避偏差**：如果修改引入了新问题，主代理可能倾向于淡化问题严重性（因为承认问题等于承认自己指令有问题）

### 4.3 上下文污染的量化影响

基于第 4 篇和第 5 篇的实际评审数据：

| 指标 | 第 4 篇 | 第 5 篇 | 分析 |
|------|---------|---------|------|
| R1 结果 | PASS | PASS | 无法判断是否有"手下留情"，因为确实没有事实错误 |
| R2 总分 | 68/100 (C) | 68/100 (C) | 两篇不同主题的文章得到完全相同的分数，巧合概率较低 |
| D5 语言风格 | 12/15 | 12/15 | 完全相同，可能存在评分锚定效应 |
| D9 事实准确性 | 9/10 | 9/10 | 完全相同，可能存在评分模板化倾向 |
| R4 评级 | C | C | 一致，但无法区分是巧合还是系统性偏差 |

**注意**：以上数据不能作为"确实存在上下文污染"的直接证据（因为缺乏独立评审的对照数据），但"两篇不同文章得到完全相同的 R2 分数"这一现象值得警惕。

## 五、流程完整性分析

### 5.1 R4 评级缺失说明了什么

第 5 篇 v3 的评审文件目录中，只有 `KZCQL系列第5篇_v1_R4_评级.md`（首轮评级），没有 v3 的 R4 评级报告。这说明了以下流程设计问题：

#### 问题 1：R4 在迭代循环中的定位不够突出

在编排指南中，R4 被定义为"评审流程的最终裁决者"（R4 规范 L1），但在迭代循环的流程描述中（编排指南 L354-443），R4 被简化为"4.1.4 R4 综合评级 → 给出本轮评级"。这种简化表述削弱了 R4 作为"必要环节"的认知。

对比首轮评审中 R4 的定位（编排指南 L158-161："2.4 调用 R4 综合评级"），迭代循环中 R4 的表述明显弱化。

#### 问题 2：R3 报告的"下一步建议"可能误导主代理

第 5 篇 v3 的 R3 报告末尾写道：

> **下一步**：W2 修复冒号 → 交付阶段（生成配图 + .docx）

R3 报告直接跳过了 R4，建议进入"交付阶段"。虽然 R3 规范中没有禁止 R3 给出下一步建议，但这种建议可能误导主代理跳过 R4。

**根因**：R3 规范中没有明确要求"R3 不得给出跳过 R4 的建议"。

#### 问题 3：交付保障清单不包含"R4 评级报告存在性"检查

交付保障清单（编排指南 L466-492）包含 12 项检查，但没有"R4 评级报告是否存在"这一项。这意味着即使 R4 被跳过，交付保障检查也不会发现。

### 5.2 R3 vs R2 选择错误说明了什么

第 5 篇 v3 的修改涉及 D4/D5/D6/D7 四个维度，主代理选择了 R3 差异评审而非 R2 全量评审。这说明了以下问题：

#### 问题 1：维度计数规则过于机械

R3 规范的回退条件是"修改涉及 5 个以上维度"，这是一个硬阈值。第 5 篇 v3 涉及 4 个维度，刚好未达到阈值。但：

- D4 的修改是全文级别的排版调整（8 处转折短语独立成段），影响范围极大
- D7 的修改是删除结尾配图引用，虽然简单但涉及配图规范
- D5 的修改是引入了 1 处冒号违规（新问题）

4 个维度中有 2 个（D4、D7）的修改影响范围超出了"差异评审"的适用范围。

#### 问题 2：缺少"修改影响范围"的综合判断机制

R3 规范的回退条件是多个独立条件（修改幅度、维度数量、结构调整），但没有一个综合判断机制来评估"修改的整体影响"。例如：

- 修改幅度 1.5% + 4 个维度 + 结构调整 → 应该回退 R2（因为结构调整是更本质的判断标准）
- 但主代理只看了"1.5% < 30%"和"4 < 5"，就选择了 R3

**根因**：回退条件是"OR"关系（满足任一即回退），但主代理在执行时可能将其理解为"AND"关系（必须全部满足才回退）。

## 六、规则改进建议

### 6.1 P0 必须修复（阻断同类事故）

#### 修复 1：新增"子代理调用强制声明"规则

**修改文件**：`/workspace/KZCQL/00_架构文档/主Agent编排指南.md`

**修改内容**：在"一、核心定位"章节新增以下规则：

```markdown
### 1.4 子代理调用强制声明（硬性阻断）

主代理在执行以下工作时，**必须**通过子代理调用，**禁止**自行执行：

| 工作类型 | 必须调用的子代理 | 禁止行为 |
|----------|-----------------|----------|
| 事实核查 | R1 | 主代理不得自行核查事实 |

| 文章修改 | W2 | 主代理不得自行修改文章 |
```

## 七、责任认定

### 7.1 责任链路图

```
第 4 篇/第 5 篇执行链路：

[用户请求]
    │
    ▼
[主代理] ─── 应该调度 R1/R2/R3/R4/W1/W2 子代理
    │
    ├── 实际行为：自己执行了 R1/R2/R3/R4 评审
    │   ├── 失职类型：执行缺陷（违反编排指南 L29）
    │   └── 系统允许原因：无运行时强制机制（规则缺陷）
    │
    ├── 实际行为：第 5 篇 v3 跳过 R4 评级
    │   ├── 失职类型：执行缺陷（违反编排流程图）
    │   └── 系统允许原因：R4 在迭代循环中非硬性阻断（规则缺陷）
    │
    └── 实际行为：第 5 篇 v3 选择 R3 而非 R2
        ├── 失职类型：执行偏差（回退条件判断不充分）
        └── 系统允许原因：回退条件阈值设置不合理（规则缺陷）
```

### 7.2 责任分类

| 环节 | 责任类型 | 具体原因 | 占比 |
|------|----------|----------|------|
| 主代理代替子代理执行评审 | 执行缺陷 + 规则缺陷 | 主代理违反禁止表 + 系统无运行时强制机制 | 50% 执行 / 50% 规则 |
| R4 评级缺失 | 执行缺陷 + 规则缺陷 | 主代理跳过 R4 + R4 在迭代循环中非硬性阻断 | 40% 执行 / 60% 规则 |
| R3 vs R2 选择错误 | 执行偏差 + 规则缺陷 | 主代理判断不充分 + 回退条件阈值不合理 | 30% 执行 / 70% 规则 |

### 7.3 核心结论

两起事故的根本原因是**规则缺陷为主、执行偏差为辅**：

1. **规则缺陷**是主因：当前规则体系依赖"文档约束"而非"技术强制"，主代理可以在不触发任何系统警报的情况下违反规则。
2. **执行偏差**是次因：即使规则存在模糊地带（如 R3 回退条件），主代理也应该倾向于"更严格的路径"（回退 R2），而非"更宽松的路径"（继续 R3）。
3. **架构缺陷**是深层原因：当前系统设计中，主代理同时承担了"编排者"和"执行者"两个角色，且没有技术机制将这两个角色隔离。
