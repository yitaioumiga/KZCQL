## 二、架构改进建议

### 2.1 流程覆盖一致性：完整流程与微调流程的统一

**问题本质**：KZCQL 系统存在两条并行的任务路由（完整流程 vs 微调流程），它们的步骤覆盖范围不一致。当微调任务被路由到完整评审链路时，配图和归档步骤无承接。

**深层原因**：这不是简单的"规则遗漏"，而是架构设计中"流程组合"的系统性问题。当前设计假设"微调任务要么走微调流程，要么走完整流程"，但实际运行中出现了"微调任务走完整评审链路"的混合路径，这条路径在架构设计中没有被预见到。

**改进建议**：

**建议 ARCH-01：引入"交付保障层"概念，与任务路由解耦**

当前架构将配图、归档、发布前检查等步骤嵌入在具体的任务流程中（完整流程的阶段五、微调流程的 Step 6/Step 9），导致不同流程的覆盖范围不一致。建议将这些步骤抽离为独立的"交付保障层"，在所有任务路由的终点统一执行。

```
当前架构：
  完整流程 → 阶段一~四 → 阶段五（含配图/归档/发布前检查）
  微调流程 → Step 1~9（含配图/归档）
  混合路径 → ???（覆盖缺口）

改进后架构：
  任意任务路由 → 评审链路（可选）→ 交付保障层（强制）
                                    ├── 配图生成与验证
                                    ├── 归档完成流程
                                    ├── 发布前检查清单
                                    └── .docx 格式输出
```

**实施要点**：
1. 在主Agent编排指南中新增"交付保障层"章节，定义为一个独立的、不可跳过的流程阶段
2. 交付保障层的触发条件：所有任务类型在进入"向用户交付"前必须执行
3. 交付保障层的内容不依赖于前置流程的类型——无论走完整流程、微调流程还是混合路径，交付保障层的内容完全相同
4. 从完整流程的阶段五和微调流程的 Step 6/Step 9 中移除配图和归档相关步骤，统一由交付保障层负责

**预期效果**：消除流程覆盖不一致问题，无论任务走哪条路径，交付保障层都能确保配图、归档、发布前检查不被遗漏。

---

**建议 ARCH-02：定义"流程升级"的明确规则**

当前编排指南 3.2 节允许微调任务触发完整评审链路，但未定义升级条件和升级后的步骤衔接规则。

**实施要点**：
1. 在编排指南 3.2 节中新增"微调任务升级规则"：
   - 升级条件：R1 事实核查发现重大事实错误（>3 项），或用户要求深度优化
   - 升级后的路径：微调流程 Step 1-5 → 完整评审链路（R1→R2→R4→W2→R3→R4）→ 交付保障层
   - 明确标注：升级后，微调流程的 Step 6-9 不再独立执行，由交付保障层统一接管
2. 如果不满足升级条件，微调任务必须严格按 9 步流程执行，不得自行切换到完整评审链路

**预期效果**：消除主Agent自行决定路径切换的灰色地带，确保所有路径都有明确的步骤覆盖。

### 2.2 阻断机制设计：从"建议性检查"到"物理阻断"

**问题本质**：本次事故中，质量门检查、发布前检查清单、归档完成流程都存在于规则文件中，但主Agent 在实际执行时全部跳过。这说明当前的"阻断机制"本质上是"建议性检查"——规则要求做，但系统设计上没有强制执行的保障。

**深层原因**：在 LLM Agent 系统中，"物理阻断"与"建议性检查"的区别在于：前者通过流程结构（而非规则文本）来确保步骤不被跳过。当前 KZCQL 的阻断机制依赖于主Agent 自觉遵守规则文本，这是一种"软阻断"。

**改进建议**：

**建议 ARCH-03：引入"交付令牌"机制**

借鉴分布式系统中的"令牌传递"模式，设计一个"交付令牌"机制：

```
交付令牌获取流程：
  1. 配图验证通过 → 获得令牌片段 A
  2. 归档完成流程执行 → 获得令牌片段 B
  3. 发布前检查清单全部通过 → 获得令牌片段 C
  4. A + B + C → 组装完整交付令牌
  5. 完整交付令牌 → 允许进入"向用户交付"步骤
```

**实施要点**：
1. 在主Agent编排指南的质量门检查章节中，将"交付令牌"作为显式的流程控制结构
2. 主Agent 在每次会话的状态管理中维护令牌获取状态
3. 令牌获取状态必须在主Agent 的"创作会话状态"模板中显式列出
4. 如果主Agent 尝试在令牌不完整时进入交付步骤，视为执行缺陷

**注意事项**：在 LLM Agent 系统中，"物理阻断"不可能像传统软件那样通过代码强制执行。交付令牌的本质是将"隐性规则遵守"转化为"显式状态追踪"，降低主Agent 因上下文压力而遗漏步骤的概率。

---

**建议 ARCH-04：在编排流程图中增加阻断节点**

当前编排流程图（03_主Agent编排层/编排流程图.md）中，发布前检查和归档步骤可能以普通节点呈现，未突出其阻断性质。

**实施要点**：
1. 在流程图中使用特殊的视觉标记（如红色边框、双线框）标注阻断节点
2. 阻断节点包括：发布前检查、归档确认、配图验证
3. 在流程图的阻断节点旁标注"不通过时的回退路径"
4. 确保流程图中不存在可以绕过阻断节点的路径

### 2.3 配图责任分配：在所有任务类型中确保配图不被遗漏

**问题本质**：配图生成在完整流程中由 W1 负责（阶段一），在微调流程中由主Agent统一处理（Step 6），在混合路径中无人负责。配图责任的分配依赖于任务路由的选择，而任务路由的选择并不总是可预测的。

**改进建议**：

**建议 ARCH-05：配图责任与任务路由解耦，统一由交付保障层调度**

结合 ARCH-01 的"交付保障层"设计，将配图生成从所有任务流程中移除，统一由交付保障层在评审链路完成后调度。

```
配图生成时机：
  完整流程：W1 初稿 → 评审链路 → 交付保障层（配图生成）→ 交付
  微调流程：Step 1-5 → 交付保障层（配图生成）→ 交付
  混合路径：Step 1-5 → 评审链路 → 交付保障层（配图生成）→ 交付
```

**实施要点**：
1. 交付保障层中的配图生成步骤定义：
   - 检查文章中是否已有图片引用（`![描述](路径)` 格式）
   - 如果无配图，根据通用配图设计规则生成配图（封面图 1 张 + 章节插图 3-4 张）
   - 如果有配图，使用 Read 工具逐张验证（无水印、无文字、风格统一）
   - 配图生成后，更新文章 Markdown 中的图片引用
2. 配图生成由主Agent 调度 GenerateImage 工具执行，不委托给任何子Agent
3. 从 W1 的职责中移除配图生成（W1 专注于文章内容撰写），从微调流程的 Step 6 中移除配图生成

**预期效果**：
- 配图生成不再依赖于任务路由的选择
- 评审链路评审的是"无配图版本"，但交付保障层确保最终交付版本有配图
- 消除"配图责任悬空"的系统性风险

**风险与缓解**：
- 风险：评审链路无法评审配图质量（因为评审时配图尚未生成）
- 缓解：交付保障层中的配图验证步骤（Read 工具逐张检查）作为配图质量的独立保障；R2 D7 的"无配图判零"机制确保评审链路会标记配图缺失，触发交付保障层的配图生成

---

**建议 ARCH-06：建立"配图状态"作为文章元数据**

在文章的版本管理中引入"配图状态"字段：

```
文章元数据模板：
  - 版本号：v0.1
  - 配图状态：未生成 / 已生成 / 已验证 / 已归档
  - 归档状态：未归档 / 已归档
  - 发布前检查状态：未检查 / 已通过
```

**实施要点**：
1. 在主Agent 的"创作会话状态"模板中增加配图状态、归档状态、发布前检查状态三个字段
2. 每次状态变更时更新对应字段
3. 交付保障层在执行前先检查这三个状态，任何状态不为"已完成"则触发对应步骤

### 2.4 归档自动化：减少对主Agent记忆的依赖

**问题本质**：归档完成流程（通用归档规则 §五）定义了 5 个步骤，但这些步骤完全依赖主Agent 自行记忆和执行。在长会话中，主Agent 的上下文压力增大，归档步骤容易被遗忘。

**改进建议**：

**建议 ARCH-07：归档步骤"清单化"并嵌入状态管理**

当前归档完成流程以段落形式描述在通用归档规则中，主Agent 需要自行记忆并执行。建议将归档步骤转化为可勾选的清单，并嵌入主Agent 的状态管理模板。

**实施要点**：
1. 在主Agent编排指南的"创作会话状态"模板中增加归档清单：
   ```
   ## 归档状态
   - [ ] 创建产出归档目录（04_工作区/产出归档/{YYYYMMDD}_{文章关键词}/）
   - [ ] 稿件归档至 产出归档/{...}/稿件/
   - [ ] 配图归档至 产出归档/{...}/配图/
   - [ ] 评审报告归档至 产出归档/{...}/评审/
   - [ ] 清理待评文章旧文件
   ```
2. 每次阶段转换时，主Agent 必须检查并更新归档清单状态
3. 归档清单的完成状态作为"交付令牌"（ARCH-03）的令牌片段 B

---

**建议 ARCH-08：归档触发时机前置**

当前归档的触发时机是"文章完成全部评审流程后"，这意味着归档是整个流程的最后一步。在长流程中，"最后一步"容易被遗忘。

**实施要点**：
1. 将归档从"一次性完成"改为"渐进式归档"：
   - R1 完成后：归档 R1 报告至评审归档/R1_事实核查/
   - R2 完成后：归档 R2 报告至评审归档/R2_全量评审/
   - R3 完成后：归档 R3 报告至评审归档/R3_差异评审/
   - R4 完成后：归档 R4 报告至评审归档/R4_终审评级/
   - 交付前：执行产出归档（稿件/配图/评审汇总）
2. 各子Agent 的归档步骤已在规范中定义（且本次事故中子Agent 均正确归档了评审报告），问题出在主Agent 的产出归档
3. 建议在 R4 最终评级完成后、进入交付前，主Agent 立即执行产出归档（而非等到交付完成后）

- D2 流程完整性：12/12（满分）

**核心矛盾**：A1 在事故发生前 3 天给出了 96 分的 A+ 评级，D2 流程完整性满分。但 3 天后就发生了 P0 级生产事故（归档遗漏 + 配图遗漏）。这说明 A1 的审查框架存在盲区——它能检测到"规则文件之间的不一致"，但无法检测到"规则存在但执行无保障"的系统性风险。

### 3.2 八维度框架是否覆盖了"交付完整性"问题

**分析**：

A1 的八维度框架中，与"交付完整性"最相关的维度是：

| 维度 | 相关性 | 本次事故中的表现 |
|------|--------|------------------|
| D2 流程完整性 | 高 | 给了满分（12/12），但未发现"微调任务走完整评审链路时配图/归档步骤无承接"的问题 |
| D3 规则覆盖度 | 高 | 给了满分（12/12），但未发现"R2 D7 对无配图文章的评分逻辑漏洞" |
| D5 验证链路完整性 | 最高 | 给了 14/15 分，但未发现"质量门检查和发布前检查可以被主Agent跳过"的问题 |

**结论**：八维度框架在概念上覆盖了"交付完整性"，但在审查方法上存在不足。A1 的审查方法是"检查规则文件之间的引用关系和一致性"，而非"模拟实际执行路径来验证步骤覆盖"。这种"静态审查"方法无法发现"规则存在但执行路径上有缺口"的动态问题。

### 3.3 A1 是否应该检查"流程覆盖一致性"

**建议 A1-01：新增"流程路径覆盖分析"审查方法**

A1 应在 D2（流程完整性）和 D5（验证链路完整性）的审查中，增加"流程路径覆盖分析"方法：

**具体做法**：
1. 枚举系统中的所有任务路由（完整流程、微调流程、仅评审、混合路径等）
2. 对每条路径，追踪从"用户请求"到"用户交付"的完整步骤链
3. 检查每条路径是否都覆盖了"交付保障项"（配图、归档、发布前检查、.docx 输出）
4. 如果某条路径缺少交付保障项，标记为 P0 预警

**实施要点**：
1. 在 A1 规范的 D2 维度检查项中新增"流程路径覆盖分析"子项
2. 在 D5 维度检查项中新增"交付保障项路径追踪"子项
3. A1 在审查时，必须实际枚举所有任务路由并逐条追踪，不能仅凭"规则文件中提到了配图和归档"就判定覆盖完整

**预期效果**：A1 能够在静态审查中发现"规则存在但执行路径上有缺口"的问题，避免本次事故中"96 分 A+ 评级后 3 天发生 P0 事故"的情况。

---

### 3.4 A1 的评分维度是否需要新增或调整

**建议 A1-02：在 D5 中增加"阻断有效性"子维度**

当前 D5（验证链路完整性）的检查项包括"规则→执行映射""执行→验证映射""评审→实际检查映射""悬空规则检测""盲评风险检测"。但缺少对"阻断机制有效性"的检查。

**新增子项**：

| 检查项 | 说明 | 判定标准 |
|--------|------|----------|
| 阻断有效性 | 关键步骤（发布前检查、归档、配图验证）是否被设计为不可跳过的阻断节点 | 阻断节点在流程图中有明确标记，且回退路径清晰 |
| 状态追踪完整性 | 主Agent 是否有显式的状态追踪机制来确保步骤不被遗漏 | 状态管理模板中包含交付保障项的状态字段 |

---

**建议 A1-03：在 D2 中增加"路径组合覆盖"子项**

当前 D2 的检查项包括"核心流程""异常处理""流程图一致性""初始化流程"。但缺少对"路径组合"的检查——即当不同流程的片段被组合使用时（如微调流程 + 完整评审链路），步骤覆盖是否完整。

**新增子项**：

| 检查项 | 说明 | 判定标准 |
|--------|------|----------|
| 路径组合覆盖 | 当不同流程的片段被组合使用时，交付保障项是否仍被覆盖 | 所有可预见的路径组合都覆盖了交付保障项 |

---

### 3.5 A1 的审查频率是否足够

**当前状态**：A1 是 P2 优先级，按需触发（系统初始化、定期审查、重大变更、效能异常、人类请求、E1 建议）。本次事故前最近一次审查是 2026-04-24（第五轮），事故发生在 2026-04-27。

**分析**：

A1 的触发条件中，"重大变更"和"效能异常"理论上可以在事故前触发审查。但实际情况是：
1. 本次事故的根因（R2 D7 评分逻辑漏洞、主Agent 跳过质量门）不属于"重大变更"范畴
2. 事故发生前没有"效能异常"的信号（系统正常运行，只是交付物不完整）
3. A1 的审查方法（静态审查）即使被触发，也可能无法发现本次事故的根因

**建议 A1-04：增加"事故后强制审查"触发条件**

在 A1 的触发条件中新增：

| 触发条件 | 说明 |
|----------|------|
| 事故后强制审查 | 发生 P0 级生产事故后，A1 必须在修复完成后进行专项审查，验证修复方案是否引入新的架构问题 |

**实施要点**：
1. 在 A1 规范的"触发条件"章节中新增"事故后强制审查"
2. 审查范围：修复方案涉及的架构文件 + 修复方案可能影响的关联文件
3. 审查重点：修复方案是否引入新的路径缺口、职责冲突或规则矛盾

---

**建议 A1-05：建立"动态审查"能力**

当前 A1 的审查方法是纯静态的（读取规则文件，检查引用关系）。建议增加"动态审查"能力：

**具体做法**：
1. A1 在审查时，除了读取规则文件，还应读取最近 1-2 篇文章的完整执行日志（评审报告、修改记录、归档状态）
2. 对比"规则要求"与"实际执行"，识别"规则存在但执行无保障"的缺口
3. 特别关注：质量门检查是否被逐项执行、归档步骤是否被完成、配图是否被生成

**实施要点**：
1. 在 A1 规范的输入规范中，将"最近 1-2 篇文章的执行日志"从"可选输入"提升为"推荐输入"
2. 在 D5 维度中新增"规则执行一致性"检查项：对比规则要求与实际执行记录
3. 在 A1 的 Prompt 模板中增加动态审查的指令
