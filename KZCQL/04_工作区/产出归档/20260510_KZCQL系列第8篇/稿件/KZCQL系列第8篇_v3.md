# 八维度审查，怎么系统评估多智能体

![封面](/workspace/KZCQL/04_工作区/产出归档/20260510_KZCQL系列第8篇/images/08_cover.jpg)

上期聊了架构审查的进化实录，系统从74分一路爬到96分，整条链路算是跑通了。但流程跑通不代表系统就没问题，那期最后留了个悬念，系统到底健不健壮，不是靠感觉就能判断的。

你搭了一个多智能体系统，跑了几轮，感觉还行。

但「感觉还行」这四个字，能信吗？

我之前也觉得能信。KZCQL 系统跑了好几篇文章，评审流程走完了，文章发出来了，读者反馈也还行，那系统应该没什么大问题吧？

后来发生了一件事，让我彻底改变了这个想法。

上期聊完进化实录之后，我对 A1（架构审查专家）的审查框架做了一次大升级。之前 D5（验证链路完整性）这个维度是串行检查的，一个 Agent 要查 187 条规则、203 个执行步骤，上下文窗口根本装不下，很多问题查不透。所以我把它拆成了四个并行子 Agent，每个只负责一个切面，检查深度直接翻了好几倍。

检查标准变了，分数自然也要重新量。就像考试从开卷变成闭卷，不是你变差了，是尺子变严了。

重新审查的结果出来，84.15 分，B 级。

怎么说呢，不是说分数有多低，而是报告里列出来的问题，全是那种「我怎么没发现」的类型。规则写了但没人执行，流程设计了但没串起来，术语在这个文件里叫这个名字、在另一个文件里又换了叫法。

那一刻我才意识到，评估一个多智能体系统，靠「感觉」是靠不住的。你需要一套系统性的方法，一个能从八个维度把系统拆开来看的框架。

这就是今天想聊的东西。

**感觉靠不住，你需要一把尺子。**

![审查概念](/workspace/KZCQL/04_工作区/产出归档/20260510_KZCQL系列第8篇/images/08_inspection.jpg)

**为什么「感觉好」不等于「真的好」**

多智能体系统有个很特殊的属性，它的复杂性不是线性的，是指数级的。

什么意思呢？你一个 Agent 的时候，规则写清楚就行。两个 Agent，你要考虑它们之间的分工。三个 Agent，你要考虑信息传递。等你到了七八个 Agent，互相之间有引用、有依赖、有上下游关系，任何一个环节出了问题，都不一定能从最终产出里看出来。

举个真实的例子。KZCQL 系统里有一条规则，叫「配图必须去水印」。这条规则在撰写规则里写得明明白白，W1（撰写 Agent）的规范里也提到了。但问题是，没有任何一个流程节点去验证「W1 到底有没有执行去水印这个步骤」。

规则存在，流程悬空。

结果就是，有时候图片去水印了，有时候忘了，全凭 W1 那一轮的上下文窗口里还记不记得这件事。文章最终发出来，读者看到图片上有水印的概率大概是一半一半。但你从文章质量上根本看不出来这个问题，因为评审 Agent 拿到的输入是 Markdown 文本，它根本看不到图片。

这种问题，你靠「跑几轮看看效果」是发现不了的。你得有一个专门的 Agent，拿着所有规则文件和所有 Agent 规范，一条一条去对，看每条规则到底有没有被执行、有没有被验证。

这就是 A1 存在的意义。

**A1 不看文章好不好，它看系统健不健。**

![八维度全景](/workspace/KZCQL/04_工作区/产出归档/20260510_KZCQL系列第8篇/images/arch-eight-dimensions.png)

**八个维度，八个切面**

A1 的八维度审查框架，是我在实践中一步步摸索出来的。一开始只有四五个维度，后来发现有些问题怎么都覆盖不到，就一个一个加，加到第八个维度的时候，才觉得差不多能把系统的方方面面都看清楚了。

我自己也还在摸索，但这八个维度确实是踩了很多坑之后才定下来的。

八个维度，满分 100 分，每个维度有不同的权重。权重不是拍脑袋定的，是根据这个维度对系统健康度的影响程度来分配的。

**D1，规范一致性，满分 12。**

这个维度检查的是所有文件之间是否「说的一样」。术语一致不一致，评分标准一致不一致，版本号对不对得上。

听着很简单对吧？但实际做起来，问题多得让你怀疑人生。举个例子，KZCQL 系统里有个 Agent 叫规则修改 Agent，代号 E1（规则修改专家）。在编排指南里，它的路径写的是 `02_子Agent规范/架构专家Agent.md`，但实际文件在 `02_子Agent规范/架构专家组/架构专家Agent.md`，差了一个「组」字。这种路径错误，E1 在执行规则变更的时候就会引用失败。

还有一个更隐蔽的问题。前置撰写规则里列了 18 个禁用词，但快速执行手册里记录的是 14 个，R2（全量评审专家）的规范里引用的也是 14 个。也就是说，有 4 个新增的禁用词，评审 Agent 根本不知道要去检查。

这些问题不影响单次执行，但会慢慢侵蚀系统的可靠性。就像你家里有一扇门锁坏了，平时不出事，但一旦出了事就是大事。

**D2，流程完整性，满分 12。**

这个维度检查的是流程有没有断点。核心流程能不能从头走到尾，异常处理有没有兜底，流程图和文字描述是不是对得上。

KZCQL 的核心流程是 W1 写稿，R1（事实核查专家）事实核查，R2 全量评审，R4（评级路由专家）评级路由，W2（迭代修改专家）修改，R3（差异评审专家）差异评审，再 R4 终审。这条链路看起来很清晰，但如果你仔细看异常处理，就会发现很多细节。

R1 查出事实错误怎么办？跳过 R2，直接进 R4 降级评级，然后 W2 修改，修改完 R1 复检。这个路径是完整的。

R3 发现 W2 引入了新问题怎么办？标记「需 R1 复检」，阻断流程，退回 W2 重新修改。这个路径也是完整的。

迭代超过 5 轮怎么办？P0 是达到 85 分 A 级可以直接退出，P1 是连续 2 轮分数没提升就退出，P2 是满 5 轮兜底退出。三级退出机制，层层递进。

这些异常路径，平时很少触发，但一旦触发，如果没有设计好，整个流程就会卡住或者跳步。D2 就是确保这些路径都是通的。

**D3，规则覆盖度，满分 12。**

这个维度检查的是该有的规则是不是都有了。文章从撰写到交付的每个阶段，每个已知的风险点，是不是都有对应的规则覆盖。

有个概念叫「悬空文件」，就是文件存在但没有被任何流程引用。KZCQL 早期就出现过这种情况，发布前检查清单写得挺详细的，但发布流程里压根没提到这份清单。清单就躺在目录里，像一个没人看的告示牌。

还有一个更严重的问题，去 AI 感检测清单里有 27 项通用检测，涵盖了句式、用词、结构、逻辑四个层面。但这 27 项检测，W1 的四层自检里没有逐项检查，R2 的 10 维度评分里也没有覆盖。也就是说，这 27 条规则写了等于没写。

**D4，职责清晰度，满分 12。**

这个维度检查的是每个 Agent 的职责是不是清楚，有没有重叠，有没有遗漏，有没有冲突。

举个例子，R1 做事实核查，R2 的评分维度里有一项叫「事实准确性」。这两个是不是职责重叠？其实不是，R1 是逐条核查事实，R2 是评估事实核查的结果是否被正确处理。但如果不把边界划清楚，R1 和 R2 可能会对同一个事实问题重复判断，浪费 Token 还可能给出矛盾的结论。

W2 的职责也很有意思。W2 的核心原则是「最小化改动」，只改 R2 指出的问题。但规则同时要求 W2「必须修复所有 P0 问题」。如果 R2 指出的 P0 问题和最小化改动原则冲突了怎么办？这种职责内部的矛盾，如果不提前设计好优先级，W2 执行的时候就会纠结。

**D5，验证链路完整性，满分 15。**

这是八个维度里权重最高的，也是最有意思的一个。

![D5并行拆解](/workspace/KZCQL/04_工作区/产出归档/20260510_KZCQL系列第8篇/images/arch-d5-parallel.png)

D5 回答的核心问题是，你写的每一条规则，到底有没有被强制执行？不是「应该执行」，不是「最好执行」，而是「在流程设计上，不执行就过不去」。

这个维度我拆成了四个并行子 Agent 来检查。

D5-1 检查规则到执行的映射，每条「必须」或「禁止」规则，是不是在至少一个 Agent 的工作流里有对应的步骤。权重 30%。

D5-2 检查执行到验证的映射，每个执行步骤后面，是不是有验证机制来确认执行到位了。权重 25%。

D5-3 检查悬空规则，有没有规则写了但没有任何 Agent 去执行的情况。权重 25%。

D5-4 检查盲评风险，评审 Agent 是不是被要求评审它根本拿不到的信息。权重 20%。

为什么要拆成四个并行？因为 D5 的检查量太大了。KZCQL 系统有 13 个规则文件、187 条规则、11 个 Agent 规范、203 个执行步骤。如果串行检查，一个 Agent 的上下文窗口根本装不下这么多信息。拆成四个并行子 Agent，每个只负责一个切面，检查的深度和准确度都会大幅提升。

**D5 的真实进化，从 6.85 到 13.15。**

![D5进化](/workspace/KZCQL/04_工作区/产出归档/20260510_KZCQL系列第8篇/images/08_d5_evolution.jpg)

这块我想展开讲讲，因为 D5 的进化过程特别能说明问题。

有一轮审查，D5 得了 11.15 分，C 级。当时我觉得还行，毕竟其他维度也才 B 级。

下一轮，D5 直接掉到 7.15 分，D 级。

你没看错，不是进步了，是退步了。分数掉了 4 分。你敢信？

为什么会退步？因为那轮审查的时候，D5-1 做了一次更深入的检查，发现了很多之前没查出来的悬空规则。之前查出来 3 个 P0 悬空规则，这轮又翻出来一堆。不是系统变差了，是检查变严了。

再下一轮，D5 还是 6.85 分，D 级。几乎没动。

这轮的问题很明确，三个 P0 悬空规则一个都没修。发布前检查清单还是悬空的，R1 配图路径还是没声明，去 AI 感的 27 项通用检测还是没有 Agent 执行。不是不知道怎么修，是修起来牵涉太多文件，改完一个要检查一堆关联。

最后一轮，D5 跳到 13.15 分，A- 级。

从 6.85 到 13.15，涨了 6.3 分，几乎翻倍。

来看看具体发生了什么。

D5-1 规则到执行映射，从 10 分涨到 15 分，满分。那三个 P0 悬空规则全部修复了。发布前检查清单嵌入到了 R4 的执行流程里，R1 增加了配图文件路径列表作为必需输入，去 AI 感的 27 项通用检测集成到了 R2 的 Step 4.4。悬空率从大约 7% 降到了 1.1%，187 条规则里只有 2 条还处于悬空状态。

D5-2 执行到验证映射，从 7 分涨到 10 分。每个执行步骤后面的验证机制补全了不少，但还没到满分，说明有些执行步骤的验证还不够强制。

D5-3 悬空规则检测，从 6 分直接跳到 15 分，满分。4 个悬空文件和 1 个路径错误全部修复。之前有些规则文件存在但没被任何流程引用，现在每个规则文件都被至少一个 Agent 的规范引用了。

D5-4 盲评风险检测，从 4 分涨到 13 分。R1 配图路径未声明的 P0 问题修复了，评审 Agent 不再被要求评审它拿不到的信息。

**规则写在纸上不叫有规则，执行路径上有检查点才叫。**

这轮 D5 的跃升，直接把系统总分从 78.85 拉到了 88.65，在新审查体系下首次进入 B+ 等级。一个维度的修复，带动了整个系统的升级。

**D6，人类反馈闭环，满分 12。**

这个维度检查的是人的反馈能不能有效地变成系统改进。不是「收集了反馈」，而是「收集了之后有没有分析、有没有转化成规则变更、变更之后有没有验证」。

KZCQL 的反馈闭环是这样的，人类终审意见会被记录到反馈记录文件里，A1 在审查时会读取这些反馈，检查有没有重复出现的问题。如果同一个问题出现两次以上，就会触发规则修改流程，由 E1 来落实变更。变更完成后，下一轮 A1 审查会验证这个问题是否真的解决了。

这个闭环听起来很完美，但实际运行中有很多摩擦。比如人类反馈有时候是自由文本，没有结构化，A1 很难自动分类。再比如有些反馈涉及的是「感觉」层面的东西，比如「这篇文章读起来有点 AI 味」，很难直接转化成一条可执行的规则。

D6 目前是 10.5 分，B+ 级。还有提升空间，但核心机制已经跑通了。这块其实吧比较难搞，人的反馈有时候就是很模糊的，不像代码 bug 那样有明确的对错。

**D7，产出物可追溯性，满分 13。**

这个维度检查的是每个 Agent 的每次产出是不是都有归档，归档是不是完整、可检索、可回溯。

KZCQL 的归档体系是按文章组织的，每篇文章有一个独立的归档目录，里面有稿件目录、评审目录、配图目录。评审报告的命名格式是固定的，包含 Agent 代号、轮次、日期。通过这些归档，可以回溯任意一篇文章的完整评审历史。

D7 目前是 11.5 分，B+ 级。主要扣分点是一些命名格式在不同文件中的描述不完全一致。

**D8，新增规则集成度，满分 12。**

这个维度检查的是当你新增一条规则或一个 Agent 的时候，是不是所有该知道的地方都知道了，有没有产生孤岛。

KZCQL 早期就吃过这个亏。很多朋友可能不知道，新增了一份通用配图设计规则，但只有 W1 的规范引用了，W2、R2、R3 都不知道这份规则的存在。结果就是 W1 按新规则生成配图，W2 修改的时候不知道新规则的要求，R2 评审的时候也不知道要按新标准检查。

D8 目前是 10 分，B 级。新增规则的集成流程已经建立起来了，但历史遗留的集成缺失还在逐步修补。

**什么样的系统算及格？**

![评分标准](/workspace/KZCQL/04_工作区/产出归档/20260510_KZCQL系列第8篇/images/08_scoring.jpg)

聊完八个维度，你可能想问，到底多少分算好？

根据我这十几轮审查的经验，大概可以分几个档。坦率的讲，这些档位不是理论推导出来的，是实打实踩坑踩出来的。

60 分以下，D 级或 F 级，系统有严重缺陷。不是某个维度差，是多个维度同时有问题。规则悬空、流程断裂、职责混乱，这种系统跑出来的结果完全不可预测。

60 到 75 分，C 级，系统能跑但不可靠。核心流程是通的，但异常处理有漏洞，有些规则写了没执行，有些 Agent 的职责边界不清晰。偶尔能产出好结果，但质量不稳定。

75 到 85 分，B 级，系统基本可靠。核心流程完整，大部分规则有执行覆盖，异常处理有兜底。日常使用没什么大问题，但还有一些 P1 级的优化空间。

85 到 95 分，A 级，系统比较健壮。八个维度都在 B+ 以上，没有 P0 问题，P1 问题也很少。这种系统的输出质量比较稳定，你可以放心地让它自动运行。

95 分以上，S 级，接近完美。上期聊的那个审查体系下最高到过 99 分，但那是旧尺子量的。换成新尺子之后，全面审查的最高分是 88.65 分，B+ 级。尺子越严，分数越低，但发现的问题越真实。

我有时候觉得，90 分可能就是一个多智能体系统的「实用天花板」。不过呢，不是不可能到 95，而是到了 90 分以后，每提升 1 分的投入产出比会急剧下降。从 74 分到 82 分，修的是 P0，每一分都关乎系统能不能正常运行。从 82 分到 89 分，修的是 P1，每一分关乎系统的稳定性。但从 89 分往上，修的是 P2，是「锦上添花」的优化，投入的时间和精力可能比之前加起来还多。

**这套框架，哪些是通用的？**

八个维度里，我觉得 D1 到 D5 是通用的，任何多智能体系统都需要关注。

规范一致性，不管你搭什么系统，文件之间的术语和定义必须统一。你不可能在一个文件里叫「用户」，另一个文件里叫「客户」，第三个文件里叫「User」，然后指望系统正常运行。

流程完整性，任何有流程的系统都需要。哪怕是最简单的三步流程，你也要考虑中间断了怎么办。

规则覆盖度，只要你的系统有规则，就需要检查规则是不是被执行了。

职责清晰度，只要你有多个 Agent，就需要确保它们的职责不重叠不遗漏。

验证链路完整性，这是最核心的一个。不管你的系统多复杂，每条规则都必须有对应的强制执行检查点。没有检查点的规则，就是一张废纸。

D6 到 D8 相对更领域特定一些。D6 人类反馈闭环，对于需要持续迭代的系统很重要，但对于一次性任务可能不那么关键。D7 产出物可追溯性，对于需要审计和回溯的系统很重要，但对于快速原型可能过度设计。D8 新增规则集成度，对于经常变更的系统很重要，但对于稳定的系统可能不是痛点。

**怎么在自己的项目里用？**

如果你也在搭多智能体系统，想用这套框架来评估，我的建议是从 D5 开始。

不是从 D1 开始按顺序来，而是先抓 D5。因为 D5 是最容易暴露问题的维度，也是最直接影响系统可靠性的维度。你花半天时间把所有「必须」和「禁止」规则列出来，然后逐条检查有没有对应的执行步骤和验证机制，大概率会发现一堆问题。

修完 D5，再去看 D1 和 D4。D1 解决的是一致性问题，D4 解决的是职责问题。这两个维度修完，系统的基本骨架就稳了。

D2 和 D3 可以并行处理，一个查流程，一个查规则覆盖。

D6、D7、D8 放在最后，等核心问题都解决了再来优化。

做不动的时候怎么办？先只做 D5-3 悬空规则检测，把所有「有规则但没执行」的情况找出来，挑最严重的 3 个修掉。这一个动作，可能就能让你的系统可靠性提升一个档次。

这套框架也不是万能的。它擅长的是发现结构性问题，对于「Agent 输出质量好不好」这种更上层的问题，还需要配合评审体系一起用。A1 管的是系统的骨架健不健壮，R2 管的是文章的质量过不过关。两层检查，各有分工。

**下一期，我们聊人类反馈闭环。**

这个话题很有意思，因为它是整个系统里唯一一个「人」真正参与的环节。Agent 写稿、Agent 评审、Agent 修改，但最终说「行」还是「不行」的，还是人。人的反馈怎么收集、怎么转化、怎么验证，直接决定了这个系统能不能越跑越好。

如果你也在搭多智能体系统，或者正在被「系统跑起来了一但质量不稳定」这个问题困扰，希望这套八维度框架能给你一个排查的思路。觉得有用的话，转给你身边也在搞 Agent 协作的朋友，他们可能正好需要。
