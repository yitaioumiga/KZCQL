# AI系统不是建完就完

上期聊了八维度审查框架，最后一个维度 D6 人类反馈闭环，当时只开了个头，说这个维度检查的是人的反馈能不能有效地变成系统改进。今天就把这个话题彻底展开。

先说一个结论放在前面，多智能体系统不是建完就完，它需要持续进化，而人类反馈就是进化的燃料。

你可能觉得这话像废话。哪个系统不需要持续维护呢？但多智能体系统有个很特殊的地方，它的很多问题是 Agent 自己发现不了的。Agent 写稿 Agent 评审 Agent 修改，整条链路看起来很完整，但最终说「行」还是「不行」的，还是人。人的判断是整个系统里唯一不可替代的环节。

今天想聊两件事，结构化反馈模板的设计思路，以及规则演进路径是怎么跑通的。这两件事，是我踩了无数坑之后才慢慢想明白的。

**Agent 的自我评价，靠不住。**

![封面](images/09_cover.jpg)

**为什么人类反馈是必要的**

很多人可能觉得，既然已经有 R1 做事实核查、R2 做全量评审、R3 做差异评审，那人的作用是不是越来越小了？

其实不是的。

Agent 的评审有个根本性的局限，它用自己的尺子量自己。R2 的评分标准是规则文件里定义的，W1 的写作规范也是规则文件里定义的。如果规则本身有盲区，R2 就不可能发现 W1 的问题。

举个 KZCQL 真实发生过的事。

第 8 篇文章里，W1 写了一段关于架构审查分数的描述，引用了新审查体系下的 84.15 分和 88.65 分。看起来没问题，数据是对的，来源也是对的。

但问题是，第 7 篇文章已经发布过了，里面写的是旧审查体系下系统从 74 分一路爬到 96 分。读者看完第 7 篇，知道这个系统最高到过 99 分。然后第 8 篇突然说新体系下只有 84.15 分，读者会怎么想？这两个数据放在一起，看起来是矛盾的。

R1 查了事实，84.15 这个数字没错。R2 评了质量，这段话写得也通顺。R3 做了差异评审，修改前后也没什么问题。

三个 Agent 全部通过。

但人一看就知道有问题。不是数据错了，是跨篇的事实冲突。第 7 篇建立的读者认知和第 8 篇呈现的数据之间存在张力，这个张力 Agent 感知不到，因为它不会「读」已发布文章在读者心里留下的印象。

这种问题，只有人能发现。

**反馈不是纠错，是进化信号。**

我之前也觉得，人的反馈就是帮系统「纠错」的，哪里不对指出来，改掉就好了。后来发现不是这么回事。

纠错是修一个具体的 bug，进化是让系统以后不再犯同类 bug。这两件事的层级完全不同。

当你说「这段话和上一篇文章冲突了」，你不仅仅是在指出一个错误，你是在告诉系统，它缺少一种能力，跨篇历史事实一致性检查。这个能力缺失不是修一段话就能解决的，它需要在 W1 的自检清单里加一条检查项，在 R1 的核查规范里加一条规则，甚至需要创建一份作者事实档案作为所有 Agent 的输入。

一次反馈，触发三个层面的变更，这才是反馈真正的价值。

**非结构化反馈，是一场灾难。**

![非结构化反馈](images/09_unstructured.jpg)

说到人类反馈，我踩过的最大的坑，就是反馈太随意。

坑一，反馈太模糊。

「感觉不太好。」「有点 AI 味。」「读起来不太顺。」

这种反馈你让 Agent 怎么改？「不太好」是哪里不好？是逻辑不好还是语言不好？是开头不好还是结尾不好？AI 味是哪个词 AI 味？是句式太整齐还是用词太书面？

Agent 收到这种反馈，只能猜。猜错了就过度修改，把本来没问题的部分也改了。猜对了也是碰运气，下次遇到类似问题还是不知道怎么处理。

坑二，反馈太情绪化。

「这写的是什么东西。」「完全不行，重写。」

你想想，Agent 收到「完全不行重写」这条指令会怎么做？它会把整篇文章推翻重来，包括那些本来写得很好的部分。结果就是，修改幅度直接飙到 30% 以上，R3 的差异评审根本看不过来，反而引入了新的问题。

坑三，反馈太零散。

微信消息里说一句，评审文件里批一句，口头聊天时提一句。没有汇总，没有分类，没有优先级。

这种零散的反馈，最大的问题是不追踪。你说了三次「注意跨篇一致性」，但系统里没有任何地方记录这件事，第三次说完，第四次照样犯。因为没有结构化，反馈就只是一阵风，吹过就没了。

**没有结构化的反馈，就是在浪费人类最宝贵的判断力。**

你花时间读了文章，发现了问题，想出了改进方向，这些判断力是非常有价值的。但如果反馈是非结构化的，Agent 没法有效执行，你的判断力就被浪费了。

所以后来我设计了一套结构化反馈模板。

**三个字段，解决反馈的结构化问题。**

模板其实不复杂，核心就三个字段。

**问题描述**，发生了什么事，具体是哪里出了问题。这个字段要求用事实描述，不用情绪表达。不是「写得不好」，而是「第 3 段第 2 句引用的 84.15 分与第 7 篇已发布的 96 分存在数据张力，可能造成读者困惑」。

**严重程度**，这个问题有多严重。我分了三个等级，P0 是事实错误或跨篇冲突，必须立刻修。P1 是质量缺陷，影响阅读体验，本轮修。P2 是优化建议，不修也行，但修了会更好。

**修改建议**，具体怎么改。不是「改好一点」，而是「在第 3 段第 2 句后补充说明，解释新旧审查体系的差异，消除读者可能的困惑」。

为什么是这三个字段？

因为它们分别回答了三个问题。是什么，有多严重，怎么办。这三个问题是任何反馈都必须回答的，缺一个就不完整。

只用「问题描述」，Agent 知道出了问题但不知道优先级，可能去修一个 P2 而忽略了一个 P0。只用「严重程度」，Agent 知道问题很严重但不知道具体是哪里出了问题。只用「修改建议」，Agent 知道怎么改但不知道为什么要改，下次遇到类似情况还是不会判断。

三个字段合在一起，才构成一条完整的反馈。

**用真实案例跑一遍这个模板。**

回到刚才那个跨篇事实冲突的例子。

问题描述，第 8 篇引用的架构审查分数（84.15/88.65）与第 7 篇已发布的进化数据（74 到 96）存在数据张力。第 7 篇给读者建立的认知是系统最高到过 99 分，第 8 篇呈现的新体系分数可能让读者困惑。根因是 W1 的自检清单（L2）和 R1 的核查规范（V-02）此前没有「跨篇历史事实一致性」检查项。

严重程度，P0。跨篇事实冲突会直接损害系列文章的可信度。

修改建议，W1 L2 新增「跨篇历史事实一致性」检查项，R1 V-02 扩展定义将跨篇一致性纳入核查范围，创建作者事实档案作为 W1 和 R1 的必需输入。

你看，这样一条反馈，Agent 收到之后就知道该做什么了。不是模糊的「注意一致性」，而是具体的三个动作，每个动作都有明确的执行主体和执行内容。

再举一个例子。

问题描述，再平衡段落中，「这恰恰是定投初期该有的状态」之后直接跳到「我每月投 500 块」，两句话之间缺乏逻辑衔接。读者读完第一句期待看到解释（为什么这是定投初期该有的状态），但第二句突然变成了操作描述（每月投多少），逻辑链条断裂。

严重程度，P1。不影响事实正确性，但影响阅读体验。

修改建议，在两句话之间补充过渡句，解释为什么当前状态是正常的，然后再引出具体的定投操作。

这个反馈后来也转化成了一条规则。W1 L2 新增了「段落逻辑连贯性」检查项，专门检测段落内部是否存在「观点直接跳到操作」的断链问题。R2 的 D4 维度评分标准也做了细化，把段落逻辑连贯性纳入了扣分范围。

**模板的扩展字段。**

三个核心字段能解决大部分问题，但有些场景需要更多信息。

**来源追踪**，这条反馈是在哪个环节、哪篇文章、哪个版本中发现的。这个字段是为了做根因分析。如果同一个问题在多篇文章中出现，来源追踪能帮你快速定位是哪个 Agent 的规范有缺陷。

**关联规则**，这条反馈涉及哪些现有规则。如果反馈指出的问题已经有对应规则但没被执行，那问题不在规则缺失，而在执行链路断裂。如果没有对应规则，那需要新增规则。

**验证方法**，修改完之后怎么验证问题真的解决了。这个字段经常被忽略，但特别重要。你改了规则，怎么知道下次不会再犯？得有一个验证手段。可以是 A1 下一轮审查时重点检查这个点，也可以是 R1 在核查时增加一个专项检查。

**反馈的使用流程。**

整个流程跑起来是这样的。

发现问题，人在终审时发现一个质量或事实问题。

填写模板，按三个核心字段加扩展字段把反馈结构化。

转化为规则变更，E1（规则修改 Agent）解析反馈，分析是规则缺失还是执行链路断裂，设计变更方案。

验证闭环，变更完成后，下一轮审查验证这个问题是否真的解决了。如果解决了，这条反馈就闭环了。如果没解决，分析原因，再调整。

**反馈不是终点，规则变更是终点。**

![规则演进](images/09_evolution.jpg)

**规则演进路径**

反馈收集来了，结构化也做好了，下一步就是把它变成系统的能力提升。

这个环节的主角是 E1，规则修改 Agent。

E1 的工作流程是这样的。收到一条结构化反馈之后，先分析这条反馈指向的是什么类型的问题。是规则缺失（系统里根本没有这条规则），还是规则不足（有规则但不够细），还是规则错误（规则本身写错了）。

分析完之后，E1 会设计变更方案。变更方案有三种类型。

**新增规则**，系统里没有覆盖这个问题的规则，需要从零创建。比如「跨篇历史事实一致性」这条规则，就是从零创建的。之前系统里完全没有这个概念，是第 8 篇的跨篇冲突事故逼出来的。

**修改规则**，系统里有相关规则但不够准确或不够完整。比如 R1 的 V-02 事实冲突检查，原来只检查文章内部的事实一致性，修改之后扩展到了跨篇一致性。规则本身没变，但检查范围扩大了。

**删除规则**，系统里有规则但已经不适用了，或者与其他规则冲突了。这种情况比较少见，但也会发生。比如早期有一条规则要求「W2 修改幅度不能超过 10%」，后来发现这个限制太死，经常导致 P0 问题无法彻底修复，就删除了，换成了更灵活的「修改幅度超过 30% 退回 R2 全量评审」。

**变更必须验证，不能改了就完。**

这是我踩过的另一个坑。

有一次 E1 修改了一条规则，改完之后直接提交了，没有跑验证。结果下一篇文章出来，发现修改引入了新的问题。新规则和另一条现有规则产生了冲突，W1 执行的时候左右为难。

从那以后，E1 的变更流程里强制加了一个验证环节。规则变更完成后，必须跑一轮 A1 审查，确认变更没有引入新的 P0 或 P1 问题，才能正式生效。

这个验证环节看起来多花了一轮审查的时间，但省掉了后面无数潜在的麻烦。就像你改了代码要跑测试一样，改了规则也要跑审查。

**跨篇一致性规则是怎么来的。**

我拿那个跨篇事实冲突的例子，完整跑一遍规则演进路径。

第 8 篇发布后，读者反馈第 8 篇引用的分数和第 7 篇的数据存在张力。

我按模板填了结构化反馈。问题描述里写清楚了具体是哪两段数据冲突，严重程度标了 P0，修改建议列了三个动作。

E1 接到反馈后做了根因分析。它发现 W1 的 L2 自检清单里有 12 项检查，但没有「跨篇历史事实一致性」这一项。R1 的 V-02 事实冲突检查也只覆盖文章内部，不覆盖跨篇。也就是说，这个问题不是 Agent 执行不到位，是规则本身有盲区。

E1 设计了变更方案。W1 L2 新增第 13 项检查「跨篇历史事实一致性」，要求 W1 在撰写系列文章时，必须核对前序文章中已建立的事实数据。R1 V-02 扩展定义，将跨篇一致性纳入事实冲突核查范围。同时创建一份「作者事实档案」，记录作者的真实资产数据、投资操作、工作状态等事实，作为 W1 和 R1 的必需输入。

变更完成后，A1 跑了一轮专项审查，确认三项变更都没有引入新问题。正式生效。

下一篇文章，W1 在自检时就会核对前序文章的事实数据，R1 在核查时也会检查跨篇一致性。这个问题，从那以后再也没出现过。

**一次事故，变成了一条系统规则。这就是反馈闭环的价值。**

**反馈闭环的三个层次。**

![三个层次](images/09_three_layers.jpg)

聊到这里，我想把反馈闭环分成三个层次来讲。这三个层次是我自己在实践中慢慢意识到的，不是一开始就设计好的。

**第一层，单次反馈闭环。**

发现问题，修复问题，验证问题解决了。这是最基础的层次。

比如你发现第 8 篇有个跨篇冲突，改了那段话，确认改完没问题了。闭环完成。

这个层次解决的是「这篇文章的问题」，但不解决「下一篇文章会不会有同样的问题」。

**第二层，规则演进闭环。**

发现问题，分析根因，变更规则，验证规则生效。这是中间层次。

跨篇一致性那个例子就是第二层闭环。不是只改那一段话，而是从规则层面确保以后不会再犯。W1 的自检清单加了检查项，R1 的核查规范扩展了范围，作者事实档案创建了。这些变更一旦生效，后续所有文章都会受益。

这个层次解决的是「这类问题」，比第一层高了一个维度。

**第三层，系统自进化闭环。**

持续反馈，持续改进，系统越来越强。这是最高层次。

第一层和第二层都是被动的，出了问题才去修。第三层是主动的，系统有一套机制能够持续吸收反馈、持续优化规则、持续提升能力。

KZCQL 目前处于哪一层？

坦率的讲，大部分时候在第二层。出了问题，分析根因，改规则，验证。这套流程已经跑通了，而且确实在发挥作用。第 8 篇的跨篇冲突事故催生了三条新规则，作者事实档案、段落逻辑连贯性、跨篇历史事实一致性，这三条规则让后续文章的质量有了明显提升。

但第三层还差得远。系统还不能主动识别「哪些规则需要优化」，也不能自动从反馈中提取模式。目前所有的规则变更都是人驱动的，人发现问题，人填反馈模板，E1 执行变更。如果人不反馈，系统就停滞了。

我有时候觉得，第三层可能需要更高级的能力，比如让 A1 在审查时自动识别高频反馈模式，主动建议规则变更。或者让系统定期回顾历史反馈，找出重复出现的问题类型，提前堵住规则漏洞。这些能力目前还没有，但方向是清晰的。

**人的判断力，是系统进化的上限。**

回到最开始的话题。为什么人类反馈是多智能体系统持续进化的核心动力？

因为 Agent 的能力边界是由规则定义的，而规则的边界是由人的判断力拓展的。

Agent 不知道「跨篇一致性」是个问题，因为它的规则里没有这个概念。是人发现了这个问题，把判断转化成了规则，规则拓展了 Agent 的能力边界。

结构化反馈模板，就是把人的判断力转化为系统进化力的工具。三个字段，问题描述、严重程度、修改建议，看起来简单，但它们确保了人的每一次反馈都能被系统有效吸收，而不是随风飘散。

规则演进路径，就是系统吸收反馈之后的消化机制。E1 解析反馈、分析根因、设计变更、验证效果，这套流程确保反馈不只是修了一个 bug，而是变成了系统的永久能力。

如果你也在搭多智能体系统，或者正在用 AI Agent 做内容创作，我建议你认真对待人类反馈这个环节。不是随便看看说一句「还行」或「不行」，而是用结构化的方式把你的判断力注入到系统里。

你的每一次认真反馈，都是在帮系统进化。
