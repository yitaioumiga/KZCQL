## 一、执行摘要

**核心发现**:

1. **SHA256校验机制存在但未启用** -- `SCRIPT_SHA256_EXPECTED` 值为 `"AUTO_GENERATED"`，导致校验逻辑在第54行直接 `return 0` 跳过，实际不执行任何校验

4. **正面改善** -- ARC-01检查项已从"存在但未调用"变为"已激活"（第377-389行），脚本覆盖率从28.6%提升到35.7%

**评分: 9/15**（中等风险，较P37基线的10/15下降1分）

**评分下降原因**: 修复措施引入了"虚假安全感"这一新的盲评风险维度，净效果为负面。

## 二、逐项分析

### 2.1 SHA256校验机制评估

#### 2.1.1 机制设计审查

脚本第33-36行定义了校验配置：

```bash
SCRIPT_VERSION="v1.1"
SCRIPT_SHA256_EXPECTED="AUTO_GENERATED"
```

第49-65行实现了校验逻辑：

```bash
verify_script_integrity() {
    local script_path="${BASH_SOURCE[0]}"
    local actual_hash=$(sha256sum "$script_path" 2>/dev/null | awk '{print $1}')

    # 注：AUTO_GENERATED表示首次运行或开发模式，跳过校验
    if [ "$SCRIPT_SHA256_EXPECTED" = "AUTO_GENERATED" ]; then
        return 0
    fi

    if [ "$actual_hash" != "$SCRIPT_SHA256_EXPECTED" ]; then
        echo -e "${RED}[SECURITY ERROR]${NC} 脚本完整性校验失败！"
        exit 2
    fi
}
```

#### 2.1.2 有效性判定: **未生效**

| 评估项 | 结果 | 说明 |
|--------|------|------|
| 校验值是否已填入 | 否 | 值为 `"AUTO_GENERATED"`，非实际SHA256哈希 |
| 校验逻辑是否被执行 | 否 | 第54行 `if` 条件为真，直接 `return 0` 跳过 |
| 校验失败是否有阻断 | 是（设计层面） | 若校验值被正确填入且不匹配，脚本会 `exit 2` |
| 校验覆盖范围 | 仅脚本自身 | 不覆盖日志文件、配置文件等 |

#### 2.1.3 根本问题分析

SHA256校验存在一个**自引用悖论**：

1. 校验值 `SCRIPT_SHA256_EXPECTED` 存储在脚本自身内部（第36行）
2. 如果主Agent修改了脚本内容，它可以同时修改校验值
3. 因此，即使校验值被正确填入，主Agent只需在修改脚本后重新计算SHA256并更新变量，即可绕过校验

**正确的实现方式**应该是将校验值存储在脚本外部的受保护位置（如I1督察的配置文件中），由I1在审计时独立比对。当前实现将"锁"和"钥匙"放在同一个盒子里，形同虚设。

#### 2.1.4 风险等级: P1（中风险）

虽然当前未生效，但设计本身存在结构性缺陷，即使启用也无法提供有效保护。

### 2.2 chmod 444 日志加固评估

#### 2.2.1 机制设计审查

脚本第68-81行实现了日志加固：

```bash
secure_log_files() {
    local log_file="$1"
    local summary_file="$2"

    chmod 444 "$log_file" 2>/dev/null || true
    chmod 444 "$summary_file" 2>/dev/null || true

    echo "=== 安全加固 ===" >> "$log_file"
    echo "日志文件权限已设为只读 (chmod 444)" >> "$log_file"
    echo "注意：在SOLO环境中此设置可被主Agent覆盖，但会留下修改痕迹" >> "$log_file"
}
```

#### 2.2.2 SOLO环境实测验证

**测试1: 当前用户身份**
```
uid=0(root) gid=0(root) groups=0(root)
umask: 0022
```
结论: SOLO环境以root用户运行。

**测试2: chmod 444后能否写入**
```
touch test_file && chmod 444 test_file
echo "data" > test_file   # 成功写入，exit code: 0
cat test_file              # 输出: data
```
结论: root用户可对444文件执行写入操作，chmod 444对root无效。

**测试3: chmod 444后能否chmod回改**
```
touch test_file && chmod 444 test_file
chmod 644 test_file        # 成功，输出: "Override successful"
```
结论: root用户可随时将444权限改回644，chmod 444对root无效。

#### 2.2.3 有效性判定: **完全无效**

| 评估项 | 结果 | 说明 |
|--------|------|------|
| chmod 444能否阻止写入 | 否 | root用户可无视Unix权限模型 |
| chmod 444能否阻止权限回改 | 否 | root用户可随时chmod回644 |
| 是否留下修改痕迹 | 是 | `stat`可显示Modify时间变化 |
| "留下痕迹"是否可被利用 | 部分 | I1督察需主动检查stat信息 |

#### 2.2.4 脚本自身的诚实性评价

值得肯定的是，脚本第80行明确承认了这一限制：

> "注意：在SOLO环境中此设置可被主Agent覆盖，但会留下修改痕迹"

这种自我声明是诚实的，避免了P37中"规范声称与实际不一致"的问题（NR2）。但这同时也意味着，chmod 444的实际安全价值为零——它只是一个"痕迹标记"，而非真正的访问控制。

#### 2.2.5 风险等级: P2（低风险，因诚实声明而降级）

虽然机制无效，但脚本没有虚假声称其有效性，不会误导I1督察。

## 三、新增盲评风险点

### 3.1 NR4: SHA256校验的虚假安全感（中风险）

**问题描述**:

`verify_script_integrity()` 函数的存在可能在I1督察心中建立"脚本完整性已受SHA256保护"的认知。然而：

1. 校验值为 `AUTO_GENERATED`，校验逻辑被跳过 -- **保护未启用**
2. 即使启用，校验值存储在脚本自身内部 -- **保护可被绕过**
3. 脚本头部安全声明（第14-18行）将SHA256校验列为第一项安全措施 -- **暗示保护已生效**

**盲评风险**:

I1督察在审计时，如果看到脚本包含SHA256校验函数，可能推断"脚本完整性已被保护"，从而跳过对脚本内容的逐行审查。这种"看到安全机制就认为安全"的认知偏差，恰好为恶意篡改提供了掩护。

**风险等级**: P1

**与P37 NR2的关系**: P37 NR2是"规范声称与实际不一致"，NR4是"代码存在与实际效果不一致"。两者本质相同但表现形式不同。NR4更具隐蔽性，因为代码确实存在，只是未生效。

---

### 3.2 NR5: "留下修改痕迹"的审计假设（低风险）

**问题描述**:

脚本第80行声称chmod 444"会留下修改痕迹"。这个假设依赖于I1督察主动使用 `stat` 命令检查文件的Modify时间。但：

1. I1督察的审计检查清单（规范v1.0第4.3节）中**没有**包含"检查文件修改时间"这一步骤
2. 即使I1检查了Modify时间，主Agent可以使用 `touch -t` 命令回溯时间戳
3. root用户可以 `chmod 644` 回改权限后再修改，再 `chmod 444` 恢复权限，痕迹仅限于Modify时间

**风险等级**: P2

## 六、关键问题回答

### Q1: SHA256校验机制是否有效？

**答案: 否。当前完全未生效，且即使启用也存在结构性缺陷。**

1. **未生效**: `SCRIPT_SHA256_EXPECTED="AUTO_GENERATED"` 导致校验在第54行被跳过
2. **结构性缺陷**: 校验值存储在脚本自身内部，主Agent修改脚本时可同步更新校验值
3. **正确做法**: 校验值应由I1督察持有，在审计时独立计算并比对

### Q2: chmod 444在SOLO环境中是否有效？

**答案: 完全无效。**

经实测验证：
- SOLO环境以root(uid=0)运行
- root可对444文件执行写入（`echo "data" > file` 成功）
- root可随时将444改回644（`chmod 644 file` 成功）
- Unix文件权限模型对root用户不构成限制

### Q3: 安全加固是否降低了盲评风险？

**答案: 净效果为负面。**

| 措施 | 预期效果 | 实际效果 | 净贡献 |
|------|---------|---------|--------|
| SHA256校验 | 防止脚本篡改 | 未启用 + 虚假安全感 | 负面 |
| chmod 444 | 防止日志篡改 | 对root无效 | 零 |
| ARC-01激活 | 扩大检查覆盖 | 覆盖率+7.1% | 正面 |
| 诚实声明 | 避免误导I1 | NR2风险降级 | 正面 |

### Q4: 仍存在的盲评风险点？

**核心未解决风险**:

1. **脚本可被主Agent任意修改** -- SHA256未生效，且即使生效也可被绕过
2. **日志可被主Agent任意修改** -- chmod 444对root无效
3. **脚本调用可被绕过** -- 无强制调用机制
4. **64.3%的检查项仍依赖自报告** -- 9/14项未被脚本覆盖

### Q5: 新增风险？

1. **NR4（P1）**: SHA256校验函数的存在制造虚假安全感，可能降低I1督察的审查警觉性
2. **NR5（P2）**: "留下修改痕迹"的假设依赖I1主动检查stat，但审计清单未包含此步骤

## 七、改进建议

### 7.1 P0级建议（必须实施）

**建议1: 删除或正确实现SHA256校验**

两个选择：
- **选择A（推荐）**: 删除 `verify_script_integrity()` 函数和相关的安全声明，避免虚假安全感。将脚本完整性验证的责任明确交给I1督察（I1在审计时独立计算SHA256并与基准值比对）
- **选择B**: 将SHA256基准值存储在脚本外部（如I1督察的配置文件），由I1在审计时传入参数进行校验。但这需要修改脚本接口，增加复杂度

**建议2: 删除chmod 444或替换为有效机制**

两个选择：
- **选择A（推荐）**: 删除 `secure_log_files()` 中的chmod调用，保留诚实声明。将日志防篡改的希望寄托在I1督察的主动审计上
- **选择B**: 如果SOLO环境支持 `chattr +i`（不可变属性），使用 `chattr +i` 替代 `chmod 444`。但需注意root仍可通过 `chattr -i` 解除

### 7.2 P1级建议（强烈建议）

**建议3: 更新I1督察审计检查清单**

在检查执行日志管理规范的审计检查清单中增加：
- "检查脚本 `SCRIPT_SHA256_EXPECTED` 是否为 `AUTO_GENERATED`（如是，则SHA256校验未启用）"
- "使用 `stat` 命令检查日志文件的Modify时间，确认无事后修改"
- "检查脚本头部安全声明中列出的措施是否实际生效"

**建议4: 继续扩大脚本覆盖面**

当前5/14项（35.7%），建议将FMT-02（md最终版本检查）、REV-01（R4评级报告存在性检查）纳入脚本，目标覆盖率50%以上。

### 7.3 P2级建议（长期优化）

**建议5: 引入外部完整性基准**

建立一个独立的"脚本基准值"文件，由I1督察维护，存储各版本脚本的SHA256值。I1在审计时独立计算并比对，实现真正的完整性验证。

## 八、结论

P37修复针对P0-3（脚本和日志文件权限未加固）的响应是**及时的，但执行效果不佳**：

1. **SHA256校验**: 代码存在但未启用（`AUTO_GENERATED`），且存在自引用悖论的结构性缺陷。更严重的是，它的存在制造了虚假安全感，这是比"没有保护"更危险的状态
2. **chmod 444**: 在root环境下完全无效，但脚本诚实地承认了这一点，值得肯定
3. **ARC-01激活**: 是本次修复中唯一产生实际正面效果的变更

**评分变化**: 10/15 → 9/15（-1分）

**核心教训**: 在安全工程中，**无效的安全措施比没有安全措施更危险**。没有安全措施时，所有人都知道系统是脆弱的，会保持警惕；有了无效的安全措施，人们会放松警惕，反而降低了系统的实际安全水平。这就是SHA256校验函数虽然"无害"却导致评分下降的原因。

**建议**: 下一步应优先执行建议1（删除或正确实现SHA256校验），消除虚假安全感，然后通过建议3（更新I1审计清单）和建议4（扩大覆盖面）来实质性提升防伪造能力。
