### 4.2 check_arc01()定义但未调用——新的悬空规则

**发现**：pre_delivery_check.sh第191-215行定义了check_arc01()函数，功能为检查归档目录结构（稿件/、评审/、images/子目录是否存在）。但main()函数（第221-377行）中**未调用**此函数。

**影响分析**：

| 维度 | 评估 |
|------|------|
| 悬空类型 | 代码死路径（dead code path） |
| 严重程度 | P1 |
| 影响范围 | ARC-01归档目录结构检查未被自动化覆盖 |
| 与P36对比 | P36时ARC-01被标记为"已落地"（通过ls命令人工验证），P37引入自动化脚本后反而产生了新的悬空 |
| 根因 | 脚本开发时编写了函数但未在main()中接入调用 |

**风险**：
- check_arc01()的存在给人"ARC-01已自动化"的错觉
- 实际上ARC-01仍完全依赖人工检查
- 未来维护者可能误以为ARC-01已被自动化覆盖而放松人工检查

**建议**：
1. 在main()函数中FMT-01检查之后添加check_arc01()调用
2. 需要解决归档目录参数传递问题（当前脚本仅接收md文件路径和关键词，需据此推导归档目录）
3. 或在函数上方添加注释说明暂不启用的原因

### 4.4 自检三问悬空状态评估

**规则原文**（主Agent编排指南v1.7 5.6节）：

> 主Agent在标记"全部通过"前，必须回答：
> 1. 我是否对每个检查项使用了指定的验证工具？
> 2. 我是否记录了每个检查项的工具输出？
> 3. 我是否根据工具输出（而非主观判断）做出通过/不通过判定？
> 任一答案为"否" -> 重新执行检查

**Enforcement机制分析**：

| 要求 | 当前机制 | P36状态 | P37状态 | 变化 |
|------|----------|---------|---------|------|
| 强制回答三问 | 文本描述要求 | 无技术强制 | 无技术强制 | 无变化 |
| 答案记录 | 建议记录到检查执行日志 | 无强制归档 | 无强制归档 | 无变化 |
| "否"触发重检 | 文本描述 | 无自动化阻断 | 无自动化阻断 | 无变化 |
| 答案验证 | 无 | 无第三方验证 | 无第三方验证 | 无变化 |
| 与自动化脚本集成 | 不适用 | 不适用 | **未集成** | 自检三问未嵌入pre_delivery_check.sh流程 |

**结论**：自检三问在P37补丁后**仍完全悬空**。虽然pre_delivery_check.sh建立了自动化防线，但自检三问作为人工检查阶段的质量保障机制，仍然缺乏：
1. 技术强制手段（主Agent可跳过不答）
2. 结构化记录模板（答案格式不统一）
3. 与检查日志的集成（未嵌入日志模板）
4. 第三方验证机制（I1无法审计自检三问答案）

**缓解因素**：pre_delivery_check.sh的自动化检查在一定程度上替代了自检三问的部分功能——脚本强制验证IMG-01/01b/02/FMT-01，不依赖主Agent自觉。但自检三问覆盖的是全部15项检查，而脚本仅覆盖4项，剩余11项的验证质量仍无保障。

### 9.2 总体评估

P37补丁通过建立自动化第二道防线，在Enforcement强度和事故预防能力方面取得了实质性进步。pre_delivery_check.sh的硬性阻断机制有效解决了"主Agent可跳过验证"的核心问题。

但系统仍存在两个结构性短板：
- **目视检查不可靠**：IMG-03/05/06依赖主Agent目视检查，是INC-2026-0520类事故的根本原因，P37未触及
- **自检三问形同虚设**：连续两次评估（P36、P37）均指出此问题，但均未修复

从D级提升至C级是显著进步，但距离B级（>=10分）仍差0.5分。要达到B级，至少需要再修复1项P1级悬空（如激活check_arc01()或为自检三问建立enforcement机制）。
