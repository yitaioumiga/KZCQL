## §11 方法论可执行性规范（汲取评论部精华）

### 11.1 4步行动法格式

当文章涉及方法论/工具分享/操作指南时，每个方法步骤必须包含以下4个要素：

1. **触发场景** -- 什么时候用这个方法
2. **具体动作** -- 具体做什么（不允许抽象口号）
3. **时间成本** -- 需要花多长时间
4. **失败退路** -- 做不动时怎么办

### 11.2 降级动作机制

- 方法段中必须包含「做不动时怎么办」的降级方案
- 降级动作至少2条
- 设计原因：读者可能情绪低落、精力不足、环境受限，需要有更小粒度的替代动作

### 11.3 边界声明机制

- 每个方法都要明确「适用前提」，即什么条件下有效、什么条件下无效
- 边界声明至少2条
- 设计原因：防止「放之四海而皆准的废话」

## §12 文章结构增强选项（汲取资讯部精华）

以下段落不是强制结构，而是根据文章原型可选的增强段落：

### 12.1 影响拆解段（可选）

- 当文章涉及现象解读/行业分析时，可增加「影响拆解」段落
- 按人群分层说明影响（如：不同技术背景的读者/不同使用场景）
- 每个分层至少1条具体影响描述

### 12.2 方法路径段（可选）

- 当文章原型为「工具分享型」或「方法论分享型」时，可增加按时间维度拆解的方法路径
- 今天1-2个低负担动作 / 本周1个进阶动作 / 本月1个长期动作
- 每个动作必须包含5要素：动作动词 + 对象 + 时间 + 适用人群 + 成功判定

### 12.3 误区与避坑段（可选）

- 列出2-3条读者容易犯的错误认知或行动偏差
- 这是一个「防错机制」，帮助读者避免常见误区

### 12.4 替代方案（推荐）

- 结尾部分建议提供至少3条可执行路径作为替代方案
- 回答「如果这个方法不适合我，还有什么选择」
- 给读者退路和备选

### 12.5 背景控制

- 背景描述控制在20%以内，把篇幅留给核心内容和观点
- 背景膨胀>20%应触发修改

## §13 内容深度规范（汲取评论部+内容部v2精华）

### 13.1 素材引用密度

- 文章中引用的具象细节（案例/数据/场景/引语）不少于8处
- 用硬性指标确保文章有足够的支撑

### 13.2 主题锚点落地

- 文章标题核心概念在正文中至少出现3次，每次有叙事支撑
- 分布均匀（开头/中段/结尾各至少1次）
- 防止主题「悬空」

### 13.3 「先理解人再下判断」原则

- 禁止站在道德高点训话
- 先共情再给建议
- 与卡兹克风格「有见识的普通人在认真聊一件打动他的事」一致
- 允许读者不同意：文章中应自然体现「如果你不这么想，也完全没问题」的开放态度，不强迫读者接受单一观点

### 13.4 禁止项补充（汲取评论部7大类）

在卡兹克禁用词基础上，额外禁止：

- 训话句（「你应该」「你必须」「一定要」）
- 空洞大词（「赋能」「认知升级」「意义深远」「底层逻辑」）
- 模板衔接词（「总而言之」「综上所述」「在这个时代」）
- 宏大口号（「等风来不如追风去」）
- 爹味说教（「年轻人应该懂得......」）
- 模板化理中客（「各有各的道理」）
- 情绪操控（制造对立、贩卖焦虑）

## §14 理财/投资类文章风险提示规范（P24补丁）

### 14.1 适用范围

以下类型的文章必须包含完整风险提示：
- 理财访谈系列所有文章
- 涉及基金、股票、定投、理财、投资组合等金融话题的文章
- 提及具体投资标的的文章

### 14.2 风险提示标准模板

文章末尾必须包含以下完整风险提示（使用引用格式）：

```
> 本文仅供学习交流，不构成投资建议。市场有风险，投资需谨慎。过往业绩不代表未来表现。定投不保本，基金名称仅为举例，不构成推荐。
```

### 14.3 风险提示完整性检查清单

| 检查项 | 内容 | 必须 |
|--------|------|------|
| 不构成投资建议 | "本文仅供学习交流，不构成投资建议" | ✅ |
| 市场有风险 | "市场有风险，投资需谨慎" | ✅ |
| 过往业绩声明 | "过往业绩不代表未来表现" | ✅ |
| 定投不保本 | "定投不保本" | ✅ |
| 非推荐声明 | "基金名称仅为举例，不构成推荐" | 如提及具体基金名称 |

### 14.4 收益表述规范

| 允许表述 | 禁止表述 |
|----------|----------|
| "过去X年平均年化收益率" | "保证X%收益" |
| "大概率" | "一定赚钱" |
| "可能获得" | "稳赚不赔" |
| "历史数据显示" | "用了这种方法就能..." |

### 14.5 W1自检项

- [ ] 文章属于理财/投资类（检查关键词：基金、股票、定投、理财、投资组合）
- [ ] 文章末尾包含完整风险提示模板
- [ ] 文中无绝对化收益承诺（"保证"、"一定"、"稳赚"等）
- [ ] 如提及具体基金名称，已声明"仅为举例，不构成推荐"
- [ ] 收益数据使用概率性表述（"可能"、"大概率"）而非确定性表述

### 14.6 免责声明使用场景指南（P25补丁）

**核心判断原则**："是否邀请读者模仿"

| 内容类型 | 示例 | 需要声明 | 声明位置 |
|----------|------|----------|----------|
| **个人经历分享** | "我基本不做止盈，拿住就好" | ❌ 不需要 | - |
| **给读者的参考方案** | "可复用的配置模板" | ✅ 需要 | 方案前 |
| **具体数字规则**（作为方案一部分） | "30%止盈线，卖出50%" | ⚠️ 建议 | 规则前或方案前 |
| **个人资产数据** | "我的AI持仓收益率24.25%" | ❌ 不需要 | - |
| **知识讲解** | "什么是核心-卫星策略" | ❌ 不需要 | - |

**14.7 具体基金/股票名称声明（P26补丁）**

提到具体基金、股票名称时，采用"首次提及+文末确认"原则：

| 声明位置 | 内容 | 示例 |
|----------|------|------|
| **首次提及** | 基金名称后加括号声明 | `沪深300（仅为举例，不构成推荐）` |
| **方案前** | 免责声明中强调 | "**文中基金名称仅为举例，不构成推荐。**" |
| **文末** | 通用风险提示 | "基金名称仅为举例，不构成推荐" |

**声明模板**：

1. **首次提及声明**：
   ```
   沪深300（仅为举例，不构成推荐）
   ```

2. **方案级声明**（模板、配置方案前）：
   ```
   > **免责声明**：以下配置仅供参考，不构成投资建议。具体比例和标的应根据你的风险偏好、投资目标和市场情况来定，必要时咨询专业投资顾问。**文中基金名称仅为举例，不构成推荐。**
   ```

3. **规则级声明**（具体操作规则前）：
   ```
   > **提示**：以下规则仅供参考，每个人的风险偏好不同，具体止盈线应根据自身情况设定。
   ```

**不需声明的情况**：
- 仅分享个人经历，不邀请读者模仿
- 纯知识讲解（如"什么是宽基指数"）
- 陈述客观事实（如"我的持仓数据"）
- 在已有完整风险提示的文章中自然融入的个人做法

## §15 APAG文章结构规范（多平台适配新增）

> 基于Dan Koe的APAG框架，适配公众号长文创作。本节为增强选项，与§3文章结构规范互补，不替代。

### 15.1 APAG四阶段

| 阶段 | 名称 | 占比 | 核心要求 |
|------|------|------|----------|
| **A** | Attention（注意力钩子） | 10-15% | 用四种钩子之一抓住读者注意力 |
| **P** | Perspective（独特视角） | 20-25% | 提出与主流不同的观点或观察 |
| **A** | Advantage（价值交付） | 40-50% | 核心内容，方法论+案例+可执行建议 |
| **G** | Gamify（行动号召） | 10-15% | CTA设计，引导读者行动 |

### 15.2 A阶段：注意力钩子（四种类型）

| 类型 | 说明 | 示例 |
|------|------|------|
| **反直觉事实** | 用一个违反常识的事实开场 | "AI越来越强，却越来越不靠谱" |
| **个人故事** | 用自己的经历引入主题 | "昨天发了一条动态，大意是说..." |
| **痛点共鸣** | 描述读者正在经历的困扰 | "很多人可能觉得，事实核查嘛..." |
| **好奇心缺口** | 提出一个读者想知道答案的问题 | "为什么AI写了10版，还是不像人写的？" |

**规则**：
- 每篇文章至少使用一种钩子类型
- 不要连续两篇文章使用同一种钩子类型
- 钩子必须在文章前3句话内出现

### 15.3 P阶段：独特视角

| 要求 | 说明 |
|------|------|
| 用"我发现"而非"你应该" | 分享发现，不说教 |
| 提供与主流不同的角度 | 即使主流观点正确，也要找到新的切入角度 |
| 建立信任 | 用个人经历或数据支撑观点，让读者觉得"这个人真的经历过" |

### 15.4 A阶段：价值交付

| 要求 | 说明 |
|------|------|
| 方法论可执行 | 遵循§11的4步行动法格式 |
| 案例具体 | 用真实场景而非抽象概念 |
| 有降级方案 | 做不动时怎么办（至少2条） |
| 有边界声明 | 什么条件下有效/无效（至少2条） |

### 15.5 G阶段：行动号召

遵循§17 CTA设计模板。

### 15.6 APAG与现有结构的关系

APAG不替代§3的推荐结构模板，而是提供一种**内容组织思路**：

```
§3推荐结构（形式）          APAG（内容逻辑）
┌─────────────┐          ┌─────────────┐
│ 开头（引子）  │ ← A阶段 → │ 注意力钩子   │
│ 小标题1：引入 │ ← P阶段 → │ 独特视角     │
│ 小标题2-4    │ ← A阶段 → │ 价值交付     │
│ 小标题5-6    │          │             │
│ 结尾（互动）  │ ← G阶段 → │ 行动号召     │
└─────────────┘          └─────────────┘
```

**W1自检项**：
- [ ] 文章使用了至少一种注意力钩子类型
- [ ] 有明确的独特视角（非主流观点的简单复述）
- [ ] 价值交付部分包含可执行的方法论
- [ ] 结尾有CTA（遵循§17）

## §16 故事融入指南（多平台适配新增）

> 故事是公众号长文的核心竞争力。本节规定故事的融入方式、比例和质量标准。

### 16.1 故事比例

| 文章类型 | 故事占比 | 说明 |
|----------|----------|------|
| 公众号长文 | 20-40% | 故事与观点交替推进 |
| 方法论文章 | 15-25% | 以方法论为主，故事作为支撑 |
| 经验分享 | 30-50% | 以故事为主线，方法论嵌入其中 |

**硬性红线**：
- 故事占比不得低于15%（否则文章太干）
- 故事占比不得高于50%（否则缺乏方法论价值）

### 16.2 故事类型

| 类型 | 说明 | 适用场景 |
|------|------|----------|
| **个人经历** | 作者自己的真实经历 | 所有文章（首选） |
| **读者场景** | 描述目标读者可能遇到的场景 | 痛点共鸣类文章 |
| **第三方案例** | 行业案例、名人案例、产品案例 | 方法论/工具分享类 |

**优先级**：个人经历 > 读者场景 > 第三方案例

### 16.3 融入方式

**正确方式：叙事融入式**

故事不应该是独立的"故事板块"，而是自然嵌入观点论述中：

```
❌ 错误（独立故事板块）：
## 我的故事
（大段故事）
## 方法论
（大段方法论）

✅ 正确（叙事融入式）：
观点 → 故事佐证 → 方法论 → 故事验证 → 升华
```

**融入模式**：

| 模式 | 结构 | 示例 |
|------|------|------|
| **观点-故事-方法** | 先抛观点，用故事支撑，再给方法 | "我发现定投最难的不是策略...（故事）...所以我的做法是..." |
| **故事-观点-方法** | 先讲故事，提炼观点，再给方法 | "那天我打开账户一看...（故事）...这件事让我意识到...（观点）...具体怎么做呢..." |
| **方法-故事-验证** | 先给方法，用故事验证有效性 | "我的规则是30%止盈线...（方法）...第4篇文章就是最好的例子...（故事）" |

### 16.4 故事质量标准

| 标准 | 要求 |
|------|------|
| **真实性** | 必须与作者事实档案一致，不得虚构。不确定的事实标注"待确认" |
| **具体性** | 有时间、地点、人物、具体数字，而非泛泛而谈 |
| **相关性** | 故事必须直接支撑观点，不能为了讲故事而讲故事 |
| **简洁性** | 每个故事控制在3-5段（150-300字），不要写成小说 |
| **情感真实** | 不夸大情绪（"太离谱了""我当时就愣住了"），用克制的方式表达 |

### 16.5 故事素材来源

- **作者事实档案**（`01_共享知识库/作者事实档案.md`）：个人经历、资产数据、投资操作、能力画像（§5）
- **用户写作前必问清单**（§8）：每次写作前向用户收集案例方向
- **往期文章**：已发布文章中的案例可以跨篇引用，但不要大段复述

**W1自检项**：
- [ ] 文章中至少包含1个故事（个人经历优先）
- [ ] 故事占比在15-50%之间
- [ ] 故事采用叙事融入式，非独立板块
- [ ] 故事与作者事实档案一致
- [ ] 每个故事有具体的时间/数字/场景

## §17 CTA设计模板（多平台适配新增）

> CTA（Call To Action）是文章结尾的行动号召。好的CTA让读者觉得自然、不生硬，同时有效引导互动。

### 17.1 CTA类型与选择

| CTA类型 | 适用场景 | 示例 |
|---------|----------|------|
| **场景代入式** | 通用（首选） | "如果你也每天在用AI干活，这篇文章应该能帮到你。点个赞、转发给同事，他们可能正好需要。" |
| **好奇心驱动** | 方法论文章、系列文章 | "下一篇我会分享具体怎么搭建这套审查体系，关注我不迷路。" |
| **社群引导** | 需要引流到社群 | "想一起讨论AI写作的，评论区见，或者来社群聊聊。" |
| **轻互动** | 日常文章、随笔 | "觉得有用就点个赞吧，你们的支持是我继续写的动力。" |
| **反问式** | 观点类文章 | "你用AI写东西的时候，有没有遇到过类似的问题？评论区说说。" |

### 17.2 CTA设计规则

| 规则 | 说明 |
|------|------|
| **必须有CTA** | 每篇文章结尾必须有行动号召，不能突然结束 |
| **匹配文章类型** | 方法论文章用好奇心驱动，观点文章用反问式，通用文章用场景代入式 |
| **不生硬** | 禁止"既然看到这里了""别忘了点赞关注"等模板化CTA |
| **不堆砌** | 一篇文章只用一种CTA类型，不要同时要求点赞+关注+转发+评论 |
| **有辨识度** | CTA风格要与文章整体风格一致，保持卡兹克的个人特色 |

### 17.3 CTA禁用清单

| 禁用CTA | 原因 | 替代 |
|---------|------|------|
| "既然看到这里了" | 说教语气 | 场景代入式 |
| "别忘了点赞关注" | 模板化 | 自然融入文章语境 |
| "如果觉得有帮助请..." | 过于客套 | 直接说"点个赞" |
| 表情符号 | 不符合卡兹克风格 | 纯文字 |
| "作者：卡兹克" | 文末不出现署名 | 场景代入式互动自然收尾 |
| "投稿或爆料" | 文末不出现投稿信息 | 同上 |

### 17.4 CTA与APAG的G阶段关系

CTA是APAG框架G阶段的具体实现。选择CTA类型时，参考APAG的A阶段钩子类型：

| A阶段钩子 | 推荐G阶段CTA |
|-----------|-------------|
| 反直觉事实 | 反问式（"你遇到过吗？"） |
| 个人故事 | 场景代入式（"如果你也..."） |
| 痛点共鸣 | 轻互动（"觉得有用就..."） |
| 好奇心缺口 | 好奇心驱动（"下一篇我会..."） |

**W1自检项**：
- [ ] 文章结尾有CTA
- [ ] CTA类型与文章类型匹配
- [ ] CTA不在禁用清单中

## §18 跨领域透镜融入指南（多平台适配新增）

> 跨领域透镜是卡兹克文章的差异化竞争力。将心理学、历史、推演模拟等领域的视角融入AI/技术主题，能让文章从"信息传递"升级为"认知升级"。本节规定透镜的使用方式、融入原则和禁止事项。

### 18.1 可用透镜列表

| 透镜 | 核心价值 | 典型融入方式 | 适用文章类型 |
|------|----------|-------------|-------------|
| **心理学** | 揭示行为背后的动机和认知偏差 | 用心理学概念解释用户行为、AI交互体验、技术采纳障碍 | 用户体验、产品设计、行为分析类文章 |
| **历史** | 提供纵深感，将当下放入更长的时间线 | 用历史事件类比技术变革，展示"历史不会重复但会押韵" | 趋势分析、行业演进、技术范式类文章 |
| **推演模拟** | 展示因果链和二阶效应，避免线性思维 | 用系统动力学思维推演"如果A发生，B会怎样，然后C会怎样" | 策略评估、决策分析、风险预判类文章 |

### 18.2 融入原则

| 原则 | 说明 | 正确示例 | 错误示例 |
|------|------|----------|----------|
| **自然融入** | 透镜应服务于文章主题，而非独立展示 | 在讨论AI工具依赖时自然引入心理学的"认知卸载"概念 | 突然插入一段心理学理论，与前后文无关 |
| **克制使用** | 每篇文章最多使用1-2个透镜，每个透镜最多出现2-3次 | 全文只在关键论点处使用心理学透镜 | 每段都引用不同领域的概念，变成"概念展览" |
| **读者友好** | 用日常场景解释透镜概念，不假设读者有领域知识 | "认知卸载，简单说就是把脑子里的东西外包给工具" | 直接使用"认知卸载"而不解释，假设读者懂心理学 |
| **服务主题** | 透镜的目的是让读者更好地理解文章主题，而非展示作者知识面 | 用历史类比让读者理解"为什么AI的这次变革和之前不一样" | 引用历史只是为了显得有学问，对理解主题没有帮助 |

### 18.3 融入方式

**方式一：类比融入**（最常用）
```
❌ 错误：认知卸载是心理学概念，指的是...
✅ 正确：心理学有个概念叫认知卸载，说白了就是把脑子记的东西外包出去。
         你用导航不就是认知卸载吗，不用记路了。
```

**方式二：解释融入**（用于分析深层原因）
```
❌ 错误：从心理学角度看，这是因为确认偏误。
✅ 正确：为什么我们总觉得自己是对的？心理学叫确认偏误——
         人会本能地寻找支持自己观点的证据，忽略反面信息。
```

**方式三：推演融入**（用于预测和策略）
```
❌ 错误：根据系统动力学，这会产生正反馈循环。
✅ 正确：你越依赖AI，越不会自己做决策；越不做决策，就越依赖AI。
         这不是简单的恶性循环，而是一个加速螺旋。
```

### 18.3.1 融入时机（Dan Koe评价经验补充）

> 不同透镜有最佳融入位置，放错位置会显得生硬。

| 透镜 | 最佳融入位置 | 原因 |
|------|-------------|------|
| 心理学 | "发现问题"阶段 | 解释"为什么有经验却不会表达"等深层原因 |
| 趋势预测/推演 | "方法论"或"系统介绍"阶段 | 展示前瞻性思考，强化方法论的价值 |
| 历史 | "对比"场景 | 用历史规律佐证当前观点 |

**心理学融入示例**（发现问题阶段）：
```
后来我学了点认知心理学才明白，这其实是一种"知识的诅咒"——
当你对一件事足够熟悉，你就很难想象不知道它是什么感觉。
```

**趋势预测融入示例**（方法论阶段）：
```
我甚至在脑子里推演过：如果AI继续发展，5年后内容创作的门槛
会降到什么程度？我的结论是——门槛会降到零，但"被看见"的
门槛会升到天际。
```

### 18.4 禁止事项

| 禁止行为 | 原因 | 替代做法 |
|----------|------|----------|
| **堆砌透镜** | 一篇文章使用3个以上透镜，变成"知识展览" | 每篇最多1-2个透镜 |
| **学术化表达** | 用论文口吻解释透镜概念，读者看不懂 | 用日常场景和类比解释 |
| **透镜独立成段** | 透镜应融入论述，不应成为独立的知识科普段落 | 将透镜嵌入观点论证中 |
| **假设读者有领域知识** | 直接使用专业术语而不解释 | 首次使用时用一句话解释 |
| **透镜与主题无关** | 为了"显得有深度"而强行引入不相关的透镜 | 只在透镜能帮助理解主题时才使用 |
| **同一透镜过度使用** | 同一个透镜在全文出现超过3次 | 控制频率，点到为止 |

### 18.5 与作者事实档案的关联

跨领域透镜的使用应与作者事实档案§5（能力画像）保持一致：
- 透镜的"掌握程度"决定了文章中可以使用的深度
- 心理学透镜：中等掌握程度，适合用日常场景解释，不适合深入理论分析
- 历史透镜：中等掌握程度，适合用广为人知的历史事件类比，不适合冷门历史考据
- 推演模拟透镜：中等掌握程度，适合用系统动力学的基本概念（反馈回路、延迟效应），不适合复杂系统建模

**W1自检项**：
- [ ] 文章中使用的透镜数量不超过2个
- [ ] 每个透镜出现次数不超过3次
- [ ] 透镜概念首次出现时有日常场景解释
- [ ] 透镜与文章主题有明确的逻辑关联
- [ ] 透镜使用深度与作者事实档案§5的掌握程度一致
