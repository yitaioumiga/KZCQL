## §1 标题规范

### 1.1 硬性要求

- **字数限制**：不超过 20 个字
- **避免**：过长、过于学术、堆砌关键词

### 1.2 好标题的特征

- 有画面感或悬念感
- 简洁有力，一个标题说清楚一件事
- 可以用反差感制造好奇（如：AI 越来越强，却越来越不靠谱）

### 1.3 标题案例

| 标题 | 字数 | 风格 |
|------|------|------|
| AI的三层架构，人的三次蜕变 | 14 | 突出核心比喻 |
| 人的成长，就是一场自我微调 | 12 | 简洁有力 |
| AI 越来越强，却越来越不靠谱 | 14 | 反差感制造好奇 |

## §2 语言风格

### 2.1 首选风格

- **深度思考/随笔**：语言流畅有感染力，像和朋友聊天，不像写论文
- **适合平台**：公众号、博客、知乎专栏

### 2.2 去 AI 味要点（重要）

AI 生成文章最容易暴露的问题是过度使用引号包裹概念，读起来像在写术语表。

**必须做的 4 项**：
1. 删除不必要的双引号和「」包裹：能用顿号或直接叙述替代的，就不要加引号
2. 加入个人语气词：像、吧、呢、其实、说实话、你会发现
3. 用具体场景替代抽象描述：用你第一次学开车替代技能习得的初期阶段
4. 避免机械排比：三段式结构可以保留，但每段的句式要有变化

**可以保留的 2 项**：
1. 英文专业术语首次出现时的标注（如 context rot）
2. 直接引语（如别人说的话）

### 2.3 加粗金句规则

文章中每隔一段放一句加粗居中的核心洞察，作为视觉锚点：

```markdown
**外部知识的注入，不等于内部能力的增长。**

**架构是外挂的操作系统，不是内生的本能。**

**真正的成长，是参数级的改变。**
```

规则：
- 每个核心章节末尾放一句
- 不要超过 5 句，否则失去冲击力
- 句式要短，最好在 15 字以内

### 2.4 khazix-writer Skill 写作规范（卡兹克风格）

当使用 khazix-writer Skill 撰写文章时，除了遵守上述通用规范外，还需遵循卡兹克的个人风格：

#### 2.4.1 核心风格定义

**「有见识的普通人在认真聊一件打动他的事。」**

这是整篇文章的灵魂定位。作者不是专家在说教，也不是小白在仰望，而是一个有真实经验、有独立思考的普通人在分享一件真正打动自己的事。

#### 2.4.2 绝对禁用词列表

以下词汇是 AI 味重灾区，文章中**严禁出现**：

| 禁用词 | 替代建议 |
|--------|----------|
| 说了算 | 删掉，直接陈述观点 |
| 说白了 | 删掉，直接说 |
| 意味着什么 / 这意味着 | 换成具体描述 |
| 本质上 | 删掉或换成「其实就是」 |
| 换句话说 | 删掉，直接换个说法 |
| 不可否认 | 删掉 |
| 综上所述 | 删掉，用口语化总结 |
| 总的来说 | 删掉或换成「反正我觉得」 |
| 首先...其次...最后 | 删掉编号，用转场词连接 |
| 值得注意的是 | 删掉 |
| 不难发现 | 删掉 |
| 让我们来看看 / 接下来让我们 | 换成「你猜怎么着」「有意思的是」 |
| 在当今...的时代 | 删掉，直接进入话题 |
| 随着...的发展 | 删掉，直接说现状 |
| **坦率的讲** | 允许使用，但**单篇不超过1次** |
| **老实说** | 删除或换成「其实」 |
| **说实话** | 删除或换成「其实」「其实吧」，且单篇不超过1次 |
| **说真的** | 允许使用，但**单篇不超过1次** |

**说明**：坦率的讲/说真的 AI 味相对较轻，允许单篇使用 1 次，用多了还是会暴露。老实说/说实话是 AI 写作的高频套话（「坦白地说」「honestly speaking」是英文 AI 写作的典型特征），严禁使用。如果需要表达个人判断，用「其实」「其实吧」更自然，单篇不超过 2 次。

#### 2.4.3 标点禁令

- **不使用冒号「：」**，用逗号代替
- **不使用破折号「——」**
- **不使用双引号**（""和""），用「」或直接不加

#### 2.4.4 推荐口语化词组

文章中至少使用 8-10 个以下口语化词组，分 5 类：

| 类别 | 推荐词组 |
|------|----------|
| **转场** | 其实吧、不过呢、怎么说呢、回到xxx这块、其实呢、说真的 |
| **判断** | 我有时候觉得、我一直觉得、不是说xxx不行而是说、其实 |
| **自嘲** | 其实我也不确定、我自己也还在摸索、愚钝如我 |
| **情绪** | 这种感觉太爽了、我当时就愣住了、太离谱了、你敢信？？？ |
| **拉近** | 很多朋友可能不知道、你如果关注这个领域的话 |

**注意**：「其实」「其实吧」是推荐词，但也要控制频率，全文不超过 2 次，用多了反而刻意。详见 §2.4.2 禁用词说明。

#### 2.4.5 结构要求

- **绝大多数文章不加小标题**，从头到尾一口气顺下来，靠口语化转场自然衔接
- **例外**：N条心得/方法类文章可用数字编号（1、2、3），但不是正式 markdown 标题
- **冲突解决**：如果我们的创作规范要求加小标题（长文章必须），则用**加粗行内标题**代替 `##` 标题

示例：
```markdown
## 传统写法（禁用）
### 一个被忽视的真相

**正确写法（使用加粗行内标题）**
**一个被忽视的真相。**
```

#### 2.4.6 四层自检体系

写完后必须执行以下四层检查：

**L1 硬性规则**（必须全部通过）：
- 禁用词扫描：全文搜索 §2.4.2 中的所有禁用词
- 禁用标点扫描：搜索冒号、破折号、双引号
- 结构套话扫描：搜索「在当今...时代」「随着...发展」
- 空泛工具名检查：确保工具名有具体版本或上下文

**L2 风格与一致性检查**：
- 开头检查：是否自然、有代入感、不生硬
- 节奏检查：长短句交替，不连续出现相同句式
- 口语化检查：是否使用了至少 8-10 个推荐词组
- 标点禁令二次确认：L1 扫描后再次人工确认
- **叙事一致性检查**：检查例子/场景是否与已建立的人物状态、资源、时间线一致
  - 资源一致性：例子中的资源/能力与已建立的状态是否匹配
  - 时间一致性：时间线是否前后一致
  - 能力一致性：人物能力/经验是否与已建立的背景匹配
  - **跨篇历史事实一致性**：如果是系列文章，检查本文引用的历史数据、分数、时间线是否与前序已发布文章一致。如发现不一致，必须在文章开头添加"审查体系变更"等说明性段落
- **作者事实一致性**：如涉及作者个人经历、资产数据、工作状态等，必须与《作者事实档案》保持一致。不确定的事实留空或标注"待确认"，不得虚构
- **段落逻辑连贯性**：检查每个段落内部的逻辑链条是否完整
  - 特别注意"结论→解释→操作"的衔接，不能从结论直接跳到操作
  - 如发现断链（如"这恰恰是定投初期该有的状态"后直接跳到"我每月投500块"），补充解释性句子（如"为啥？因为..."）
  - 常见断链模式：观点→操作（缺解释）、问题→方案（缺分析）、现象→结论（缺推导）
- **跨段落逻辑自洽性**（SG联合督查补丁P1）：检查相邻段落之间是否存在逻辑矛盾
  - 特别注意"目标"与"触发条件"之间的矛盾（如"目标70-80%"与"达到80%就降回50%"）
  - 检查方法：对每个涉及数字/比例/规则的段落，与前后段落交叉验证是否自洽
  - 常见矛盾模式：目标值=触发值、操作与目标方向相反、条件A的结论与条件B的前提冲突
- **AI高频对比句式检查**（SG联合督查补丁P2）：扫描"不是...而是..."句式
  - 全文"不是...而是..."出现次数不得超过2次
  - 超过2次必须改写，用其他方式表达对比（如直接陈述、用"其实""说白了"等口语化转折）
  - 改写示例："定投最难的不是策略，而是在漫长熊市里保持心态稳定" → "定投最难的是在漫长熊市里保持心态稳定，策略反而是次要的"

**L3 内容质量**：
- 观点支撑：每个核心观点是否有案例或逻辑支撑
- 知识输出方式：是否用读者能理解的方式传递知识
- 文化升维：是否在技术内容之外有更深的思考
- 对立面与同理心：是否考虑了不同读者的感受

**L4 活人感终审**：
- 温度感：读起来像不像一个真人在说话
- 独特性：去掉署名，读者能不能认出是你的文章
- 姿态检查：是否在说教、是否在装懂
- 心流检查：从头到尾读一遍，是否顺畅、有没有卡壳的地方

#### 2.4.7 文章原型判断

写作前先判断文章属于哪种原型，不同原型有不同的核心写法：

| 原型 | 核心写法 | 关键词 |
|------|----------|--------|
| **调查实验型** | 亲自下场做一件事，核心是「我替你去做了」 | 我试了、我跑了、我测了 |
| **产品体验型** | 拿到产品实际使用，核心是「跟我一起玩」 | 我用了、我发现、这个功能 |
| **现象解读型** | 观察现象层层分析，核心是「你注意到了吗？背后是什么？」 | 你有没有发现、背后的逻辑 |
| **工具分享型** | 分享实用工具，核心是「我发现了一个好东西」 | 推荐一个、太好用了、效率翻倍 |
| **方法论分享型** | 系统分享经验，核心是「我把压箱底的东西掏给你了」 | 我总结了、踩了好多坑、终于想明白 |

### 2.6 排版与分段规范（重要）

**核心原则：给读者喘息的空间**

#### 2.6.1 转折短语必须独立成段

以下转折/过渡短语必须单独成行，形成停顿感：

| 类型 | 短语示例 | 正确排版 |
|------|----------|----------|
| 认知转折 | 没那么简单、其实不然、真相是 | 前一段结尾。/ **没那么简单。** / 下一段开始 |
| 时间转折 | 后来发现、再后来、直到那天 | 独立成段或置于段首 |
| 口语化过渡 | 怎么说呢、说实话、坦率的讲 | 独立成段或置于段首 |
| 场景切换 | 举个例子、回到那天、以这个系列为例 | 独立成段 |
| 设问引出 | 这条规则是怎么来的？为什么是 30%？ | 独立成段，制造悬念 |

**错误示例**：
```
很多人可能觉得，事实核查嘛，就是检查一下文章里有没有写错的东西。没那么简单。R1 手里有 19 条一票否决规则...
```

**正确示例**：
```
很多人可能觉得，事实核查嘛，就是检查一下文章里有没有写错的东西。

没那么简单。

R1 手里有 19 条一票否决规则...
```

#### 2.6.2 专业术语密集处必须分段

当一段中出现 3 个及以上专业术语或概念时，必须拆分：

**触发分段的信号**：
- 连续出现多个编号（V-01、V-02、V-03）
- 连续列举多个维度（D1、D2、D3...）
- 连续解释多个概念（同情分偏差、进步分偏差、完美主义偏差）
- 连续罗列多个等级（S级、A级、B级、C级）

**分段策略**：
1. 每个术语/概念独立解释 → 每解释完一个就分段
2. 或每 2 个一组 → 形成节奏感
3. 总结句单独成段 → 让读者有时间消化

**错误示例**：
```
R2 评分的时候有一个硬性要求，每个维度的扣分必须引用原文。不能只说「标题不够好」就扣分，得指出具体哪句话、哪个表达不够好，为什么不够好。这个要求是为了防止评审 Agent 凭主观印象打分。评分完之后，R2 还要做一件事，评审偏差自检。它要检查自己有没有犯四类常见偏差。同情分偏差，因为文章题材好就多给分。进步分偏差...
```

**正确示例**：
```
R2 评分的时候有一个硬性要求，每个维度的扣分必须引用原文。不能只说「标题不够好」就扣分，得指出具体哪句话、哪个表达不够好，为什么不够好。

这个要求是为了防止评审 Agent 凭主观印象打分。

评分完之后，R2 还要做一件事，评审偏差自检。它要检查自己有没有犯四类常见偏差：

**同情分偏差**，因为文章题材好就多给分。

**进步分偏差**，因为比上一版好就多给分。

**完美主义偏差**，对非核心问题过度扣分。

**忽略 AI 感偏差**，对 AI 味问题不够敏感。
```

#### 2.6.3 金句/加粗标题前必须有停顿

金句或行内加粗标题（如 **评分不是目的，路由才是。**）前，必须有一个空行或短段落形成停顿，不能紧跟在长段落后。

**错误示例**：
```
...底层的分数在进步。**评分不是目的，路由才是。**
```

**正确示例**：
```
...底层的分数在进步。

**评分不是目的，路由才是。**
```

#### 2.6.4 章节切换必须有空行

行内加粗标题（如 **R1，第一道门**）前后必须各有一个空行，与正文形成视觉区隔。

## §3 文章结构

### 3.1 必须有小标题

长文章（超过 30 行）必须有章节小标题，否则就是一堵文字墙，读者会疲劳。

小标题规范：
- 用 `##` 级别，不用 `#`（`#` 留给大标题）
- 每个章节 30-50 行为宜
- 小标题本身可以有信息量，不只是标签（如：Superpowers：约束即自由）

**注意**：如果使用 khazix-writer Skill 写作，Skill 要求不加小标题。此时用加粗行内标题（如 **一个被忽视的真相。**）代替 `##` 标题，既满足 Skill 的风格要求，又保证长文章的可读性。

### 3.2 推荐结构模板

```
# 大标题（不超过20字）

开头（引子）
  └─ 提及之前的相关内容（如有），自然过渡到本文主题

## 小标题1：引入/问题

## 小标题2：核心内容A

## 小标题3：核心内容B

## 小标题4：落地/实践

## 小标题5：转折/意外发现（如有）

## 小标题6：升华/行业思考

## 小标题7：总结

结尾（互动引导）
  └─ 场景代入式提醒点赞/转发/关注
```

### 3.3 开头写法

如果之前发过相关内容，新文章开头应自然提及：
- 简述之前文章的核心观点（1-2句话）
- 说明本文是补充、深入还是另一个角度
- 不要大段复述，点到为止

示例：
> 昨天发了一条动态，大意是说人的成长和 AI 的架构演进几乎一模一样。发完之后觉得意犹未尽，很多想法还没展开，所以今天想把这个比喻彻底写透。

### 3.4 结尾写法

结尾需要提醒读者互动，但不要像模板一样生硬。

**我们的风格**：场景代入 + 轻松自然

示例：
> 如果你也每天在用 AI 干活，这篇文章应该能帮到你。点个赞、转发给同事，他们可能正好需要。关注我，后面还会分享更多 AI Agent 的实战经验。
>
> 下次聊。

**避免**：
- 不要用「既然看到这里了」这种说教语气
- 不要用表情符号
- 不要跟别人的结尾格式一样，要有自己的辨识度
- **文末不出现作者署名和投稿邮箱**（如「/ 作者：卡兹克」「/ 投稿或爆料」），以场景代入式互动自然收尾即可

### 3.5 案例与结论的因果匹配（重要）

当文章用具体案例来解释某条规则或某个阈值时，**案例必须能直接推导出结论**，不能只渲染氛围。

#### 常见错误模式

| 错误类型 | 说明 | 示例 |
|----------|------|------|
| **量级断裂** | 案例的数据量级与结论的量级差距过大，无法建立因果关系 | 用"修改幅度 10%"的案例推导"为什么阈值是 30%"——10% 离 30% 差三倍，读者无法理解为什么不是 20% 或 50% |
| **方向偏移** | 案例说明的是 A 问题，结论却落在 B 问题上 | 案例说明"修改幅度需要控制"，结论却是"为什么恰好是 30%" |
| **氛围替代论证** | 用"惊险""差一点"等情绪词代替逻辑推导 | "已经相当惊险了"——惊险不等于论证，读者需要的是逻辑链而非情绪 |

#### 正确写法

**原则**：案例 → 分析 → 结论，三步缺一不可。

1. **案例**：呈现真实数据或事件
2. **分析**：从案例中提取规律或趋势
3. **结论**：基于分析给出具体数字或规则

**正确示例**：

> 为什么是 30%？
>
> 第 2 篇文章的 W2 改了 39 处标点，总修改幅度 10%。第 3 篇的 W2 重写了结尾，幅度 25%。第 4 篇更极端，W2 改了开头和中间两个章节，幅度直接飙到 35%。
>
> 前两篇 R3 差异评审还能应付，但第 4 篇 R3 明显力不从心——修改太多，差异评审根本看不过来，漏掉了 W2 引入的一个新问题。
>
> 所以 30% 是一条经验线：低于这个比例，R3 的差异评审足够覆盖；超过这个比例，就必须退回 R2 全量评审。

**关键区别**：正确写法用了**多个案例形成趋势**（10% → 25% → 35%），让读者看到"为什么 30% 是分界线"——因为 35% 时 R3 确实出了问题。而错误写法只有一个 10% 的案例，无法支撑 30% 的结论。

#### 自检方法

写完"这条规则是怎么来的？"这类段落时，问自己：
- 这个案例的数据/事实，能不能**直接推导**出我要说的结论？
- 如果把结论中的具体数字换掉（比如把 30% 换成 20%），这个案例还成立吗？
- 如果成立，说明案例太弱，需要补充更有针对性的证据。

### 3.6 读者视角自检（重要）

> **事故关联**：第6篇v1-v4迭代中发现的"差了两档""关注我们这个系列""归档遗漏缺背景"等问题

写作时必须假设读者**没有**阅读过系列其他文章，**没有**系统内部知识。

#### 3.6.1 自检三问

在文章定稿前，必须回答以下三个问题：

1. **这个表述是否假设了读者知道某件"显而易见"的事？**
   - 错误示例："你如果关注我们这个系列的文章的话..."
   - 正确示例："你如果关注这个领域的话..." 或具体说明背景

2. **专业术语/内部概念第一次出现时是否有解释？**
   - 错误示例：直接说"归档遗漏的主要原因是执行缺陷"
   - 正确示例：先说"什么是归档遗漏"，再分析原因

3. **案例引用是否有足够的背景让读者理解？**
   - 错误示例：直接引用"第4篇的R2评分68分"
   - 正确示例：先说明"第4篇是什么主题、什么场景下的评分"

#### 3.6.2 常见错误模式

| 错误模式 | 错误示例 | 正确写法 |
|----------|----------|----------|
| 系列自指 | "我们这个系列""前几篇" | "这个领域""前文提到"+具体说明 |
| 内部术语裸用 | "归档遗漏""配图遗漏" | 首次出现时解释术语含义 |
| 事实数据不核实 | "差了两档"（实际只差一档） | 核实评级区间后再写 |
| 假设读者已知 | "众所周知""显然" | 提供必要的背景信息 |

#### 3.6.3 验证方法

- 让没有阅读过系列其他文章的人阅读，检查是否有理解障碍
- 检查所有数字、评级、区间描述是否准确
- 检查专业术语首次出现时是否有括号解释或前文铺垫

### 3.7 术语友好原则（重要）

> **核心判断标准**：如果读者不理解某个术语，是否会严重影响对文章主旨的理解？如果是，必须解释。

#### 3.7.1 术语三级分类

| 层级 | 定义 | 处理方式 | 判断标准 |
|------|------|----------|----------|
| **核心术语** | 文章论点依赖它，不懂就完全看不懂 | 叙事融入式解释 | 删除该术语后，文章主旨是否还能被理解？ |
| **辅助术语** | 出现多次，不懂会有阅读摩擦 | 括号注释，一句话带过 | 该术语是否出现≥2次且非通用常识？ |
| **装饰术语** | 只出现一次，不影响主旨理解 | 不解释 | 目标读者群体是否大概率已知道？ |

#### 3.7.2 解释方式

**叙事融入式**（核心术语）：
```
我一开始想走的是价值投资这条路——简单说就是研究公司的财务报表，
算出这只股票"应该值多少钱"，低于就买入，高于就卖出。
```
- 优点：不打断阅读节奏，读者在故事中自然理解
- 适用：定投、价值投资、一票否决项等文章骨架级概念

**括号注释式**（辅助术语）：
```
我选择了沪深300（跟踪中国最大的300家上市公司的指数基金）作为核心持仓。
```
- 优点：简洁高效，不占篇幅
- 适用：沪深300、宽基指数、ETF联接等出现多次但非核心的术语

#### 3.7.3 分受众策略

术语分类和解释深度因目标读者群体而异：

| 维度 | 零基础读者（如理财小白） | 有经验读者（如AI编程从业者） |
|------|--------------------------|---------------------------|
| 核心术语判断 | 不懂就完全看不懂 | 不懂会影响对体系/流程的理解 |
| 解释深度 | 用大白话讲清楚概念本身 | 假设读者有领域基础，只解释体系特有概念 |
| 解释方式 | 叙事融入 + 括号注释 | 括号注释 + 引用文档路径 |
| 典型需解释 | 定投、沪深300、ETF | D5并行拆解、一票否决项、盲评风险 |
| 典型不需解释 | A股、股票 | Agent、Prompt、API |

#### 3.7.4 W1 自检项

- [ ] 文章中所有核心术语首次出现时已融入叙事解释
- [ ] 辅助术语首次出现时有括号注释
- [ ] 术语解释深度与目标读者群体匹配
- [ ] 不存在"裸用"的核心术语（即出现但无任何解释）

## §4 配图规范 - 文艺插图

### 4.1 与架构图的区别

文章中的图片分两类，规范完全不同：

| 类型 | 用途 | 是否允许文字 | 渲染方式 | 示例 |
|------|------|-------------|----------|------|
| **文艺插图** | 章节配图、封面、氛围营造 | **严禁任何文字** | GenerateImage | 手绘风人物、场景 |
| **架构流程图** | 解释技术架构、流程、系统设计 | **必须有清晰文字** | Playwright HTML渲染 | 四层架构图、审查流程图 |

### 4.2 风格偏好

- **首选风格**：文艺手绘风（水彩/素描质感，柔和色调）
- **备选风格**：科技扁平风、3D现代风
- **色调要求**：柔和、低饱和度、暖色调为主（米白、灰蓝、鹅黄、淡绿、珊瑚色）
- **整体感觉**：像 Moleskine 手账页，温暖、有呼吸感

### 4.3 硬性要求

- **禁止出现任何文字**：包括中英文、字母、数字、logo、水印
- **图片比例**：16:9 横版（landscape_16_9），适合公众号排版
- **构图原则**：留白充足，主体居中或偏移三分之一处

### 4.4 数量与位置

| 位置 | 数量 | 说明 |
|------|------|------|
| 封面头图 | 1张 | 标题下方，读者打开文章第一眼看到 |
| 核心章节 | 每章1张 | 放在章节小标题下方，作为视觉分隔 |
| 结尾/过渡章节 | 不配图 | 保持文字节奏的留白 |

**经验**：一篇文章 4-5 张文艺插图最合适（1封面 + 3-4核心章节），过多打断阅读节奏，过少显得单调。

### 4.5 防文字声明（强制追加）

生成配图时，在 prompt 末尾**必须**追加防文字声明：

```
Completely clean image with zero text, zero words, zero letters, zero characters, zero logos, zero watermarks, zero stamps, absolutely no written content of any kind anywhere.
```

### 4.6 水印处理

**强制执行**：每张图片生成后，**必须**立即执行以下水印处理流程。跳过此步骤视为生产事故。

图片生成工具可能会自动添加水印（如 AI生成、KIMI 等），通常在右下角。

**去除方法**：用 Pillow 裁掉右下角

```python
from PIL import Image

img = Image.open("image.jpg")
w, h = img.size

# 裁掉右下角（比例根据水印大小调整，通常 4%-6%）
cropped = img.crop((0, 0, int(w * 0.95), int(h * 0.95)))
cropped.save("image.jpg", quality=95)
```

**注意**：
- 裁剪后必须用 Read 工具检查图片，确认水印已完全去除
- 不同工具的水印大小不同，KIMI 水印需要裁 6%，其他通常 4-5% 即可
- 如果裁一次没去干净，再裁一次

### 4.7 各章节画面建议

| 章节类型 | 画面建议 |
|----------|----------|
| 概念引入/问题提出 | 人物困惑、被信息包围、思考状 |
| 方法论/框架介绍 | 结构化场景（关卡、拱门、流程图） |
| 转折/意外发现 | 挫败感（纸贴不上墙、东西散落） |
| 升华/结论 | 自由、开阔场景（山顶、日出、温室中的安全感） |

### 4.8 图片格式规范（P36补丁新增）

> **事故关联**: INC-2026-0520-图片格式（占位符格式导致交付不合格）

#### 4.8.1 标准格式要求

文章中的图片引用**必须**使用标准 Markdown 格式：

```markdown
![图片描述](images/XX/XX_编号_名称.jpg)
```

**严禁使用占位符格式**：
```markdown
❌ 图片：{封面——一个年轻人坐在书桌前...}
❌ {图片：章节插图——三个台阶...}
❌ 图片占位符：{...}
```

#### 4.8.2 格式转换责任

| 阶段 | 责任Agent | 转换动作 |
|------|-----------|----------|
| **W1初稿撰写** | W1 | 输出时必须为标准Markdown格式，不得输出占位符 |
| **ImageCheck** | 主Agent | 检查并转换任何占位符格式为标准格式 |
| **交付前** | 主Agent | 执行IMG-01b检查，确认无占位符残留 |

#### 4.8.3 转换方法

如文章中存在占位符格式，必须立即转换：

```bash
# 使用sed批量转换（示例）
sed -i 's/图片：{封面——\([^}]*\)}/![封面](images\/XX\/XX_01_封面.jpg)/g' {文件}.md
sed -i 's/图片：{章节插图——\([^}]*\)}/![章节插图](images\/XX\/XX_02_插图.jpg)/g' {文件}.md
```

**注意**：转换后必须使用 Grep 验证：
```bash
grep -c '图片：{' {文件}.md  # 必须为0
grep -c '!\[.*\](.*)' {文件}.md  # 必须≥4
```

#### 4.8.4 违规后果

- 占位符格式进入交付流程 → **视为生产事故**
- 触发联合督查 → 主Agent承担严重责任

## §5 架构图/流程图规范

### 5.1 适用场景

当文章涉及技术架构、系统设计、工作流程、层次结构等内容时，需要配架构图或流程图。与文艺插图不同，架构图**必须有清晰的中文文字标注**。

典型场景：
- AI Agent 的多层架构（如 SOLO 四层纪律系统）
- 审查/质量门流程（如两阶段审查）
- 多组件协作关系（如薄编排器 + 子代理）
- 决策流程/检查清单（如统一质量门五关）

### 5.2 渲染方案优先级

**强制规则**：架构图**必须**使用 wkhtmltoimage 或 Playwright HTML 渲染。**严禁**使用 GenerateImage 渲染包含中文文字的架构图。GenerateImage 对中文文字渲染不可控，会产生乱码和伪文。

按可用性排序：

1. **wkhtmltoimage**（首选，最稳定）
   - 系统自带，无需下载额外依赖
   - 命令：`wkhtmltoimage --quality 100 --width 1920 --height 1080 input.html output.png`
   - 缺点：不支持 deviceScaleFactor，输出分辨率为设定值（通常 1920x1080）

2. **Playwright HTML -> PNG**（高清备选）
   - 需要 Chromium，但 CDN 经常在沙箱中被阻止
   - 安装：`npx playwright install chromium --with-deps`
   - 优点：支持 deviceScaleFactor: 3，输出 5760x3240 高清图
   - 缺点：依赖下载可能超时

3. **GenerateImage**（**禁止**用于包含中文文字的架构图。仅可用于纯图形、无文字的简单示意图。）
   - 对中文文字渲染不可控，经常乱码
   - 仅适用于不需要文字的文艺插图

**为什么不用 Remotion**：需要从 CDN 下载依赖，沙箱环境中 CDN 被阻止。

**降级路径**：Playwright 不可用 -> wkhtmltoimage -> 手动调整 HTML 布局补偿分辨率

### 5.3 渲染配置

```javascript
const { chromium } = require('playwright');

const page = await browser.newPage({
  viewport: { width: 1920, height: 1080 },
  deviceScaleFactor: 3,  // 关键：3x 缩放确保文字清晰
});

await page.screenshot({
  path: outputPath,
  fullPage: false,
  type: 'png',  // 必须用 PNG，不用 JPG
});
```

**关键参数**：
- `viewport: 1920x1080`：设计基准分辨率
- `deviceScaleFactor: 3`：3x Retina 渲染，输出 5760x3240
- `type: 'png'`：无损格式，比 JPG 更清晰
- 字体最小 17px，标题 22-24px，确保放大后仍然清晰

### 5.4 配色方案

使用暖色调专业配色，避免冷硬的科技蓝：

```javascript
const COLORS = {
  bg: '#FDFBF7',          // 暖米白背景
  primary: '#4A6FA5',     // 灰蓝（主色）
  secondary: '#5B9A8B',   // 青绿（辅色）
  accent: '#E8927C',      // 珊瑚色（强调）
  amber: '#D4A574',       // 琥珀色
  lavender: '#8B7EC8',    // 薰衣草紫
  text: '#2D3436',        // 深灰文字
  textLight: '#636E72',   // 浅灰文字
  green: '#27AE60',       // 成功绿
  red: '#E74C3C',         // 警告红
  shadow: 'rgba(0,0,0,0.08)',
  border: '#E0D8CF',
};
```

### 5.5 字体配置

```css
font-family: -apple-system, "PingFang SC", "Microsoft YaHei",
             "Noto Sans CJK SC", "Helvetica Neue", sans-serif;
-webkit-font-smoothing: antialiased;
-moz-osx-font-smoothing: grayscale;
text-rendering: optimizeLegibility;
```

### 5.6 设计原则

- **圆角矩形**为主（border-radius: 14-18px），避免尖锐边角
- **柔和阴影**（box-shadow: 0 6px 24px），增加层次感
- **充足留白**，元素之间 gap 至少 16-24px
- **箭头连接**用 -> 或 CSS 绘制，不用图片
- **编号圆圈**标识步骤顺序
- **颜色编码**区分不同层级/阶段

### 5.7 文件命名

- 前缀 `arch-` 区分于文艺插图
- 格式用 `.png`（不用 `.jpg`）
- 示例：`arch-superpowers.png`、`arch-gsd.png`、`arch-solo.png`、`arch-qualitygate.png`

### 5.8 插入位置原则

架构图应紧跟在对应的文字说明之后，让读者先理解概念再看图：

| 文章内容 | 架构图 |
|----------|--------|
| 介绍完两阶段审查机制后 | 两阶段审查流程图 |
| 介绍完薄编排器 + 子代理模式后 | 薄编排器架构图 |
| 介绍完四层纪律系统后 | 四层架构图 |
| 介绍完质量门机制后 | 质量门流程图 |

**原则**：先文字解释，再放图巩固。不要在读者还没理解概念时就放图。

## §7 技术规范

### 7.1 输出格式

- **必须同时输出**：.md（Markdown）和 .docx（Word）两种格式
- **.md 用途**：直接复制粘贴到公众号编辑器，或作为存档
- **.docx 用途**：正式存档、分享给他人

### 7.2 docx 生成要点

使用 docx-js 生成 Word 文档时，关键配置：

```javascript
// CJK 字体（中文必须配置，否则乱码）
font: { ascii: "Arial", hAnsi: "Arial", eastAsia: "Microsoft YaHei" }

// Linux 环境回退字体
eastAsia: "Noto Sans CJK SC"

// 单 section，所有内容放在一个 children 数组中
// 不生成目录
// 正文 1.5 倍行距（line: 360）
// A4 页面，1 英寸边距
```

生成后必须运行 sanitize.py 清理：

```bash
python3 /path/to/sanitize.py output.docx
```

### 7.3 段落长度规范（P38补丁新增，P38.1修订）

> **关联事故**：理财访谈第5篇v4段落长度分析（52.7%段落>80字，严重偏离行业标准）
> **修订说明**：P38.1基于Adobe/Google/Cambridge/Purdue OWL等权威写作指南调研，从"一刀切≤80字"修订为"长短交替+核心原则"

移动端阅读（98.7%用户）对段落长度高度敏感。但段落长度不是越短越好——**长短交替创造节奏**（Adobe、Cambridge、Purdue OWL一致观点），**一个段落表达一个完整想法**（Google开发者文档明确要求）。

**核心原则**：长短交替，一个段落一个想法。

**段落长度标准**：

| 类型 | 长度 | 占比 | 用途 |
|------|------|------|------|
| 短段落 | 10-40字 | 40-50% | 强调、转折、对话感、金句 |
| 中等段落 | 40-80字 | 30-40% | 正常叙述、信息传递 |
| 长段落 | 80-120字 | 10-20% | 展开论证、复杂概念解释 |
| 超长段落 | >120字 | **<5%** | 仅用于不可拆分的完整论述 |

**硬性红线**：

| 规则 | 标准 | 说明 |
|------|------|------|
| 超长段落上限 | **≤120字** | 超过必须拆分（无例外） |
| 段落间留白 | **空一行** | 创造"呼吸感" |
| 超短段落下限 | **≥10字** | 避免过度碎片化 |
| 单一想法原则 | **一段一意** | 不要为了短而拆散一个完整想法 |

**拆分策略**（按优先级）：
1. 在逻辑转折点拆分（"但是"、"所以"、"然而"前）
2. 在举例/说明前拆分（"比如"、"例如"前）
3. 在问句结束后单独成段
4. 在结论/总结前拆分

**保留长段落的场景**（允许80-120字）：
- 解释一个需要上下文的复杂概念
- 讲述一个有起承转合的完整故事/案例
- 论证一个需要多层推理的观点
- 上述场景下，如果拆分会破坏逻辑完整性，则**不拆**

**禁止事项**：
- ❌ 在句子中间强行拆分（必须在句号/问号/感叹号处）
- ❌ 拆分时修改标点符号（逗号改句号等）
- ❌ 拆分散加粗金句或图片引用
- ❌ 为了追求"全短段落"而拆散一个完整想法（Google明确反对）
- ❌ 所有段落长度一致（Cambridge：长度一致会让读者犯困）

**自检方法**：
1. 统计>120字的段落数量，应为0
2. 统计80-120字的段落数量，应在10-20%
3. 检查是否存在连续5个以上长度相近的段落（节奏单调）
4. 检查每个长段落是否表达一个不可拆分的完整想法

### 7.4 文件命名规范

- 标题即文件名，简洁明了
- 示例：`AI越来越强，却越来越不靠谱.md`
- 配图统一放在 `images/` 目录下
- 文艺插图命名：`cover.jpg`、`layer1.jpg`、`solo2-cover.jpg`（前缀区分文章）
- 架构图命名：`arch-superpowers.png`、`arch-gsd.png`（arch- 前缀）

## §8 写作前必问清单

每次撰写新文章前，必须向用户确认以下 6 项：

| 序号 | 必问项 | 具体问题 |
|------|--------|----------|
| 1 | **案例方向** | 你有没有亲身经历的相关场景想放进去？有没有什么让你特别兴奋或者特别想吐槽的地方？ |
| 2 | **开头偏好** | 这一期你希望用什么方式开头？（叙事启动 / 荒诞事实 / 好奇心驱动 / 热点破题） |
| 3 | **时间线** | 文章中的事件发生在什么时间？（避免多篇文章用相同时间表达） |
| 4 | **身份确认** | 这一期你的身份设定有没有变化？（确保与用户身份档案一致） |
| 5 | **目标读者** | 这篇文章的目标读者是谁？（技术人员 / 产品经理 / 完全非技术的人）——决定技术门槛高度 |
| 6 | **交付物确认**（P35补丁新增） | 本次交付是否需要全部格式（md + docx + 配图）？还是仅需部分格式？默认全部输出。 |

## §9 用户身份档案

> 持续更新。每次写作前必须读取确认。

| 维度 | 当前值 | 备注 |
|------|--------|------|

## §10 经验教训

### 10.1 图片生成

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 文艺插图出现文字 | GenerateImage 有时忽略防文字声明 | 多次重试，加强 prompt 中的禁止文字描述 |
| 水印去不掉 | 不同工具水印位置和大小不同 | 先 Read 检查，再决定裁剪比例（4%-6%） |
| 架构图文字乱码 | GenerateImage 对中文渲染不可控 | 改用 Playwright HTML 渲染 |
| 架构图文字模糊 | viewport 太小 + deviceScaleFactor 不够 | viewport 1920x1080 + deviceScaleFactor 3 |
| Remotion 无法使用 | CDN 在沙箱中被阻止 | 降级为 Playwright 方案 |
| Playwright Chromium 下载超时 | CDN 在沙箱中被阻止 | 用 wkhtmltoimage 替代，或用 `apt-get install wkhtmltopdf` 安装 |

### 10.2 文章撰写

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| AI 味太重 | 过度使用引号包裹概念 | 删除不必要的引号，加入个人语气词 |
| AI 味太重 | 使用「坦率的讲/老实说/说实话/说真的」等高频套话 | 这些是禁用词，换成「其实」「其实吧」，单篇不超过1次 |
| 标题过长 | 想表达太多信息 | 控制在 20 字内，一个标题说一件事 |
| 没有小标题 | 长文章缺少结构 | 每 30-50 行加一个 ## 小标题 |
| 结尾像别人 | 参考了其他作者的结尾风格 | 用场景代入 + 轻松自然的方式，保持自己的辨识度 |

### 10.3 事实核查

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| GitHub 项目内容与 README 不符 | README 可能过时或夸大 | 通过 GitHub API 确认实际仓库内容 |
| 作者信息错误 | 张冠李戴或拼写错误 | 通过 commits/贡献者列表确认 |
| 版本号过时 | 模型迭代快 | 搜索确认最新版本号 |

### 10.4 khazix-writer 使用

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 文章 AI 味重 | 未执行四层自检 | 写完后必须跑 L1-L4 全部检查 |
| 小标题冲突 | Skill 要求不加标题，我们的规范要求长文章必须有 | 用加粗行内标题代替 ## 标题 |
| 结尾格式冲突 | Skill 有固定尾部格式 | 改为我们自己的场景代入式结尾 |
| 禁用词遗漏 | 修改过程中重新引入 | L1 扫描 + L2 二次确认 |

### 10.5 内容准确性

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 失忆案例选错工具 | Claude Code 在记忆处理上做得很好，不适合做「失忆」反面案例 | 失忆/上下文丢失相关案例优先用 Cursor、Copilot 等工具 |
| 模型数据过时 | 技术迭代快，上下文窗口等参数经常变化 | 每次写作前确认最新数据（Claude Sonnet 4.6 ~1M、GPT-4.1 ~1M、Gemini 2.5 Pro ~1M） |
| 工具能力描述不准 | 不同工具的自主程度、记忆能力差异大 | 写之前确认工具当前实际能力，避免凭印象描述 |
| Copilot 能力被低估 | Copilot 的自主执行、规划能力与 Claude Code、Cursor 相当 | 描述 Copilot 时不要仅定位为「代码补全」，它也能自主执行任务和做规划 |

### 10.6 系列文章写作

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 架构图包含后文内容 | 图里放了还没讲到的概念，读者看不懂 | 架构图放在对应内容全部讲完之后，作为总结全景图，不要提前剧透 |
| 系列文章之间缺乏衔接 | 每期独立写，没有引用前文 | 每期开头自然引用前文概念（「上期我们聊了xxx」「还记得那个xxx的类比吗」），但不要大段复述 |
| 插入位置只给行号 | 发布平台看不到行号，用户找不到位置 | 给出精确插入位置时，必须附上完整的上下文段落，让用户可以直接在发布平台上搜索定位 |
| 用户身份设定不一致 | 多篇文章中身份描述矛盾（如一会商学院学生一会AI实践者） | 在系列计划文件中明确用户身份设定，每次写作前读取确认 |
| 用户反馈的经验未沉淀 | 修改了文章但没更新规则文件，下次还会犯同样错误 | 每次用户纠正后，必须同步更新规则文件的经验教训章节 |
| **系列标识格式错误** | **文末系列标识用了文章标题而非系列名** | **系列标识格式为 `> *系列名 · 第X篇/共Y篇*`，系列名必须与系列计划文件中的系列名一致，不得使用文章标题** |

### 10.7 用户身份与真实感

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 故作惊讶 | 对已知事物表现出「太离谱了」「我当时就愣住了」，不符合用户实际经验水平 | 根据用户真实的技术水平调整反应。用户有丰富经验时，用「亲手验证了理论，感受不同」代替「震惊」 |
| 经历编造 | AI 编造了用户没有的经历（如「翻资料发现系统论」） | 写之前跟用户确认经历的真实性。用户提到「我之前就学过」「我自己自学过」时，必须尊重事实 |
| 身份描述不统一 | 同一用户在不同文章中身份不同 | 维护一份用户身份档案（学生/职业/工具使用经验/自学过的知识），写作前读取 |

### 10.8 工作流与协作

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 修改指令理解不完整 | 用户说「改这里」但没给完整上下文，改错了位置 | 先用 Grep 确认要修改的段落，展示给用户确认后再改 |
| 规则文件更新遗漏 | 改了文章但忘了同步更新规则文件 | 建立「改文章 -> 问自己是否需要沉淀规则 -> 更新规则文件」的标准流程 |
| 系列计划未同步更新 | 写了新文章但系列计划里的状态没更新 | 每完成一期，同步更新系列计划文件中的状态和文件路径 |

### 10.9 案例选择与用户参与

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 案例太技术化 | TypeScript 泛型推断等案例对非技术读者不友好 | 案例选择优先考虑读者的可理解性。用日常场景（AI 不用已配置工具而自己写脚本）代替纯技术概念（泛型推断） |
| 案例与工具语境矛盾 | 在 Cursor 里写备注说「Claude Code 怎么样」，逻辑不通 | 案例中的工具引用必须与语境一致，在哪个工具里操作就只提那个工具 |
| 案例未征求用户意见 | AI 自选案例，可能不符合用户真实经历 | **每次写文章前，主动向用户询问偏好的案例方向，或让用户自己举例** |
| 用户有更好的案例 | 用户对自己用 AI 的经历比 AI 更清楚 | 写作前必须问用户「有没有什么自己的经历想放进去？有没有什么让你特别兴奋或者特别想吐槽的地方？」 |

### 10.10 架构图设计

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 架构图信息过载 | 一张图塞了太多内容（四级跃迁+三个操作+确认节点），元素重叠 | 每张架构图只讲一个核心概念，复杂内容拆分为多张图 |
| 横排布局文字挤压 | 四列横排卡片太窄，中文字符重叠 | 中文内容优先使用 2x2 网格布局，给每个卡片足够宽度 |
| 架构图渲染依赖丢失 | weasyprint 在沙箱中经常被清理 | 每次渲染前先检查 weasyprint 是否可用，不可用则重新安装 |

### 10.11 用户身份档案

> 详见 §9，此处为经验补充。

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 案例不符合用户身份 | 写了「数据处理模块」「空数组边界条件」「报警电话」等传统软件开发场景 | 案例必须符合用户身份。AI 编程实践者的日常场景是项目结构、工具使用、配置规则，不是生产环境部署、报警电话、复杂数据处理 |
| 案例太技术化 | 用 TypeScript 泛型推断作为贯穿案例，非技术读者看不懂 | 案例优先用日常可理解的场景（如 AI 不用已配置工具），如果必须用技术案例，选择更通用的（如项目结构） |
| 场景夸张不真实 | 「凌晨两点报警电话」「上线第三天炸了整个链路」——学生项目不会有报警电话 | 场景必须真实可信。学生项目的「出问题」是舍友用了发现不对、自己回头看不合理，不是报警电话和生产环境崩溃 |
| 案例与工具语境矛盾 | 在 Cursor 里写备注说「Claude Code 写泛型会推断错」 | 在哪个工具里操作就只提那个工具，不要跨工具混用 |
| 写作前未征求用户案例意见 | AI 自己选了泛型案例，用户觉得太技术化 | 每次写文章前必须询问用户有没有亲身经历的场景想放进去，或让用户自己举例（已写入 §8 必问清单） |
| 案例替换不彻底 | 只改了开头几处，文中后续还有 20+ 处旧案例残留 | 替换贯穿案例时，必须用 Grep 全文搜索所有相关词汇，确保零残留 |
| 文章技术门槛过高 | 第 6 期充斥「依赖方向」「循环依赖」「扩展性设计」等软件工程术语，产品经理看不懂 | 目标读者包括非技术人员。软件工程术语必须替换为日常用语（依赖方向->前后一致性，循环依赖->前后矛盾，CI/CD->自动化检查流程）。如果用户反馈「看不懂」，立即降低技术门槛重写 |
| 术语替换后案例也要同步换 | 降低了术语门槛但案例仍是「项目结构」，非程序员不关心项目结构 | 术语和案例要同步降低门槛。如果术语换成了日常用语，案例也要换成日常场景（如活动策划方案、周报、通知邮件） |
