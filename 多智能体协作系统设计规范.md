# 多智能体协作系统设计规范

> **文档用途**：供项目主智能体（Orchestrator）读取，设计和实现多智能体协作系统
> **核心目标**：建立一套通用的、可复用的多智能体协作设计框架
> **适用场景**：任何需要多个AI Agent协作完成复杂任务的场景

---

## 一、设计理念

### 1.1 核心问题

单智能体系统存在三个根本性缺陷：

1. **自我评价失效**：用同一套认知框架评价自己的产出，无法发现盲区
2. **能力边界固定**：一个智能体难以同时精通创作、评审、优化等多个领域
3. **进化停滞**：没有外部反馈机制，系统质量会随时间衰减

### 1.2 解决思路

通过**职责分离**和**制衡机制**，让多个专业化智能体协作：
- 执行的专注执行
- 评审的专注评审
- 优化的专注优化

每个智能体只做好一件事，智能体之间形成相互制约的质量保障网络。

## 二、基础架构

### 2.1 三层架构

```
┌─────────────────────────────────────────┐
│              用户/外部系统               │
└─────────────────┬───────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────┐
│           主智能体（Orchestrator）         │
│  - 任务解析与路由                        │
│  - 流程编排与调度                        │
│  - 质量门控与交付                        │
└─────────────────┬───────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────┐
│            子智能体（Specialists）         │
│  - 执行智能体（Executor）                │
│  - 评审智能体（Reviewer）                │
│  - 优化智能体（Optimizer）              │
│  - 规则智能体（RuleAgent）              │
└─────────────────────────────────────────┘
```

### 2.2 职责定义

| 智能体类型 | 代号 | 核心职责 | 禁止行为 |
|-----------|------|----------|----------|
| 主智能体 | M | 编排、调度、交付 | 直接执行子智能体的工作 |
| 执行智能体 | E | 任务执行、产出生成 | 评审自己的产出 |
| 评审智能体 | R | 质量评估、问题发现 | 修改产出内容 |
| 优化智能体 | O | 问题修复、迭代改进 | 自己决定是否通过 |
| 规则智能体 | A | 规则管理、变更执行 | 执行具体任务 |

### 2.3 通信协议

每个子智能体调用必须遵循以下协议：

```
输入包 → 子智能体执行 → 输出验证 → 结果传递
```

主智能体**只负责**构造输入包和验证输出格式，**绝不代替**子智能体执行。

## 三、质量保障机制

### 3.1 一票否决制

设定绝对红线和边界，任何触犯必须修复：

| 编号 | 否决项 | 判定标准 |
|------|--------|----------|
| N-01 | 产出物违背基本事实 | 存在虚构/冲突的事实陈述 |
| N-02 | 产出物包含禁用内容 | 违反预设的禁用词/模式清单 |
| N-03 | 流程被跳过 | 评审/验证环节被跳过 |
| N-04 | 越权执行 | 主智能体代替子智能体执行 |

### 3.2 多维评分制

建立质量评估的多维框架：

| 维度 | 考察点 | 权重建议 |
|------|--------|----------|
| D1 | 目标达成度 | 25% |
| D2 | 规范性符合度 | 20% |
| D3 | 一致性（内部/外部） | 20% |
| D4 | 可读性/可执行性 | 15% |
| D5 | 创新性/增量价值 | 10% |
| D6 | 技术规范性 | 10% |

### 3.3 分级路由制

根据评分结果决定后续流程：

| 评级 | 分值区间 | 路由 |
|------|----------|------|
| S级 | 90-100 | 直接交付 |
| A级 | 80-89 | 人工确认后交付 |
| B级 | 70-79 | 优化后复评 |
| C级 | 60-69 | 大改后复评 |
| D级 | <60 | 重做 |

## 四、关键流程

### 4.1 标准执行流程

```
用户任务
    │
    ▼
┌─────────────────┐
│   主智能体 M     │
│  任务解析与路由  │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  执行智能体 E    │
│   产出生成      │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  评审智能体 R    │
│   质量评估      │──── N-01触发 ──→ 优化智能体 O
└────────┬────────┘                        │
         │ N-01通过                         │
         ▼                                 │
┌─────────────────┐                        │
│   评级判定      │◀────────────────────────┘
│                 │
└────────┬────────┘
         │
         ▼
    交付用户
```

### 4.2 迭代控制

为防止无限迭代，设定以下规则：

| 规则 | 阈值 | 超过阈值后 |
|------|------|-----------|
| 最大迭代次数 | 3次 | 必须人工介入 |
| 每次迭代最小提升 | 5分 | 未达标则重新分析 |
| 连续相同评分次数 | 2次 | 引入新评审维度 |

### 4.3 退出条件

迭代必须同时满足以下条件才能退出：

1. 评分达到A级以上
2. 无P0/P1级问题
3. 连续两轮无新增P2级问题

## 五、规则演进机制

### 5.1 反馈分类

每次用户反馈必须归类：

| 类型 | 处理方式 |
|------|----------|
| 产出物问题 | 修复产出物 → 验证修复 |
| 流程问题 | 分析根因 → 修改流程规则 |
| 规则问题 | 规则智能体介入 → 更新规则 |

### 5.2 结构化反馈模板

```
【问题描述】
客观描述具体问题

【严重程度】
P0：阻断性问题，必须立即修复
P1：重要问题，应在本次迭代修复
P2：优化建议，可后续处理

【修改建议】
具体的修改方向（可操作的行动指引）
```

### 5.3 规则变更流程

```
反馈输入
    │
    ▼
┌─────────────────┐
│  规则智能体 A   │
│  根因分析       │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  变更方案设计   │
│  新增/修改/删除 │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  变更执行       │
│  规则文件更新   │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  变更验证       │
│  确认生效且有效 │
└─────────────────┘
```

## 六、主智能体操作规范

### 6.1 强制委托清单

以下操作**必须**通过子智能体执行，主智能体**绝对禁止**自行替代：

| 操作 | 必须委托 | 主智能体职责 |
|------|----------|--------------|
| 任务执行 | E | 构造输入包 |
| 质量评估 | R | 构造输入包、验证输出格式 |
| 问题修复 | O | 构造输入包、验证输出 |
| 规则管理 | A | 构造输入包、验证输出 |
| 结果判定 | R（评级） | 构造输入包、执行路由 |

### 6.2 调度前自检三问

主智能体在执行任何操作前，必须回答：

1. **这个操作是否在强制委托清单中？**
   - 是 → 必须委托子智能体
   - 否 → 可自行执行

2. **我是否已经构造了子智能体的输入包？**
   - 否 → 先构造输入包
   - 是 → 继续

3. **如果我打算自己执行，理由是什么？**
   - 唯一可接受的例外：子智能体调用连续失败2次且已向用户报告

### 6.3 上下文隔离

子智能体启动时必须拥有**独立的上下文**：
- 不受主智能体对话历史影响
- 不受其他子智能体执行历史影响
- 每次调用都是全新的开始

**违反后果**：上下文污染会导致评审失去客观性。

## 七、防坑指南

### 7.1 主智能体越界

**症状**：不同任务得到完全一致的评分，评审失去区分度

**根因**：主智能体代替评审智能体执行评审，上下文被污染

**解决方案**：严格执行强制委托清单，每次评审必须调用独立评审智能体

### 7.2 规则悬空

**症状**：规则文件存在但从未被执行，同样的问题反复出现

**根因**：规则文件未被智能体引用，成为孤立文件

**解决方案**：
- 每个规则文件必须在智能体规范中被明确引用
- 每次调用必须验证规则文件的读取状态

### 7.3 反馈未转化

**症状**：用户反馈的问题被修复了，但下次还犯

**根因**：只修复了症状（产出物），没有修复病因（规则/流程）

**解决方案**：
- 每次反馈必须分析根因
- 根因是规则/流程问题时，必须走规则变更流程

### 7.4 迭代陷阱

**症状**：无限迭代，评分卡在某个点不动

**根因**：没有设定退出条件，或退出条件太宽松

**解决方案**：
- 明确设定迭代退出条件
- 超过阈值必须人工介入
- 评分低于阈值不值得继续迭代

### 7.5 质量盲区

**症状**：某些维度的质量始终无人检查

**根因**：评分维度设计不完整，或某些维度没有被执行

**解决方案**：
- 评审清单必须覆盖所有关键维度
- 每个维度必须有明确的验证方法

## 八、系统健康自检

每次任务完成后，主智能体必须验证：

| 检查项 | 验证方式 |
|--------|----------|
| 强制委托清单执行情况 | 记录每次委托，确认无遗漏 |
| 子智能体上下文独立性 | 验证每次调用是独立启动 |
| 规则引用完整性 | 确认本次执行涉及的规则都被读取 |
| 反馈闭环情况 | 确认用户反馈都有对应处理 |
| 产出物归档完整性 | 确认所有产出物都已归档 |

## 十、快速参考

### 核心原则
1. 主智能体是指挥官，不是士兵
2. 子智能体必须有独立上下文
3. 职责边界不可逾越
4. 人类反馈是进化动力
5. 质量闭环必须完整

### 关键数字
- 最少智能体数量：3个（M + E + R）
- 最大迭代次数：3次
- 评级A级门槛：80分
- P0问题恢复时限：立即

### 必要文件
- 智能体注册表（每个智能体的职责和禁止行为）
- 流程编排图（完整的工作流程）
- 否决项清单（一票否决的红线）
- 评分维度定义（每个维度的考察点和权重）
- 规则文件（可执行的规则集合）
